# 08-前端项目相关讨论｜直播笔记整理

## 概览

这场直播表面上像“前端项目讨论”，但真正的主线其实是在把：

* [[03-前端开发相关讨论｜直播笔记整理]]
* [[07-技术选型等事宜讨论｜直播笔记整理]]
* [[前端架构设计指南]]
* [[AI开发主流程]]
* [[PNPM 与 Monorepo]]

这些线再往“真实项目怎么落地”推进一步。

老师这场重点讲的不是某一个框架，而是：

1. 技术选型到底看什么
2. 现代 CSS 为什么和很多人印象里已经不是一回事
3. AI 还原设计稿为什么经常失败
4. 单页工程为什么还是要懂构建链
5. monorepo、类型安全接口和脚手架为什么会成为默认能力
6. 人在 AI 时代到底还负责什么

> [!note]
> 这份整理按老师真正的主线重构。文字稿里有不少口音词和转写漂移，凡是不够确定的，我都只做高概率判断，并统一挂到 [[风险与疑似错词]]，没有硬改成“确定结论”。

## 核心结论

这场直播里最值得保留的判断有几条：

* 技术选型不是选“最潮”，而是选“团队接得住、业务用得上、你解释得清”
* 现在的 CSS 已经进入现代能力阶段，很多老经验和老教材不够用了
* AI 不是把设计稿丢进去就能自动还原，真正的门槛是拆任务和写提示词
* 脚手架和 agent 会越来越强，但前端工程基础不会因此消失
* 真正值钱的能力正在从“纯手工实现”转向“做选择、拆任务、验收结果”

## 一、这场是从一次真实协作问题讲起的

老师开场先在吐槽一个很典型的协作失误：

* 设计稿没有把交互表达清楚
* “弹窗”和“抽屉”没有明确写出来
* 前端实现后，双方对“这到底算什么组件”理解不一致
* 提交前还缺少自测

这一段虽然像闲聊，但其实特别像真实项目里的经典问题：

* 设计稿不只是画视觉
* 还要明确交互形式
* 研发交付前要自己验证
* 不然沟通成本会被来回放大

这也和后面老师反复讲的“AI 不是替你思考”是连着的：

> 如果需求本身说不清，人做会错，AI 做也一样会错。

## 二、老师这场真正想讲的技术选型标准

这一场和 [[技术选型方法]] 基本是一条线，只是讲得更现实。

老师最看重的几个判断维度是：

### 1. 社区和生态

* 国内有没有足够热度
* 遇到问题时能不能查到资料
* 生态是不是成熟

### 2. 团队能力

* 团队会不会用
* 维护成本高不高
* 会不会因为“太超前”把团队直接拖死

### 3. 业务适配

* 技术必须为业务服务
* 如果只为了显得厉害，最后往往会变成项目负担

### 4. 简历价值

老师这场讲得非常现实：

* 如果你写的永远都是烂大街的一套
* 简历就很难有辨识度
* 面试时别人也很难看出你到底推动过什么

所以他反复强调：

* 你解决了什么问题
* 你推动了什么变化
* 你为什么这么推动
* 你给团队带来了什么结果

这几个问题最后都会落回面试表达。

## 三、老师对现代 CSS 的判断非常强

这一场最明显的一个信号是：

> 老师认为很多前端对 CSS 的理解还停留在很多年前。

他反复在强调：

* 要看 `State of CSS`
* 要看 `MDN`
* 要看 `Can I use`
* 要持续关注浏览器新特性

### 他想表达的不是“多记几个属性”

而是：

* 原生 CSS 的能力已经比以前强很多
* 以前必须靠预处理器和插件补的东西，现在很多都原生支持了
* 如果还只拿十年前的认知看 CSS，就会跟现代前端脱节

### 他回顾的样式路线

老师这场大致回顾了一条常见路径：

* 纯 CSS
* `Sass / Less`
* `PostCSS / cssnext`
* `CSS Modules`
* `CSS-in-JS`
* `Houdini`
* 原子化 CSS

他的整体态度非常明显：

* `Less` 这类东西已经太旧
* 很多旧工作流不该继续默认沿用
* 现代项目更值得关注原生 CSS 能力和原子化方案

### 为什么他明显偏向 Tailwind 一类方案

老师给出的理由很务实：

* 重复样式可以被压缩成更细的原子类
* CSS 体积更容易控制
* 首屏阻塞和白屏时间更容易优化
* AI 生成代码时也更容易维持统一风格

这一段实际上和 [[前端技术雷达]]、[[前端架构设计指南]] 是可以互相印证的。

## 四、AI 还原设计稿失败，问题往往不在 AI

这场直播有一段很重要，在继续推进 [[前端新时代开发工作流]]。

老师非常反感一种做法：

* 把整张设计稿丢给 AI
* 再说一句“帮我还原页面”

他的意思很明确：

* 人做页面都不可能这么干
* 你让 AI 一次性整页还原，本身就是错误任务定义

### 他更推荐的做法

先把页面拆成可执行的部分：

* 先讲大结构
* 再讲 `header`
* 再讲 `footer`
* 再讲搜索框、按钮、卡片
* 相同结构要求复用组件
* 相同样式要求抽离
* 复杂页面先圈区域，再分步实现

这其实已经很接近 [[AI开发主流程]] 里的任务设计方式了。

### 这段最关键的本质

老师不是在夸某个模型，而是在强调：

* 你要会把需求讲清楚
* 要会拆顺序
* 要会给边界
* 要会给复用要求
* 要会定义验收方式

所以他才反复说，很多人觉得 AI 不好用，根源其实是：

* 平时就不会提问
* 不会拆问题
* 不会把任务表达清楚

这一段和 [[MCP工具栈]]、[[Figma Make 与 Figma]] 可以直接串起来看。

## 五、老师心里的 AI 开发流程已经不是“补代码”了

这一场里，老师已经把 AI 放进完整交付链了。

他说的思路大概可以整理成三步：

### 第一阶段

* 先把一个主力工具真正用明白
* 找到适合自己的模型和入口

### 第二阶段

* 把明确流程交给 agent
* 让它自己去走
* 你不要每一步都人工盯着

### 第三阶段

* 一个模型写
* 一个模型 review
* review 不过就继续改
* 最后再由人验收

这和 [[AI工具栈]]、[[模型与扩展选择]] 其实已经是同一套思路：

* 模型之间能力不同
* 有的快，但粗
* 有的慢，但细
* 最好的方式不是迷信单模型，而是让它们彼此制衡

老师这一场最重要的判断其实是：

> 人最后不会完全消失，但人的工作会越来越偏向选型、派工、验收和风控。

## 六、为什么单页工程基础还是要会

老师这场虽然讲了很多 AI，但并没有因此放弃工程基础。

他明确在讲一套比较完整的 `SPA` 工程化思路，大致包含：

* `React`
* 路由
* 原子化样式
* 状态管理
* 不可变数据
* 构建链
* 代码规范
* 提交检查
* 单元测试
* `E2E` 测试

### 老师特别强调的点

不是“你必须自己从零造轮子”，而是：

* 你不能只会用现成 CLI
* 至少要理解构建链
* 要知道开发和生产环境到底差在哪里
* 要知道压缩、拆包、资源处理为什么会影响结果

这就是为什么他反复强调：

* `SPA` 不能只会写页面
* 构建和部署理解不清，很多性能问题就永远说不明白

这一段和 [[SPA 与 MPA]]、[[前端架构设计指南]] 是直接相连的。

## 七、monorepo、私仓和一体化脚手架，老师已经当成默认未来

老师这场花了很大篇幅在讲：

* 单独的 `SPA` 项目怎么起
* 公共组件库怎么抽
* 公共配置、数据库、授权、基础设施怎么复用
* 为什么这些东西最后会把你带进 `monorepo`

### 他想表达的不是“monorepo 很高级”

而是：

* 单项目和多项目的组织方式本来就不一样
* 如果公司内部要共享 UI、infra、auth、db
* 那么仓库组织方式就必须升级

### 脚手架演示这段最关键

老师现场演示的那类工具，本质是在做一件事：

* 把以前需要手搓很久的一套工程初始化
* 变成一连串“你要怎么选”的问题

比如：

* 路由怎么选
* 后端运行时怎么选
* 数据库怎么选
* ORM 怎么选
* 类型安全接口怎么选
* 部署到哪里
* 要不要授权、文档、代码质量工具、skills

所以老师才会说：

> 以后真正值钱的不是从零搭模板，而是你知不知道每个选择在项目里意味着什么。

这一段和 [[PNPM 与 Monorepo]] 是最强相关的。

### 老师对类型安全接口的态度

老师对 `tRPC` 这类方案的态度也很直接：

* 不要神秘化
* 本质是前后端共享契约
* 让前端少一点 `any`
* 让查询和变更的边界更清晰

他的意思不是说它一定适合所有项目，而是：

* 现代全栈 TypeScript 栈里
* 这类方案已经越来越像主线能力

## 八、自然语言测试这段，是在把自动化继续往前推

收尾时老师临时演示了一类“自然语言驱动浏览器测试”的方式。

他最开始以为这是个自带模型的测试工具，后来发现底层还是要依赖外部大模型。

但这一段依然很重要，因为它说明：

* [[前端自动化测试]] 正在被重新定义
* 过去你要自己写很多测试脚本
* 现在可以先把测试目标翻译成自然语言和流程描述
* 再让模型去驱动浏览器执行

老师最后的判断其实很朴素：

* 这不是“测试不需要懂了”
* 而是“测试入口正在变自然语言化”

这和 [[自动化测试]]、[[前端自动化测试]]、[[Gemini 前端工作流]] 都能接上。

## 九、课程安排和作业提醒

这一场最后还把课程路线又往后推了一步：

* 这周主线还是 `SPA`
* 下周会讲 `MPA`
* 以及 `MPA + SPA` 的结合
* 还会补今天没完全接进去的自动化部分
* 再往后会继续接 `WEB3` 方向

老师对同学的要求也很直接：

* 作业必须做
* 环境要补齐
* Mac 基础和工程配置不能一直靠蒙
* 求职方向最好拆开，不要把 AI、前端、WEB3 全塞在一份简历里

## 十、这场真正想训练的能力

如果把整场压成一句话，老师真正想训练的不是“前端把页面切快一点”，而是：

1. 能做技术判断
2. 能把需求拆成 AI 可执行任务
3. 能理解现代前端工程的基本结构
4. 能在 AI 帮你干活之后，继续负责最终结果

所以这场直播和前几场合在一起看，核心已经很清楚了：

> 前端正在从“手工实现岗位”变成“工程判断 + 流程调度 + 最终验收”的岗位。
