# 12-前端开发要点与安排｜直播笔记整理

## 概览

这一场基本围绕“前端项目落地+状态管理+AI协作习惯”展开，穿插了：

* Tailwind v4 配置与样式拆分
* Monorepo 下的 hooks 包拆分
* React 状态管理与面试题
* AI 生成后的 review 与流程设计
* Chrome 本地模型与隐私风险

> [!note]
> 口音词很多，尤其集中在 Tailwind、React Router、Jotai、Immer、useTransition 一类词上。我把不确定的统一收口到 [[风险与疑似错词]]，不做硬改。

## 核心结论

* 先把样式拆分清楚，再往项目里“灌业务”，否则后期维护一定崩
* AI 不是问题，问题是你有没有办法 review 并说清楚“为什么这样设计”
* React 状态管理不是只有 Redux，要能把“八种流派”讲清楚
* hooks、路由、样式库最好都能抽成独立包，减少强耦合
* 真实项目上线比“假项目”更能让人有底气

## 一、课程节奏与项目安排

老师反复强调节奏：

* 本周把前端落地细节过完
* 下周会继续后端与业务流程
* 这场没有新作业，只要求补完昨天的作业

还有一个明确的现实目标：

* 项目是日本公司真实需求
* 已经签约，目标是上线后收款

## 二、AWS 权限与区域选择

* AWS 权限如果是公司 SSO 会比较麻烦
* 区域不一定要放美国，建议东京、首尔、新加坡、悉尼
* 有些权限需要一步步点出来（现场提到 `SAM` / `RMQ` 等）

## 三、Tailwind v4 与样式拆分策略

老师在这场里反复强调“样式拆分”的重要性：

* Tailwind v4 更倾向于 `@layer` 的方式管理样式
* 公共样式不应该和 Web 业务样式混在一起
* UI 组件库只负责主题、颜色、字体、组件自身样式
* Web App 自己维护页面级样式与业务样式

他说的核心是：

* UI 组件库不能扫描 App 目录
* App 也不应该依赖 UI 的扫描路径
* 否则后期拆包、发私仓会直接崩

## 四、Monorepo 拆分与私仓发布

这场明确提出了“拆包”的方向：

* UI 组件库计划单独发布到私仓
* 需要配置 `.npmrc`
* Hooks 也要拆成独立包

同时他给了一个现实判断：

* 如果不做拆分，AI 生成出来的代码一开始就带耦合
* 后期维护成本会翻倍

## 五、数据请求与调试工具

现场提到了“redquery”的请求工具（疑似 React Query / TanStack Query）：

* 支持缓存与重试
* 比单纯 `fetch` 更稳

还有“开发调试工作台”思路：

* 线上不展示
* 但开发期可以看请求、对象、状态
* 开发工具做得好能大幅提升排障效率

## 六、React 状态管理与面试题

老师把“状态管理”拆成 8 类，强调这是高频面试题：

* Redux：老牌，复杂但很多公司还在用
* MobX：简化 action / reducer
* useReducer：偏函数式
* Context：层层 Provider 会比较重
* useState：适合局部状态
* 全局状态库：独立于组件
* 原子状态：解决状态撕裂
* 状态机：面试必问点

他提醒的面试重点：

* 不要嘲讽别人用 Redux
* 解释“为什么选它”比背概念更重要

## 七、状态撕裂与 useTransition

老师在状态撕裂里重点讲了：

* React 渲染优先级会导致“输入框变了但列表没变”
* `useTransition` 可以控制优先级
* 默认情况下可能会出现“双渲染”

面试高频 API：

* `useEffect`
* `useCallback`
* `useMemo`

## 八、不可变数据与 V8 细节

这段偏底层：

* `Object.create(null)` 会生成 V8 的“慢对象”
* 慢对象内存和 CPU 消耗都很高
* 不要直接修改原对象，建议用不可变更新
* `Immer` / `useImmer` 是更常见的方式

他强调：

* AI 生成 hook 之后一定要 review
* 要能看懂泛型、函数签名、`as const`

## 九、Hooks 包与 useEnum 演示

演示内容：

* 新建 `hooks` 包
* 用 `pnpm --filter` 把依赖加到 Web
* 写一个简单的 `useEnum` / `useImmer` 模式
* 用 `useCallback` 包裹更新函数
* 返回 tuple（value + updater）

关键点是：

* 先写函数签名
* 再写实现
* 保证泛型能推导

## 十、路由结构与 React Router v7

他把路由结构拆成两种思路：

* 旧版：routes 配置 + pages 分离
* v7：文件系统路由（flat routes）

建议是：

* routes 目录保持薄，只做路由定义
* 业务逻辑挪到 components
* 升级 v6 -> v7 可以作为“架构迁移案例”在面试里讲

## 十一、AI 协作与 Review 习惯

老师对 AI 的态度非常明确：

* 不要让 AI 先写，先确认方案
* AI 只能辅助，责任在开发者
* AI 生成的代码必须 review
* 没有 review 能力，面试一定过不了

他还强调了“权限控制”：

* 不给 AI 完全权限
* 不让 AI 自动 merge
* 改一次就 commit 一次

## 十二、Chrome 本地模型与隐私提醒

这段是关于 Chrome 本地模型：

* Chrome 里可能有本地模型（体积很大）
* 有些能力是本地 + 云端混合
* 美区账号限制仍然存在
* 部分功能会默认把数据传给 Google

老师提醒：

* 能关就关
* 隐私风险要清楚

* 项目已经足够上线标准
