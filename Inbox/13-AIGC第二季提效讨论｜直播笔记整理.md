# 13-AIGC第二季提效讨论｜直播笔记整理

## 概览

这场直播虽然名字叫“AIGC 第二季提效讨论”，但真正讲的已经不是“再推荐几个工具”，而是把课程从：

* 第一季：会用 AI 工具

推进到：

* 第二季：把 AI 接进企业工作流
* 让 AI 自动跑需求、设计、review、测试、调度

它和这些笔记是直接连着的：

* [[09-AI技术应用研讨会｜直播笔记整理]]
* [[AI开发主流程]]
* [[企业落地判断]]
* [[Agent 与 Skill]]

这场最重要的转折是：

> 第一季解决“人会不会用 AI”，第二季解决“企业能不能把 AI 变成稳定流程”。

> [!note]
> 这份整理按主线重构。文字稿里有不少发音漂移，尤其集中在 `Bedrock`、`Dispatch`、`OpenCLI`、`CDP skill`、`YOLO`、`Sonnet / Opus` 一类词上。凡是不够确定的，我都统一挂到 [[风险与疑似错词]]，没有硬改成绝对结论。

## 核心结论

* 第一季的“买账号、会聊天、会生成”远远不够，企业真正需要的是自动化闭环
* 真正的企业级提效，不是单模型直出，而是 `需求 -> 设计 -> review -> task -> 编码 -> 测试 -> 验收`
* 以后更值钱的人，不只是会写代码，而是会设计 AI 工作流、会做 skill 编排、会做人机协作验收
* AI review AI 是第二季的关键思想，单个模型直接闷头生成在复杂系统里风险太高
* 企业并不一定比程序员更懂技术，但它一定会用成本、产出和效率来重新衡量岗位价值

## 一、老师先把“第二季”和“第一季”的边界划开了

老师开场就说得很清楚：

* 第一季讲的是基本工具使用
* 第二季讲的是企业级提效

他的判断是：

* 很多人即便拿到了账号，也没有真正完成提效
* 问题不在于“有没有工具”
* 而在于不会提问题、不会设计流程、不会把 AI 放进工作里

所以这场的真正主题不是“再学一个工具”，而是：

* 怎么让 AI 不依赖人手动打开、手动盯着、手动确认每一步

## 二、企业真正想要的不是聊天，而是全流程自动化

老师这场对企业需求的定义很明确：

* 需求自动化
* 设计自动化
* `code review`
* 验收
* 自动化测试
* 最终交付

他说得很直接：

* 只会让人坐在电脑前打字，不能算企业级提效
* 企业真正要的是 24 小时可跑的流程

这点和 [[企业落地判断]] 是直接连着的：

* 老板最终看的是成本
* 看的是你能不能把流程打通
* 不是你会不会背几个模型名字

## 三、第二季第一步不是写代码，而是先把“专业终端”配好

这场前半段其实花了很大篇幅讲终端和配套工具，原因很简单：

* 你要让 AI 真正在企业环境里稳定干活
* 终端体验、上下文管理、资源监控、文件导航都不能太差

老师提到的工具线包括：

* 终端配置工具
* 系统资源监控工具
* 文件树 / 目录浏览工具
* `Amazon Q` 一类终端增强
* `Claude` 上下文管理命令

其中最值得记住的是两个思路：

### 1. 能沉浸在终端，就尽量别来回切 GUI

老师的目标是：

* 让人尽量留在终端里
* 文件结构、系统状态、模型使用情况都能直接看

### 2. 上下文管理是硬需求

老师反复强调了：

* 压缩上下文
* 重置上下文
* 区分 plan model 和 code model

他的意思其实就是：

* 不是“一个对话一直聊到底”
* 而是要把上下文、规划和执行拆开

## 四、模型分工已经从“哪个好用”变成“谁做规划、谁做执行”

这场最值得保留的一个细节是：

* 用更强但更慢的模型做 plan
* 用更快、更适合编码的模型做 code

老师现场讲的意思大致是：

* `Opus` 一类模型负责规划
* `Sonnet` 一类模型负责写代码

这和 [[模型与扩展选择]]、[[AI工具栈]] 可以接起来看。

他想强调的不是单点模型强弱，而是：

> 模型开始像团队成员一样被分工，而不是“一把梭”。

## 五、AWS Bedrock 这一层的重要性：稳定、不容易封、能接企业环境

老师这场明显很看重 `AWS Bedrock` 这一层，主要理由有：

* 走官方模型入口
* 企业环境更容易接
* 不像很多民间“中转”那样不透明
* 能通过 AWS 账户、SSO、API 密钥进入企业体系

他还特别提醒：

* 很多所谓“中转”其实不可信
* 外面卖的很多服务，底层不一定真是你以为的模型

所以这里的核心判断其实是：

* 企业场景更看重稳定、合规和接口来源
* 而不是一味追求“便宜能用就行”

## 六、告别“小龙虾”：老师在讲“截图型 Agent”的上限

这一场中老师一直在拿“小龙虾”做对比。

他想表达的不是这个工具完全没价值，而是：

* 截图驱动的方式太吃提示词
* 操作 APP、Electron、浏览器时不够丝滑
* 多 agent 协作时链路太长
* 中间经过截图、模型、skill、远程控制后，失败点太多

所以老师这场在找的是：

* 更直接的桌面控制
* 更稳的终端控制
* 更短的工具链

## 七、这场真正想推的是“手机控制电脑里的 Claude 工作流”

老师给出了三种操作形态：

### 1. `Dispatch`

它更像：

* 官方派发任务入口
* 适合从外部直接触发电脑里的工作流

### 2. 手机 / 对话入口直接驱动 code

意思是：

* 在手机里发消息
* 让桌面上的 Claude / code 执行任务

### 3. `Telegram` 通道

这个方式的好处是：

* 消息能记住
* 适合远程异步触发
* 可以把“手机就是控制面板”这件事跑起来

老师真正想落下来的能力是：

> 不在电脑前，也能通过手机把整个开发流转起来。

## 八、危险模式不是噱头，而是“真接管电脑”

这场里老师专门讲了：

* `dangerous skip` / 危险权限跳过
* `YOLO` 一类模式

他的意思是：

* 一旦开这种模式
* 电脑就进入“默认允许 AI 操作”的状态

所以他一边说它很强，一边也在提醒风险：

* 最好放在单独机器
* 最好有远程控制兜底
* 最好能随时断开后台进程

这部分最好和 [[风险与疑似错词]] 一起看，不要直接把它当作“默认推荐配置”。

## 九、`CDP skill`、`OpenCLI` 和 APP 自动化把链路接通了

老师这场最实操的一部分，就是把不同自动化组件串起来：

### `CDP skill`

他强调的价值是：

* 读已经登录的网页文档
* 读产品文档
* 直接从浏览器里拿上下文

这意味着：

* PRD、在线文档、设计材料可以直接进 AI 工作流

### `OpenCLI`

老师把它当作：

* 调桌面软件的中间桥
* 可以操作浏览器、IDE、飞书、微信、Notion 一类应用

他反复强调的是：

* 如果 AI 能调桌面应用
* 你的日报、周报、文档整理、代码 review、提交流程都能串起来

### AppleScript / 原生自动化

老师提到它比“截图点点点”更底层，原因是：

* 它是真正的系统自动化脚本能力
* 不是单纯 OCR + 截图猜界面

这部分和 [[APP自动化]]、[[VSCode自动化]] 是直接能接起来的。

## 十、真正的第二季核心：AI review AI

老师最看重的不是“让 AI 写代码”，而是：

* 让一个 AI 出需求 / 出设计
* 让另一个 AI review
* 再继续迭代

他给的实际工作流大致是：

1. 用 `CDP skill` 把产品文档拉下来
2. 先分析需求
3. 生成技术设计方案
4. 再用另一个模型 / 工具 review
5. 再拆 task
6. 再进入编码
7. 最后跑自动化测试

老师明确认为：

* 单模型直接生成复杂系统风险很高
* 多轮 review 是必须的

他还举了“权限系统 / 多签”的例子：

* 如果你只是直觉写“下属可转钱”
* 但没把多签和审批链路考虑进去
* 这类系统上线就会出大问题

这段和 [[AI开发主流程]] 可以直接互相回链。

## 十一、老师把“需求、设计、task、代码、测试”拆成了真正的流水线

这场最值得你保留的不是具体工具名，而是这条链：

* 需求
* design
* requirements
* task
* coding
* review
* 自动化测试
* 上线前验收

老师的观点非常明确：

* 只让 AI 出代码，没有意义
* 只有把它放在整条流水线上，企业才会真的觉得值钱

## 十二、Google 提到的 skill 设计模式，是这场的关键方法论

老师专门拿了 Google 的一套 agent / skill 设计模式来讲。

这部分最适合拆成参考页，所以我单独挂到了：

* [[Skill 设计模式与任务编排]]

这场里点到的核心模式包括：

* 工具包装器
* 生成器
* 审核员
* 先访谈再行动的角色
* 流水线

老师为什么看重这套东西？

因为它解决的是：

* skill 不是一个个散着放
* 而是怎么按角色组织
* 怎么按流程组合
* 怎么让它们真的为企业服务

## 十三、老师自己的方案：比“满天飞任务编排”更在意 Token 效率

老师现场说他也有一套自研方案，核心点不是炫技，而是两个现实目标：

* 省 Token
* 保结果

他自己的工作方式大致是：

* 先做需求生成
* 再做 design
* 再 review
* 再拆 task
* 最后接自动化测试

和某些“任务编排套娃”方案相比，他更在意：

* 不要把 Token 烧在无意义的流程上
* 不是流程越复杂越先进
* 而是产出要跟消耗匹配

他举的例子很现实：

* 公司不是不肯给 Token
* 但如果你烧得多、产出少
* 老板就会直接质问你值不值

这和 [[企业落地判断]] 是完全一致的。

## 十四、老师对岗位变化的判断：会出现“AI 工作流开发 / 编排 / Fix”型岗位

老师讲到最后，其实已经把岗位变迁说得很直白：

* 以后会有专门做 AI coding workflow 的岗位
* 会有专门做自动化编排、skill 编排的人
* 甚至会有专门“基于 AI 做 fix 和守护”的岗位

他的判断不是“程序员彻底没了”，而是：

* 重复执行岗位被压得更狠
* 能设计流程、验收结果、优化成本的人更值钱

## 十五、老师为什么说传统企业机会反而变多了

这场里老师明显有一个乐观判断：

* 传统企业开始全面进入 AI 改造期
* 做门窗、制造、流程管理、办公协作的企业都在碰这件事
* 它们未必懂编程，但它们会需要真正能落地的人

所以老师对就业的判断是：

* 机会不是少了
* 是重心变了

从“会不会写一个页面”，变成了：

* 会不会搭一套 AI 协作和自动化流程

## 十六、这场最值得你记住的主线

压缩成一句话就是：

> 第二季的重点，不是“再多学几个工具”，而是把 AI 从个人助手升级成公司工作流的一部分。

如果你回看这场，只记住下面四件事就够了：

* 第一季学工具，第二季学自动化
* skill 要按角色和流程组织
* AI 必须 review AI
* 最终价值不在工具，而在企业里能不能跑出稳定产出
