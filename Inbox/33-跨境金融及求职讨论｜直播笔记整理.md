# 33-跨境金融及求职讨论｜直播笔记整理

这场课的核心是一个 [[JuniApp-JuniGo架构]] / [[跨境金融系统]] 案例：用 [[Solana]]、[[USDC]]、[[稳定币]]、[[双钱包归集]]、[[KYC]] / [[KYB]] / [[KYT]] / [[AML]]、[[Onramp]] / [[Offramp]] 和 Serverless 架构，把链上资产和法币结算连接起来。

相关导航：[[33-跨境金融及求职讨论-知识地图]]

## 一句话总结

老师想表达的是：[[Web3]] 不一定只等于炒币、交易所或灰产，它也可以作为跨境金融、全球结算、稳定币清算和法币出入金的底层技术。对求职来说，跨境金融项目很适合写成“Web2 合规 + Web3 资金流”的项目，但必须能讲清楚资金流、身份认证、风控、异步回调和合规边界。

## 1. 项目背景：用 Web3 技术解决跨境金融问题

老师介绍了一个自己做过架构顾问的跨境金融项目。项目早期由同学参与开发，后期因为金融逻辑、系统架构和外部系统对接越来越复杂，老师介入重新设计架构。

这个项目的核心不是“发币”，而是：

- 用户把链上稳定币转入平台。
- 平台做身份认证和链上交易风控。
- 资金进入用户子钱包，再归集到主钱包。
- 通过 payout provider 把稳定币换成目标国家的法币。
- 最终以同名账户或合规收款路径完成结算。

老师强调，这类项目可以作为 [[Web3简历]] 中比较有现实业务价值的一类方向，因为它把 [[稳定币]]、钱包、链上转账、合规认证和传统金融通道串在一起。

## 2. 为什么商户需要法币，而不是只收 USDT / USDC

老师用义乌商户、海外收款和尼日利亚等场景说明：稳定币适合全球转账，但不是所有商户都能直接使用稳定币。

现实问题是：

- 用户或客户可能愿意付 [[USDT]] / [[USDC]]。
- 商户要发工资、进货、纳税、做账，通常仍然需要法币。
- 国内商户直接收到 U 后，怎么花、怎么入账、怎么换成法币，都是难点。
- 一些国家或地区可以更顺畅地支持银行卡充值、稳定币支付或法币结算。

所以跨境金融系统的价值在于：把链上资产和现实法币体系连接起来。

## 3. 资金流：子钱包、主钱包和归集

老师重点讲了 [[双钱包归集]]。

典型流程是：

1. 用户发起充值或转账。
2. 资金先进入用户的子钱包。
3. 平台通过 [[KYT]] / 风控判断这笔链上交易是否安全。
4. 风险通过后，再把子钱包资金归集到主钱包。
5. 主钱包再对接 payout provider 完成出金或法币结算。

这样设计的原因是：不要让风险资金直接进入主钱包，避免平台主账户被污染，也避免用户在平台内用风险资金继续操作。

这里涉及一个高风险点：平台是否托管用户私钥、如何管理私钥、如何做权限隔离和审计。面试时如果写这类项目，不能只说“有钱包”，还要能说明托管、非托管、多签、权限、审计和风控边界。

## 4. KYC / KYB / KYT / AML 是跨境金融的基础词

这场课反复强调几个术语：

- [[KYC]]：Know Your Customer，个人身份认证。
- [[KYB]]：Know Your Business，企业认证。
- [[KYT]]：Know Your Transaction，交易监控 / 链上资金风险识别。
- [[AML]]：Anti-Money Laundering，反洗钱。

转写里的 “PYT” 大概率是 [[KYT]]；“SUB SUB / sum / samaltman / 3M” 大概率是 [[Sumsub]]。

老师提醒，KYC/KYB 会收集大量隐私信息，包含证件、自拍、地址、护照等。这样的系统绝不能在国内随便自己收集和保存公民身份信息，更适合接入合规第三方服务，并且要有清晰的数据保护和权限控制。

## 5. Onramp / Offramp：把链上资产送进法币体系

老师提到的 payout provider 包括：

- [[Tazapay]]
- [[Yellow Card|YellowCard]]
- [[Busha]]

这些服务的作用是连接银行、法币账户和本地支付通道。平台会根据国家、汇率、费用、到账速度和可用通道，综合选择 provider。

结合架构图，这里应拆成两类能力：

- [[Onramp]] / Direct Deposit：法币入金，报价、支付、到账。
- [[Offramp]] / Beneficiary：法币出金，quote、payout、status。

一个典型路径是：

- 平台把主钱包里的稳定币转给 payout provider。
- provider 接收目标账户、收款人信息和转账资料。
- provider 通过本地银行或同名账户路径，把钱结算成美元、人民币或目标国家法币。

老师特别提醒，实际入境、结汇、银行解释、税务材料等都属于高敏感合规问题。笔记里只保留业务和系统设计，不把口述中的“伪造材料”沉淀成可执行做法。

## 6. 为什么 USDC 在这类场景里更常见

转写里多次出现 “UIDC”，应纠正为 [[USDC]]。

老师的判断是：在部分海外合规场景里，[[USDC]] 比 [[USDT]] 更容易被机构接受，因为 USDC 背后关联 Coinbase、Circle、美国监管环境和更强的合规叙事。

这并不代表 USDT 没有价值。更准确的理解是：

- [[USDT]] 在币圈流动性和使用习惯上非常强。
- [[USDC]] 在一些合规机构、支付、结算、稳定币应用里更容易被接受。
- 不同国家、不同通道、不同 provider 对稳定币的接受程度不同。

## 7. 系统架构：多端、多第三方、多回调

结合架构图，项目名应整理为 [[JuniApp-JuniGo架构]]。它覆盖 C 端、后台、KYC/KYB、法币 Onramp/Offramp、链上钱包/转账、核心服务、数据与外部集成。

老师把这个项目拆成四类端和多个外部系统：

- PC 端。
- 移动端。
- 运营后台。
- 文档 / 节点 / 外部回调系统。
- 外部回调入口：`/webhooks/privy`、`/sumsub`、`/fiat-gateway/notifications`。

第三方系统包括：

- [[Privy]]：注册、钱包或身份入口。
- [[Sumsub]]：KYC / KYB / KYT / AML。
- [[Tazapay]] / [[Yellow Card|YellowCard]] / [[Busha]] 等 payout provider。
- [[WhatsApp]]：消息通知。
- [[Mailgun]]：邮件通知。
- [[Slack]]：监控和告警。
- [[Sentry]]：错误监控。

这类系统难点不是单个 API 难，而是外部系统太多：每个 provider 都有自己的接口、回调、错误码、审核流程、费用、汇率和人工沟通成本。

## 8. AWS Serverless：Lambda 是 stateless，异步状态要兜底

架构图里明确的技术栈包括：

- [[Monorepo]]：pnpm + Turbo。
- [[TypeScript]] 全栈。
- [[Hono API Gateway]]：统一 API 入口，处理 CORS、CSRF、RateLimit、Error、Sentry init、Env 校验、Lambda/HTTP。
- [[路由层]]：`users / wallets / transfers / assets / sumsub / onramps / offramps / webhooks`。
- [[核心业务服务]]：`tx.svc`、`wallet.svc`、`business.svc`、`ramp shared`、`notification`、`funds collection`。
- [[PostgreSQL 与 Drizzle]]
- [[Redis 与 BullMQ]]
- [[EventBridge]]
- [[SQS]]
- [[S3]]
- [[Sentry]]
- [[Cloudflare 与 AWS Lambda]]

转写里的 “SQRS” 应纠正为 [[SQS]]；“Cloudera”结合架构图应纠正为 [[Cloudflare 与 AWS Lambda]]。之前记录的 “the better state” 没有在图里出现，暂时不再强行解释为 Better-T-Stack。

重点是：[[Lambda]] 是 stateless 服务，不能假设一次函数调用会一直保存状态。跨境转账、KYC 审核、provider 出金这些流程往往是异步的，可能等待很久才回调。因此需要：

- 把任务状态持久化到数据库或对象存储。
- 用 [[EventBridge]] 做定时兜底和重试。
- 用 [[SQS]] 解耦异步任务。
- 对外部 webhook 做幂等处理。
- 用 Sentry / Slack 监控错误。

这部分非常适合面试里讲“我不是只会调接口，我知道跨境金融系统为什么需要状态机、重试、幂等和告警”。

## 9. Web3 不是天然灰色，关键看业务和合规边界

老师反复强调：这个项目是用 Web3 技术做现实金融问题，不是“链上就是骗子”。

它的价值点在于：

- 用稳定币提升跨境结算效率。
- 用链上交易增强资金流透明度。
- 用 KYC / KYB / KYT / AML 接入合规流程。
- 服务一带一路、跨境贸易、海外商户和全球结算场景。

但这类方向也有明显风险：

- 稳定币政策变化。
- 银行入账和结汇解释。
- 个人信息保护。
- 跨境资金来源证明。
- provider 合规资质。
- 用户私钥托管和资产安全。

所以它不是“无风险通道”，而是一类合规、风控、工程复杂度都很高的业务。

## 10. 求职建议：选一个项目打深，不要每个都浅浅碰一下

老师最后回到求职：

- 如果对 [[跨境金融系统]] 感兴趣，可以自己模拟一套流程。
- 如果对交易所感兴趣，可以继续打磨 [[DEX永续合约]]。
- 如果做 App 出身，可以考虑钱包 App、跨境支付 App、链上资产 App。
- 如果写简历，可以把跨境金融写成“Web2 合规 + Web3 资金双层架构 + 全球结算”。

他提醒大家不要什么都写一点，但每个都没有细节。真正能帮求职的是：选一个适合自己背景的项目，持续打磨到能讲清楚业务、架构、异常、风控和真实限制。

## 11. 不要再等：结课后尽快进入投递

老师最后的提醒很直接：结课后就要逐步开始找工作，不要一直等。

原因是：

- 技术会忘。
- 项目热度会过去。
- 心气会被等待消耗。
- 简历和作品必须在投递中迭代。

他建议线下同学现场验收成果，线上同学也要自己给自己一个交代：到底完成了什么，能不能写进简历，能不能经得起面试追问。
