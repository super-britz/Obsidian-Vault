# 28-SSR与Serverless探讨｜直播笔记整理

## 给没认真听的速读版

这节课其实在回答两个问题：

1. 一个 React SSR 项目，除了“把组件渲染成 HTML”之外，到底还要补哪些工程环节
2. 为什么老师最后会把问题一路引到 [[Serverless架构]]、[[边缘渲染与流式渲染]] 和云上弹性部署

如果只记 8 句话，可以先记这 8 句：

* `SSR` 最核心还是两件事：[[AIO 与 SEO]] 和性能
* 但 `SSR` 不是只会 `renderToString` 就结束了，还要补资源收集、数据预取、脱水/注水、路由匹配和 head 注入
* Node 侧现代 React SSR 的主流 API 是 `renderToPipeableStream`
* 边缘运行时更适合 `renderToReadableStream`
* 数据预取这一层，老师用的是 `TanStack Query`
* `QueryClient` 一定要按请求创建，不能挂成全局单例
* 真正限制 Node SSR 上限的，往往不是 React 本身，而是部署、扩容、日志、监控和资费
* 老师的核心判断非常明确：对高并发 SSR 而言，`Serverless + IaC + 边缘节点` 比“单机 Node 服务”更像长期答案

这场和这些笔记是连续的：

## 这场里需要先纠正的错词和概念

### 高概率错词

* `资源` 在这场大量是 `SSR`
* `Su / so` 基本都是 `SEO`
* `IAO / ADO / AIO` 这里统一按 `AIO` 理解
* `TTLB` 基本是 `TTFB`
* `render to payabel stream` 是 `renderToPipeableStream`
* `render to readabel stream` 是 `renderToReadableStream`
* `Rio Poly / recquery / Tecno query` 基本都是 `TanStack Query`，老师口头上也带着旧名 `React Query`
* `dehered / deheredrated` 是 `dehydrate / dehydrated`
* `hydrationboundary` 是 `HydrationBoundary`
* `headroot` 是 `hydrateRoot`
* `Stack Router` 是 `StaticRouter`
* `reactme async / hello米` 是 `react-helmet-async / Helmet`
* `Vessel / Verso / verto` 基本都是 `Vercel`
* `Cloudfront` 在平台语境里高概率是 `Cloudflare`
* `global watch` 是 `CloudWatch`
* `IAC / IC` 指 `IaC`，也就是 Infrastructure as Code
* `JTI` 是 `JIT`

### 需要纠正的几个概念

* `renderToString` 不是“彻底废弃”，但在现代 React SSR 里通常不再是首选，因为它不支持流式输出
* `SSR` 通常会改善首屏可见内容和抓取友好度，但不代表所有指标都天然更好；如果部署在单一中心节点，`TTFB` 反而可能变差
* `useQuery` 不是靠服务端执行 `useEffect` 生效的。更准确的说法是：服务端先 `prefetch + dehydrate`，客户端再 `hydrate`
* `Serverless` 不是所有 Node SSR 的唯一正确答案，但在“高并发、按量计费、快速上线、弹性扩缩容”这些目标下，确实非常有吸引力
* 老师对 `Vercel`、`Next.js`、传统运维的评价很强烈，这是他的工程经验总结，不等于所有团队都必须照搬

## 核心结论

* 这节课是 [[23-SSR框架技术讨论｜直播笔记整理]] 的实现篇和部署篇
* 老师真正想教的，不是“Next.js 文档怎么背”，而是“自定义 React SSR 从入口到上线到底补哪些环节”
* SSR 链路至少要想清楚这几层：
* `render API`
* 路由命中
* 代码分割资源收集
* 数据预取
* 脱水 / 注水
* head / meta 注入
* server build
* 部署与扩容
* 服务端数据预取最重要的工程细节，是“每个请求一份 `QueryClient`”，否则就会出现跨用户数据污染和内存问题
* 在老师的判断里，SSR 真正的瓶颈往往不在 React，而在“你怎么把它扛到高并发、低成本、全球节点、可观测”的生产体系里
* 所以这场后半段才会一路转向 `Lambda`、`Workers`、自动扩缩容、冷启动、`IaC`、日志监控和计费模型

## 一、00:24-04:18 先把上一节的结论捡起来：SSR 还是为了两件事

### 1. 第一件事：[[AIO 与 SEO]]

老师一开头就在回顾上一节的核心判断：

* `SSR` 不是为了“看起来高级”
* 还是先解决 `搜索可见性`
* 但现在不应该只说 `SEO`
* 更要一起理解 `AIO`

他的意思是：

* 以前主要面对搜索引擎爬虫
* 现在还要面对各种 AI 检索系统
* 这些系统同样会抓取页面、抽取内容、评估质量和权重

所以这节课开头反复强调：

* 页面数量
* 内容质量
* 页面性能
* 抓取友好度

这些东西依然是站点曝光的基本盘，只是抓你内容的对象已经不再只是传统搜索引擎。

### 2. 第二件事：性能

老师继续用时间线去解释 [[SSR 与 CSR]]：

* `CSR` 通常要等 HTML 回来、JS 下载完、JS 执行完，页面内容才真正可见
* `SSR` 在 HTML 返回时就能更早给出有内容的首屏

这个结论本身没问题，但要加一个更准确的补充：

* `SSR` 更容易让用户更早看到内容
* 但总体验还取决于：
* 服务端渲染耗时
* 资源体积
* hydration 成本
* 节点距离
* 缓存和边缘分发

### 3. `TTFB` 不是天然更好，部署方式会直接决定结果

老师这里其实说得很对：

* `SSR` 并不保证 `TTFB` 一定更低
* 如果你的服务只部署在单一机房，离用户太远，`TTFB` 反而会变长

所以他很自然地把话题引到了：

* 不要只看渲染模式
* 还要看节点部署
* 还要看边缘分发

这和 [[TTFB 与 Web Vitals]]、[[边缘渲染与流式渲染]] 是直接相连的。

## 二、04:18-10:43 React SSR 真正的入口：不是“渲一下”就完了

### 1. 老师代码里的入口，是从服务端请求处理开始

他明确说上一节大概只讲了“入口”，这一节才开始讲完整链路。

他的实现主线大概是：

* 服务端请求进来
* 进入 `app.get(...)` 之类的处理函数
* 在 `response` 侧开始真正渲染

然后根据运行环境选不同 API：

* Node 侧：`renderToPipeableStream`
* 边缘运行时侧：`renderToReadableStream`
* 旧式非流式：`renderToString`

### 2. 这里真正的重点是“流式渲染”而不是“字符串渲染”

老师的方向很明确：

* 现代 React SSR 更应该走 streaming
* `renderToString` 顶多算兼容性方案，不是主推路线

这节课讲的是一个很现实的工程判断：

* 你不是只想“把 HTML 拼出来”
* 而是想更早把内容流给用户
* 让页面更早变得可见

### 3. 在真正把 React 树渲染出来之前，还有一堆前置工作

老师这里强调得很清楚：

* 在开始输出 HTML 之前
* 先要把当前页面需要的 `JS / CSS / preload` 资源收集好

换句话说，一个真实 SSR 流程不是：

* `render() -> return html`

而更像：

1. 先确定当前命中页面
2. 确认它依赖哪些 chunk
3. 把这些资源标签写进 HTML
4. 再把真正的页面内容返回出去

## 三、05:25-10:43 资源收集：`@loadable` 这层到底在补什么

### 1. 只会路由懒加载还不够，服务端还要知道“命中了谁”

老师这段讲的是 SSR 非常关键、但很多人会漏掉的一层：

* 如果页面做了路由级懒加载
* 服务端必须知道当前命中的是哪个页面
* 才能把对应的 JS/CSS 一起提前塞进 HTML

否则就会出现一个典型问题：

* 主 HTML 回来了
* 但当前页面自己的 chunk 还没跟着回来
* 浏览器还得再等一次
* 首屏速度会被拉慢

### 2. 这节课用的方案是 `@loadable/component`

老师提到的完整链路大概是：

* `@loadable/component`
* `@loadable/server`
* `@loadable/webpack-plugin`
* Babel 或 SWC 的对应插件

老师把它理解成一套“服务端收集页面依赖”的机制，这个理解是对的。

它们一起做的事情大致是：

* 识别哪些地方做了懒加载
* 构建出 chunk 映射
* 生成 `loadable-stats.json`
* 服务端根据命中的路由去查这个映射
* 把需要的资源标签注回 HTML

### 3. `loadable-stats.json` 是这层机制的关键中间产物

老师现场点开了构建产物，重点其实就是这句话：

* 服务端不是凭空知道该回哪些 JS/CSS
* 它是通过 `loadable-stats.json` 这种映射文件反查出来的

这个文件记录的是：

* 某个页面 / 某条路由
* 对应依赖了哪些 chunk
* 哪些是 JS
* 哪些是 CSS

所以你会看到老师一直在说：

* 不能只是懒加载
* 要让服务端也理解这份懒加载关系

### 4. 预加载也是这里的一部分

老师顺手提到：

* 除了把脚本和样式标签塞回去
* `preload` 也很重要

这和他的目标是一致的：

* 尽可能减少浏览器二次等待
* 把“知道要加载什么”这件事尽量提前到服务端

## 四、11:26-18:43 数据预取：这节课最值得记的实现层细节

### 1. 老师这里用的是 `TanStack Query`

课堂转到数据预取时，老师用的还是大家更熟悉的旧叫法 `React Query`，但现在更准确的名字是 `TanStack Query`。

他这里想表达的其实很简单：

* 服务端先把当前页面要的数据取好
* 再把带数据的 HTML 返回给客户端
* 客户端接手后直接复用，不要再打一遍同样的请求

### 2. 最大的坑：`QueryClient` 不能做成全局单例

这是整节课最重要、也最容易在面试里失分的一个点。

老师反复强调：

* `QueryClient` 一定要放在请求处理函数里创建
* 不能挂到模块顶层做全局共享

原因很直接：

* 服务端不是浏览器
* 多个用户请求会共用同一份进程内存
* 你一旦把 `QueryClient` 放全局
* 不同用户就可能共享缓存
* 轻则内存膨胀
* 重则数据串号、缓存污染、请求混乱

这是“只写前端的人”一做 SSR 就很容易踩的坑。

### 3. 老师把 `queryKey` 和 `loadData` 绑在路由上

这节课的组织方式不是 `Next.js` 那套约定式 `getServerSideProps`，而更像：

* 路由表上自己扩展字段
* 给每个路由挂：
* `queryKey`
* `loadData`

然后组件里对应：

* `useQuery(queryKey, queryFn)`

老师非常强调：

* 路由上的 `queryKey / loadData`
* 和组件里的 `useQuery`
* 必须一一对应

不对应就会出现：

* 服务端虽然取了
* 客户端却取不到
* 或者客户端又重新发一遍请求

### 4. 为什么他不走 `getServerSideProps` 风格

老师这里给出的理由是：

* 如果完全走 `Next.js` 那种编译期抽函数的路线
* 框架会变得很重
* 需要更多编译期魔法
* 还会越来越依赖约定式路由

所以他更偏向：

* 自己保留一套路由表
* 自己决定数据预取位置
* 保留更高的自由度

这和 [[Next.js 与自定义SSR]] 的权衡完全一致：

* `Next.js` 省很多事
* 但自定义 SSR 更自由
* 同时也意味着你要自己补齐更多工程细节

## 五、27:52-31:34 脱水 / 注水：服务端取到的数据，怎么安全传给客户端

### 1. 路由匹配后，不要一个个串行 `await`

老师这里的实现思路很清楚：

* 先用 React Router 去匹配当前命中的所有路由
* 嵌套路由会匹配出一个数组
* 然后把每一层路由里定义的 `loadData` 收集起来
* 形成一组 promise

最重要的判断在这里：

* 不要一个个串行 `await`
* 也不要简单地用 `Promise.all`
* 更合理的是 `Promise.allSettled`

因为老师考虑的是：

* 某一个接口失败了
* 不应该把整个 SSR 响应直接拖死
* 至少要让服务端知道“哪些完成了，哪些失败了”

### 2. `dehydrate` 是把服务端缓存变成可传输状态

等数据预取结束后，下一步就是：

* `dehydrate(queryClient)`

这一步的本质，是把 Query 缓存转成一份可序列化的状态对象。

然后这份对象会被：

* 传给服务端渲染层
* 注入到 HTML
* 让客户端在接手时复用

### 3. `HydrationBoundary` 是客户端复用这份状态的关键

老师这里用到的就是 `HydrationBoundary`。

更准确地理解是：

* 服务端：`prefetch -> dehydrate`
* 客户端：读取初始状态 -> `HydrationBoundary` 包起来 -> `useQuery` 直接命中缓存

这样做的结果就是：

* 客户端不需要一上来重新发同一批请求
* hydration 阶段更顺
* 链路更短

### 4. 老师还提了一个很实用的性能提醒

他说自己是把 `dehydratedState` 直接序列化后塞进 HTML 里，再让客户端取出来。

但他也顺手提醒了：

* 如果状态非常大
* `JSON.stringify / JSON.parse` 会有额外开销

所以这一步的工程目标不是“能传就行”，而是：

* 状态尽量小
* 只传首屏真正需要的数据
* 不要把整个缓存树毫无节制地全塞回去

## 六、31:14-38:24 客户端接手、路由与 head 注入

### 1. 客户端真正的区别，主要就是 `hydrateRoot`

老师这里把客户端渲染说得很直白：

* 相比纯 CSR 应用
* SSR 客户端最关键的区别就是要走 `hydrateRoot`

如果是降级或纯客户端场景，才会退回 `createRoot`。

这也是 React 官方语义上的区别：

* `createRoot`：从零创建一棵客户端树
* `hydrateRoot`：接管服务端已经输出的 HTML

### 2. 路由这节课没有走“超重的约定式”，而是扩展了自己的路由表

老师这里比较强调“自由度”：

* 他没有把整套系统做成超重的编译期约定
* 而是在自己的路由表上加字段
* 给每个路由挂 `loadData` 等扩展能力

这个方案的好处是：

* 可控
* 易理解
* 改造老项目时没那么痛

### 3. 服务端路由要用 `StaticRouter`

老师旧代码里用的是 `StaticRouter` 风格。

它的目的很简单：

* 服务端不需要浏览器那套 `history`
* 只需要按当前请求 URL 做静态匹配

所以服务端和客户端路由职责不同：

* 客户端：需要 history 和交互导航
* 服务端：只需要把当前 URL 对应页面匹配出来

### 4. 做 SEO / AIO 还得补 `Helmet`

老师在这里补了 `react-helmet-async`。

这一步是为了把这些内容也跟着 SSR 回去：

* `title`
* `meta`
* 甚至 `html`、`body` 上的一些属性

所以这场课里的 SSR，并不是只渲染“页面主体”，而是完整页面语义的一部分。

## 七、36:23-38:24 服务端构建：Webpack 不等于客户端那一套照抄

### 1. 服务端构建目标和客户端构建目标不一样

老师这里提醒得很实用：

* 服务端构建不是把前端 webpack 配置原样复制一份
* 目标、入口和打包边界都不同

尤其在他这套体系里：

* 本地运行一个入口
* Serverless 运行又可能是另一个入口

### 2. Node 内建模块不要乱打进包里

老师提到 server build 要注意 `externals`，本质是在说：

* `fs`
* `path`
* `net`

这类 Node 内建能力，不应该被错误地打进浏览器侧 bundle。

### 3. 资源文件不是所有都应该进 server bundle

像：

* 图片
* SVG
* 静态资源

在 server build 里要区别对待，别按客户端那套全盘打包。

这一段虽然讲得快，但其实是老项目里经常出问题的地方：

* server bundle 目标写错
* asset 处理规则混了
* 结果运行时才发现包不对、入口不对、依赖也不对

## 八、41:22-44:24 选择性水合、`startTransition` 和 AI 编程插曲

### 1. 老师对“选择性水合”的态度比较克制

他这里的结论很明确：

* 这个概念社区吵了很多年
* 新版 React 已经比早年成熟很多
* 但对复杂应用而言，它不是最大的性能杠杆

老师更看重的是：

* 部署方式
* 边缘节点
* 交互切片
* 真正的渲染瓶颈

### 2. `startTransition` 在他的语境里更像“交互优化手段”

他说它更偏向：

* 优化交互体验
* 提升 `INP`
* 避免把高优先级交互阻塞住

所以这里的判断不是“不要学选择性水合”，而是：

* 先抓大头
* 对复杂业务页，部署和结构经常比花哨特性更重要

### 3. 课堂插曲：AI 已经显著降低自研框架和陌生库的门槛

这节课中间有大量插曲都在讲同一件事：

* 以前自研一套 SSR 框架很重
* 现在通过 `Claude Code / Cursor / 测试驱动 + AI`，成本低了很多
* 老师还举了社区有人用 AI 对着 `Next.js` 测试集去做兼容实现的案例

这段真正有价值的结论是：

* AI 可以极大降低“陌生框架”和“复杂工具链”的门槛
* 但这不意味着你可以放弃架构控制
* 你仍然要约束：
* prompt
* skill
* 代码组织
* 设计边界

否则 AI 很容易把代码写散。

## 九、44:24-01:00:50 为什么老师后半节一直在强推 Serverless

### 1. 他认为前端做 Node SSR，迟早会被问到“高并发怎么扛”

老师后半节的逻辑非常明确：

* 你简历写了熟悉 Node.js
* 写了部署过真实 SSR 应用
* 面试官就一定会问：
* 你们怎么扩容
* 怎么扛并发
* 怎么算成本

而这恰好是很多人最空的一层。

### 2. SSR 本身其实还是个相对简单的单体服务

老师把 SSR 和复杂后端系统区分得很明显：

* SSR 主要是接请求
* 取数据
* 渲染页面
* 返回客户端

它没有那些特别复杂的后端链路：

* 消息队列
* 多级异步任务
* 多种数据库组合
* 实时分析流水线

所以他的潜台词是：

* 正因为 SSR 没那么复杂
* 更应该把部署层做好

### 3. 单机 / 固定机器的上限很快会出现

老师给了几个很具体的口径：

* 他自己压测里，一个 `4C8G` 节点，大概也就 `300~400 QPS`
* 如果你要到 `10000 QPS`
* 粗略算下来就可能需要 `25` 台左右

他还举了一个非常夸张但有冲击力的例子：

* 某电商类高流量 SSR 业务，用了几百台物理服务器顶 Node SSR

无论这个案例细节如何，它要表达的就是：

* 只靠堆固定机器，是能做，但代价很高

### 4. 老师对 `Vercel` 的判断

这节课里老师对 `Vercel` 的看法比较鲜明：

* 它对个人项目、快速起步、小团队非常友好
* 但成熟公司会更在意：
* 成本
* 平台绑定
* 兼容层
* 私有化与多云部署

所以他才会顺手提到：

* `OpenNext`
* `Cloudflare`
* `AWS`

也就是把问题从“怎么部署 Next.js”转成“怎么把 SSR 放到更灵活的平台上”。

## 十、01:00:50-结束 Serverless 真正吸引老师的四件事

### 1. 自动扩容和自动缩容

这是老师最强调的一点：

* 流量来了，自动扩容
* 流量下去了，自动缩容
* 甚至缩到接近零实例

相比之下，传统：

* `EC2 / ECS / EKS`
* 虽然也能做扩缩容
* 但规则配置、最小实例、维护成本都更麻烦

### 2. 平台自带监控、日志、告警

老师把 `Serverless` 的另一个价值说得很现实：

* 不是只有算力弹性
* 还包括平台化的可观测能力

比如：

* 监控
* 日志
* 告警
* 容灾相关组件

这些在传统自建模式里都要自己补很多。

### 3. 按量计费，且对很多 SSR 场景非常便宜

老师反复强调：

* Serverless 的典型计费维度就是：
* 调用次数
* 执行时长
* 内存

他给了几组非常强的经验值：

* 稳定高 QPS 的场景，Serverless 成本依然可能远低于固定机器
* 某些真实业务从接近 `$1000/月` 降到几十美元级别
* 公司整体年云成本也可能因此降很多

这些数字具体会受：

* 峰值流量
* 执行时长
* 内存规格
* 出网流量
* 缓存命中率
* 厂商折扣

影响，但他的结论很稳定：

* 至少对“短请求、无状态、SSR/API 网关”这类场景，Serverless 经常非常划算

### 4. 结合 `IaC`，上线速度会非常快

老师这里说了一个非常重要的工程收益：

* 如果你把基础设施也代码化
* 那一个新项目从 0 到 1 的上线速度会快很多

他拿旧模式和新模式做了对比：

* 旧模式：申请资源、手工点控制台、手工配环境、手工扩容规则
* 新模式：用模板和 `IaC` 创建一整套环境，几分钟就能拉起来

这对：

* 小团队
* 高速试错
* 多业务线快速试新站点

非常关键。

## 十一、老师也明确讲了 Serverless 的边界

### 1. 不适合长连接和特别长的任务

老师这里讲得很直接：

* `Lambda / Workers / 各类 FaaS`
* 都有执行时长限制
* 长连接也不是它们最舒服的工作模式

所以更适合的场景是：

* SSR
* API 网关转发
* 短请求
* 短时函数编排

### 2. 冷启动依然是必须面对的问题

虽然这节课没有展开太细，但老师已经点到了：

* 冷启动
* 语言运行时差异
* AOT / JIT 对启动成本的影响

这里需要做一个更准确的整理：

* 老师是在表达“运行时特性会影响冷启动表现”
* 尤其是重量级运行时，冷启动更敏感
* 他把 `Java` 放在“不那么适合 Serverless 首选”的位置

即便不展开到语言大战，结论也足够明确：

* 你不能只看“能跑”
* 还要看启动成本、时长限制和真实请求模型

## 十二、这场最后真正想提醒你的，不只是技术

### 1. 真正难的往往不是代码，而是推动组织接受它

老师最后一段其实非常现实：

* 推 `Serverless`
* 推 Node SSR 高并发
* 推前端接近基础设施的能力

很多时候最大的阻力不是代码，而是：

* 运维权限
* 安全审核
* 后端团队既有认知
* 组织分工边界

### 2. 不要越权，但要把方案讲透

老师的建议可以整理成：

* 不要直接替别人拍板
* 不要越过运维、安全、后端的职责边界
* 但如果你真懂这套东西，就要能：
* 讲清方案
* 给出落地路径
* 手把手协助推进

这其实比“会写一版 demo”更接近真实工作价值。

### 3. 小公司更容易先跑起来

老师最后的判断是：

* 小公司包袱少
* 人少
* 权限链路短
* 更适合快速把新架构跑起来

而大公司、强合规行业、复杂部门协作环境里，技术正确不等于推进就容易。

## 这节课最值得带去面试的回答

如果面试官问你：

### 1. `SSR` 主要解决什么问题？

可以答：

* 第一是 [[AIO 与 SEO]]
* 第二是性能，尤其是首屏可见内容更早返回
* 但实际体验还受 `TTFB`、节点部署、hydration 成本和缓存策略影响

### 2. 一个 React SSR 项目除了渲染还要补什么？

可以答：

* 路由匹配
* chunk 收集
* preload
* 数据预取
* `dehydrate / hydrate`
* `hydrateRoot`
* head / meta 注入
* 服务端构建和部署

### 3. 服务端数据预取最容易踩什么坑？

可以答：

* 不要把 `QueryClient` 做成全局单例
* 要按请求创建
* 路由上的 `queryKey / loadData` 和组件里的 `useQuery` 要一一对应
* 并发预取时更适合 `Promise.allSettled`

### 4. 为什么很多团队会把 SSR 部署到 Serverless？

可以答：

* 为了弹性扩缩容
* 为了降低固定机器成本
* 为了更快上线新项目
* 为了拿到平台级监控、日志、告警
* 但前提是业务适合短请求、无状态和有限执行时长
