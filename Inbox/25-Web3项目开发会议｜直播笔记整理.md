# 25-Web3项目开发会议｜直播笔记整理

## 概览

这场直播虽然标题叫“Web3 项目开发会议”，但真正讲的不是单纯“继续念概念”，而是把一条最小 Web3 项目链路拆开给大家看：

1. 先部署一个最小 `ERC20`，再顺手提到 `ERC721`
2. 再去读 `OpenZeppelin` 风格实现，理解 `IERC20`、`approve`、`allowance`、`transferFrom`
3. 然后把它收束成一个“用 `ERC20` 资产购买商品”的最小 DApp 需求
4. 中间又插进了大量企业 AI 工作流、[[Cloudflare Tunnel]]、`Zero Trust`、JIRA、Figma 还原和 AI 自动化经验

这场最重要的收获不是“会不会发一个币”，而是三件事：

* 你要知道 `ERC20` 购买流程到底怎么跑
* 你要知道 `approve` 为什么危险、为什么又绕不过去
* 你要知道现在企业里最值钱的不只是写代码，而是把 AI、文档、产品 review 和执行流程接起来

## 这场里需要先纠正的错词和概念

### 高概率错词

* `12320 / 压缩20 / ERS20 / IES20 / ERC30` 基本都是 `ERC20 / IERC20`
* `open zlink / open Zlink` 是 `OpenZeppelin`
* `Trap / Truff / traef` 基本是 `Truffle`
* `headhunt / headhatch` 基本是 `Hardhat`
* `easers / Easter / easters` 基本是 `ethers / ethers.js`
* `type train / typestring` 是 `TypeChain`
* `webmi / 挖个米` 是 `wagmi`
* `wait` 是 `Vite`
* `SSP` 高概率是 `Sepolia`
* `closeair` 是 `Cloudflare`
* `bytework / byterock / badrock` 是 `AWS Bedrock`
* `cloud watch` 是 `CloudWatch`
* `message sender` 是 `msg.sender`
* `symbol / sybole` 是 `symbol`
* `Allance / alless` 是 `allowance`

### 需要纠正的几个概念

* `ERC20` 接口本身不定义 `mint`，`mint / burn` 是实现方或扩展层去补
* `approve` 不是“免密支付”本身，而是“授予某个合约花你某种代币额度的权限”；“免密”只是帮助理解的类比
* `allowance` 的本质是 `owner -> spender -> amount` 的授权映射
* `ETH` 这种原生资产可以直接转，`ERC20` 资产若要由别人代扣，通常就要先 `approve`
* `ERC721` 不是“钱”，更准确地说是“唯一 tokenId 的所有权记录”
* `ERC1155` 也不是简单的“ERC20 + ERC721 合体”，更准确地说是多 token 标准
* `Zero address` 不是 JS 里的 `undefined`，它是链上约定俗成的特殊哨兵地址
* `burn` 在标准语义里常表现为“转到 Zero address”，但实现层通常不是单纯把余额真存进去，而是同步减少 `totalSupply`
* `decimals` 影响的是展示精度，不会自动让币“更值钱”
* 测试网不要直接把主网稳定币地址生搬过去，教学里更稳的是自部署 `mock ERC20`

## 核心结论

* 最小的 `ERC20` 商品购买流程就是：商品定价 -> 用户持币 -> `approve` -> `buy` -> 展示购买结果
* 真正该先啃的不是手写整套标准，而是先看懂 `OpenZeppelin` 这类成熟实现的接口和核心逻辑
* `approve` 是 `ERC20` 里最常见也最危险的一环，面试和实战都会反复问
* DApp 前端至少要补两套能力：`wagmi + viem` 的现代路线，以及 `ethers v6 + TypeChain` 的基本功
* 企业 AI 工作流最关键的不是“能生成”，而是需求拆解、产品 review、文档沉淀和无人值守执行
* `Cloudflare + AWS + Zero Trust + Tunnel + Bedrock + CDP` 这类能力，在老师的语境里已经不是炫技，而是企业开发基础设施
* 简历上更值得写的是“企业 AI 提效”和“自动化落地”，不是只写“会 prompt、会用 AI”

## 一、00:03-15:30 从发 ERC20 到本地钱包导入

### 1. 这节课一开始就在讲最小购买流程

老师开场直接把流程说得很清楚：

* 商品定价
* 用户先 `approve`
* 然后再购买
* 最后再查询你授权了多少 token、是否购买成功

这说明这场课真正要落的是：

* 一个最小 Web3 商品购买闭环

而不只是：

* 再讲一遍代币标准名词

### 2. “Web3 大学”本质上是一个课程练手项目

老师把这个最小项目继续沿用了以前那套“Web3 大学”练习题思路：

* 部署 `ERC20`
* 再提一下 `ERC721`
* 最后把这些能力塞进一个能购买商品的 DApp

这里更适合理解成课程练手框架：

* 不是成熟商业系统
* 是为了把钱包、合约、前端交互、购买流程串起来

### 3. 部署代币时老师默认用了成熟实现思路

这一段最值得记住的是：

* 不一定要自己从零写标准实现
* 实战里更常见的是直接用 `OpenZeppelin` 的 `contracts`

老师现场的意思其实很现实：

* 你当然可以手写
* 但如果要省事、要更稳、要少踩坑，直接基于成熟实现更常见

### 4. ERC20 最小自定义点有哪些

老师现场带着大家改的几个最小点，大致是：

* token 全名 `name`
* token 简称 `symbol`
* 初始发行量
* `decimals`
* 初始 `mint`

其中最重要的不是语法，而是后面的经济含义：

* 初始发行量不是随手乱填
* 增发也不是想增就增
* 这些会直接影响用户对项目和代币的信任

### 5. 这里要补一个概念纠偏：价值不来自“改个 decimal”

老师这段里把 `decimals`、发行量、增发、流通价值揉在一起讲了。

更准确地说：

* `decimals` 只是显示单位
* 价值来自用途、流动性、分配方式、信用、交易深度和市场预期
* 发币很容易，但“让别人愿意持有或使用”完全是另一回事

所以这段最应该记成：

* `ERC20` 发出来很快
* 真正难的是通证经济和市场接受度

### 6. 本地链和钱包导入这一步，老师是在演示最小环境

这部分大致是：

* 本地链没开，`7545` 就连不上
* 开本地链之后再部署合约
* 再把本地账户导入钱包
* 最后把自定义 token 加进钱包里

这里高概率对应的就是：

* [[Ganache]] 或类似本地链
* 钱包切到 `localhost:7545`
* 再导入测试账户和 token 地址

这个链路很适合和 [[Web3钱包与测试网]] 一起看。

### 7. ERC721 在这段里只是被当作“下一类资产”带过

老师提到：

* `ERC20`
* `ERC721`

他这里不是要深讲 NFT，而是要先建立一个基本区分：

* `ERC20` 更像同质化资产
* `ERC721` 更像独一份所有权

## 二、16:03-31:31 从 OpenZeppelin 源码看 ERC20 到底怎么长

### 1. 老师这段最值钱的点，是带大家去看源码结构

他没有停留在“会用就行”，而是顺着 `OpenZeppelin/contracts/token/ERC20` 往里找：

* `ERC20.sol`
* `IERC20.sol`
* `IERC20Metadata.sol`
* `Context.sol`
* `draft-IERC6093.sol`

这一步很重要，因为它把“会用标准”推进到了：

* 理解标准接口
* 理解实现层
* 理解扩展和错误处理

### 2. `IERC20` 真正定义的东西很少

老师把接口扒开之后，想说明的是：

* `IERC20` 本身就那几个核心方法和事件

也就是常见的：

* `totalSupply`
* `balanceOf`
* `transfer`
* `allowance`
* `approve`
* `transferFrom`

以及两个核心事件：

* `Transfer`
* `Approval`

这段最值得记住的纠偏是：

* `mint` 不在接口里
* `burn` 也不在接口里
* 它们是更具体实现里的扩展能力

### 3. 元数据扩展只是补 `name / symbol / decimals`

老师后面继续看 `IERC20Metadata` 时，真正想说明的是：

* 元数据扩展就是把名字、简称、精度这类信息补上

所以它不是核心转账规则，而是：

* 展示层 / 描述层补充

### 4. `abstract / virtual / override` 这块就是 Solidity 的“抽象类 + 重写”

这段老师反复在讲：

* 抽象合约
* 虚方法
* 重写

用更日常的话说就是：

* 框架先把骨架给你
* 真正项目里你再补具体逻辑

这和 TS / OOP 的抽象类思路非常像，所以老师一直在说：

* `Solidity` 语法本身没有那么难

### 5. `draft-IERC6093` 这块不要记错

文字稿里把它讲得有点乱。

更稳妥的记法是：

* 这是 `OpenZeppelin` 引入的 `draft-IERC6093` 自定义错误接口草案
* 主要是为了规范一些 `ERC20 / ERC721 / ERC1155` 相关错误的表达方式

不要把它简单记成：

* “又一个 ERC 标准”

更准确地说，它更偏：

* 错误接口 / 错误表达约定

## 三、23:53-31:31 approve、allowance 和 transferFrom

### 1. 老师把 `approve` 类比成“免密支付”，这个类比有用但不能照字面理解

他现场用了支付宝免密来帮助大家理解，意思是：

* 你不是直接花钱
* 你是先授权一个对象以后可以在一定额度内替你花

这对入门很有帮助，但要补一层更准确的定义：

* `approve` = 授权某个 `spender` 可以支配你某种 `ERC20` 的一定额度

### 2. `owner -> spender -> amount` 才是 allowance 的本质

老师这段最重要的点其实是：

* `allowance` 本质就是一个二维授权表

可以直接记成：

* `owner`：谁的钱
* `spender`：谁被允许花
* `amount`：能花多少

也就是：

* `mapping(address => mapping(address => uint256))`

### 3. `approve` 最大的风险是无限授权

老师在这段里强调得很猛，这点是对的：

* 额度给大了很危险
* 无限额度尤其危险

这和 [[approve 风险]]、[[无限授权]]、[[钱包弹窗审查]] 是直接连着的。

最值得记的不是“别授权”，而是：

* 你要看这次交易到底需要多少
* 你授权给了谁
* 你是不是授权了远大于实际需要的额度

### 4. `transfer` 和 `transferFrom` 不是一回事

老师这里其实已经讲清楚了：

* `transfer` 更像“我自己把钱转出去”
* `transferFrom` 更像“别人基于我给的授权，从我这扣钱再转给别人”

所以在商品购买这类 DApp 里常见的链路就是：

1. 先 `approve`
2. 再由合约 `transferFrom`

### 5. 无限授权为什么老被用

老师现场讲了一个核心细节：

* 如果当前额度等于 `type(uint256).max`
* 那就等同于“无限授权”

这也是很多产品里常见的做法：

* 少让用户反复签名

但副作用也很明显：

* 如果合约有问题，风险会被直接放大

### 6. ETH 为什么不需要 approve

老师这段想强调的是：

* `approve` 是 `ERC20` 资产语境
* `ETH` 这种原生 gas 资产不是这样走的

所以不要把所有链上支付都理解成：

* 必须先 `approve`

更准确地说：

* “需要先 `approve` 再消费”是典型 `ERC20` 支付模式

## 五、01:03:29-01:28:08 继续剖 ERC20：Zero address、burn、_update、_spendAllowance

### 1. 这一段是在把老师前面“会用”推进到“看懂实现”

老师回到源码以后，真正把几个关键点讲开了：

* `Context`
* `_msgSender()`
* 自定义错误
* `unchecked`
* `_update`
* `_approve`
* `_spendAllowance`

这些地方其实就是面试和读源码时最容易卡住的点。

### 2. `_msgSender()` 是对 `msg.sender` 的一层封装

老师说“message sender 包装了一下”，这点方向是对的。

更准确地说：

* `Context` 把 `msg.sender` 和 `msg.data` 包了一层
* 这样后面做元交易等扩展时更灵活

### 3. `Zero address` 是哨兵地址，不是 JS 的 `undefined`

老师想用 `undefined` 类比它，但这个类比不能完全照搬。

更稳妥的理解是：

* `0x000...000` 是链上约定俗成的特殊地址
* 常用于表示：
  * mint 的来源
  * burn 的去向
  * 非法目标检查

所以它更像：

* 语义上的特殊占位符

而不是：

* 一个普通能正常持有资金的用户账户

### 4. `burn` 不是简单“真把钱打给零地址”

老师这段口头里把它讲成“转到 Zero address”，这个对事件语义是成立的，但实现层要再精确一点：

* 在很多标准实现里，`burn` 会表现为 `to = Zero address`
* 但真正逻辑往往是：
  * 减少账户余额
  * 同时减少 `totalSupply`

所以别把它机械理解成：

* “只是普通转账，只不过转给了一个没人能拿到的地址”

### 5. `_update` 是新版实现里很核心的记账入口

老师这里其实抓到了关键点：

* 许多 mint / burn / transfer 最后都会收束到一套统一的余额和总量更新逻辑

这意味着：

* 你读合约源码时不要只盯着 `transfer`
* 要看底层记账究竟汇总到哪

### 6. `unchecked` 不是乱来，而是“前面确认过后省 gas”

老师后面专门让 AI “说人话”，这个解释方向是对的：

* 如果前面已经做过不会溢出的检查
* 后面某些局部计算就可以省去重复检查

这本质上是：

* 用确定性换 `gas` 优化

### 7. `ERC721` 这段最该记住的纠偏

老师这里想表达的是：

* `ERC721` 更像转移一个唯一 `tokenId`

这比“转钱”更接近事实。

更准确地说：

* 你转的是唯一标识和其所有权
* 价值来自这个 `tokenId` 所代表的资产、权益或元数据

## 六、01:31:39-01:45:38 最小商品购买 DApp 需求

### 1. 老师把产品分成两类：纯 DApp 和半中心化

他的分类大致是：

* 纯 DApp：更多规则、商品、定价都放链上
* 半中心化：有后台，部分数据或管理逻辑放服务器

这点很有价值，因为真实项目里并不是所有东西都必须上链。

### 2. 最小商品场景就是“卖一个 200U 的产品”

老师用的例子很具体：

* 一个商品卖 `200U`

然后围绕它把完整路径展开：

* 商品怎么添加
* 价格放哪里
* 用户怎么买
* 先授权还是先买
* 买完如何展示

### 3. 商品添加和定价应该受 owner 控制

老师这里给出的最小约束是：

* 合约里可以添加商品
* 商品有定价
* 但只能 `owner` 添加

这背后的原因很简单：

* 如果谁都能上架和定价，系统就会乱掉

### 4. 测试环境里，更稳的做法是先用 mock ERC20

老师中间在“USDT 合约地址”这段里说得比较乱。

更稳的整理方式是：

* 真做练习时，优先自部署一个 `mock ERC20`
* 或使用明确可用的测试资产
* 不要把主网 token 地址和测试网直接混着用

换句话说，课程作业更适合这样理解：

* 先把“资产支付流程”跑通
* 不要把时间耗在纠结某个主网稳定币地址上

### 5. 最小购买流程应该长这样

老师这段最关键的业务链路可以压成：

1. 合约里有商品和价格
2. DApp 展示商品
3. 用户钱包里已经有支付用的 `ERC20`
4. 点击购买时先检查余额 / 授权
5. 不够就先 `approve`
6. 授权后执行 `buy`
7. 购买成功后再展示“已购买商品”

这里最值得记住的是：

* `approve` 和 `buy` 是两个动作
* UI 上也要把这个过程分开表达

### 6. 老师对兑换这步的处理很现实

他提到了两种路线：

* 用户自己先在钱包或现成 swap 平台兑换好资产
* 更深入时，再自己做池子或走 `Uniswap` 这类 AMM 思路

但他当场给出的教学取舍很明确：

* 先不把复杂度拉高
* 用户先自己持有支付资产
* 课程先聚焦 `approve + buy`

### 7. 这就是一条非常典型的面试 / 作业题

老师后面其实已经把它写成了一个标准作业需求：

* 合约添加商品并定价
* 前端列出商品
* 点击购买
* 先授权再购买
* 不够则置灰
* 买完有展示

这类题之所以常见，是因为它正好覆盖了：

* 钱包连接
* 资产读取
* 授权
* 合约调用
* 前端状态

## 七、作业、技术栈和求职方向

### 1. 今日作业和明日作业其实分成了两条线

老师最后布置的作业大致有两类：

* 今日作业：继续用现有 `SPA` 架构把红包系统补完整
* 明日作业：做一个 `ERC20` 购买商品项目

同时他还提到另一个作业，转写里写成了“供应链”，高概率其实是在说：

* 自己开发一个公链

这和上一场的语境是连着的。

### 2. 今日红包作业强调的是 `ethers v6 + TypeChain`

老师这段说得很明确：

* 虽然现在大家更常提 `wagmi`
* 但 `ethers v6 + TypeChain` 仍然是基本功

因为它解决的是很实际的问题：

* 不再到处 `any`
* 能生成类型
* 能生成工厂
* 能更顺地实例化合约

### 3. 明日购买 DApp 作业的推荐栈

老师给出的推荐大致是：

* 合约侧：`Hardhat`
* 前端侧：`Vite + TypeScript + wagmi`

再结合他前面提到的能力，可以补成更完整的一句话：

* 用 `Hardhat` 管合约、用 `Vite + TS + wagmi` 做 DApp 前端，再把购买流程接通

### 4. TypeChain 这段一定要记住“为什么”

老师后面明显有点着急，因为同学忘了上午讲过什么。

他真正想强调的是：

* 不是为了背一个库名
* 而是要明白它解决的是“类型缺失”和“实例化繁琐”

也就是：

* 以前 `new Contract` 很容易全是 `any`
* 现在可以直接拿到更明确的 TS 类型和工厂封装

### 5. 求职上，老师最想让大家突出的是“企业 AI 提效”

最后一段虽然已经不在讲 Web3 代码，但职业判断很鲜明：

* 简历不要只写会模型名
* 更值得写的是：
  * 企业提效
  * 自动化
  * AI 增效
  * 能帮传统企业把流程拉起来

他给出的现实画像也很典型：

* 老板可能很认可 AI
* 但他自己并不懂怎么用
* 真正把问题落地的人，价值就在这里

这和 [[企业落地判断]] 是完全一致的：

* 你不是去当“会聊天的人”
* 而是去当“能把 AI 变成效益的人”
