# 09-AI技术应用研讨会｜直播笔记整理

## 概览

这场直播表面上叫“AI 技术应用研讨会”，但真正的主线其实是把：

* [[02-AI时代开发相关讨论｜直播笔记整理]]
* [[06-技术就业AWS讨论｜直播笔记整理]]
* [[08-前端项目相关讨论｜直播笔记整理]]
* [[AI开发主流程]]
* [[课程路线图]]

这几条线正式接成一条“企业考核 + 项目实战 + AIGC 变现”路线。

老师这场重点讲了三件事：

1. 企业现在怎么考核“你到底会不会用 AI 干活”
2. 这一期课程为什么开始进入 [[AIGC 与 AI视频生成]] 实战
3. 真实项目要怎么从需求、设计、UI 到实现一步步做出来

> [!note]
> 这份整理按主线重构。文字稿里有不少发音错误和转写漂移，尤其集中在模型名、AIGC 工具名和视频生成平台名。凡是不够确定的，我都统一挂到 [[风险与疑似错词]]，没有硬改成“绝对结论”。

## 核心结论

这场直播最值得留下来的判断有几条：

* 企业现在已经不再只问“你用过哪个模型”，而是直接考核你能不能把需求落成结果
* 以后更值钱的，不只是会写代码，而是会写结构化提示词、会验证、会 review、会兜风险
* AIGC 方向已经不是玩具，而是可以直接变成真实产品和副业方向
* 生成式视频链路已经越来越成熟，但版权、审核、排队、成本和稳定性仍然是硬限制
* 课程线已经从“会用 AI”推进到“基于 AI 做真实项目”

## 一、老师先讲的不是工具，而是企业怎么考核 AI 能力

老师开场花了很大篇幅在讲他最近做企业培训时的考核标准。

这部分其实很重要，因为它把“会不会用 AI”从口头表达，变成了非常具体的能力要求。

### 企业考核想看什么

老师给出的企业标准，大致包括：

* 能把自然语言需求转换成结构化提示词
* 能用 AI 生成符合要求的代码
* 能识别 AI 输出中的缺陷、遗漏和高风险点
* 能自己做验证、缺陷修复和回归
* 能对自己和 AI 生成的代码做有效 `code review`
* 能解释设计思路、实现方法、编辑过程和验证方式

### 不达标的典型表现

老师讲得非常直接：

* 只会复制 AI 输出
* 解释不清代码逻辑
* 不会验证 AI 结果对不对
* 没有测试和异常处理意识
* 识别不出明显风险
* 在限制时间里跑不通“提示词到代码运行”的闭环

这其实已经不是“工具使用题”，而是非常像真实交付能力测试了。

### 这场背后对应的现实变化

老师提到：

* 现场实操已经开始成为企业考核和面试的一部分
* 不同行业都在加速往这条线靠
* 哪怕是传统岗位，也开始要求你能把设计、PRD、代码、测试、运维串起来说清楚

这部分和 [[企业落地判断]] 可以直接放在一起看。

### 这场里提到的考核细则

老师这次讲得非常像正式制度，里面有几个细节不要漏：

* 适用岗位不只是前端，还包括：
  * 后端工程师
  * 测试工程师
  * 运维工程师
  * 技术负责人
  * 架构师
  * 研发经理
* 现场会给一个真实需求
* 限时大约 `180` 分钟
* 采用现场实操、结果评测和评审结合的形式
* 老板会直接参与

老师提到的评分思路里，至少明确出现了这些维度：

* 提示词设计能力
* 功能实现完成度
* 风险识别
* 测试与回归
* code review 能力
* 对本人方案和代码的解释能力

老师还强调了“硬性红线”：

* 核心功能跑不起来
* 测试完全失真
* 明显高风险问题没识别出来
* 无法解释自己提交的核心代码逻辑
* 直接提交 AI 输出且没有人工判断

也就是说，这种考核不是在看“你会不会聊天”，而是在看你能不能独立把控交付质量。

## 二、老师对不同模型的分工判断

这场直播里，老师没有把模型当成“谁最强就完了”，而是更强调角色分工。

他提到的大致判断是：

* 某些模型写得快，但考虑不全
* 某些模型慢一些，但更完整
* Gemini 一类模型在前端方向更有优势
* 不同模型适合承担不同任务

他真正想表达的还是那一句：

> 最好的方式不是迷信一个模型，而是让工作流去调度多个模型。

这和前几场里他讲的“一个写、一个 review、一个看流程”的思路是完全一致的。

## 三、这场开始把“AI 会不会用”拉到了求职和面试上

老师举了最近同学面试的例子，里面已经不是传统的八股题，而是：

* 现场给需求
* 不限制你用什么 AI 工具
* 老板直接看你怎么拆、怎么做、怎么验证

老师想表达的是：

* 现在的面试越来越像真实工作演示
* 真正拉开差距的，不只是会不会某个 API
* 而是你能不能把一套方案从设计到落地摆在桌面上

这也和他一开始给企业做的 AI 能力考核是一回事：

> 会不会用 AI，不是嘴上说“我会”，而是当场做出来。

### 直播里出现的面试问题样例

老师举的同学面试题，信息量其实很大，大致包括：

* 团队规模与分工
* 项目是不是从 `0 -> 1`
* 架构体系怎么设计
* 为什么选 `Node.js`
* 为什么选某个数据库
* 工单系统的用户量和使用场景
* `RAG` 里的前期文档解析流程
* 前置数据来自哪里
* 文档拆分后用什么 `embedding`
* 召回阶段怎么做
* 底层调用的是什么大模型
* 本地 `DeepSeek` 怎么部署
* 调官方 API 还是自建 API
* AI 编程工具进入前端工程后，如何约束生成代码
* 非前端同事也在用 AI 写前端代码时，作为前端负责人如何防止代码被“搞烂”
* 面向海外 C 端用户时，浏览器兼容、性能、国际化和 `CI/CD` 怎么处理

这类题已经不是传统意义上的“前端八股”，而是非常靠近真实业务和全链路交付。

## 五、这次真实项目到底是什么

这场最重要的实战部分，是老师拿出了一个真实项目：

* 项目落地场景在日本景点
* 主要面向大阪和东京
* 每个景点放一个二维码
* 用户扫码进入 H5
* 上传头像、语音和现场照片
* 系统结合景点已有的照片 / 视频素材
* 把用户“克隆”进景点纪念视频里
* 最后生成一个约 15 秒的视频，用户付费下载

### 老师给出的产品逻辑

大致链路是：

1. 用户扫码进入
2. 登录
3. 上传头像和语音
4. 采集或上传现场照片
5. 系统自动定位当前景点
6. 用户选择景点素材
7. 点击生成
8. 支付
9. 进入个人中心查看和下载生成结果

这不是一个练习用假题，而是老师明确说了：

* 产品方案已经聊完
* 日本那边有人在谈景点合作
* 做完之后就是准备上线的真实项目

所以这场的价值很大一部分在于：

> 它把“AI 做内容”从概念，直接推到了有场景、有付费、有交付目标的真实产品。

### 产品页结构这次其实讲得挺细

老师现场梳理出来的页面，大致是：

* 登录页
  * 支持 Google 登录
  * 也支持用户名密码登录
* 工作区
  * 上传头像 `photo`
  * 上传或录制语音 `voice`
  * 上传现场照片
  * 根据定位自动识别当前景点
  * 选择景点已有的素材
* 生成页
  * 展示生成中的视频结果
* 支付页
  * 付费后才能下载
* 个人中心
  * 查看自己生成过的项目和视频
* 官网 / H5 入口
  * 扫码用户从这里进入

老师后面在演示 UI 调整时，甚至细到：

* 顶部背景图怎么换
* 底部 `Tab` 怎么统一
* 哪些按钮和返回箭头没必要保留
* 哪些图只是贴图，不一定要作为正式设计资产保留

## 六、老师对 AIGC 行业链条的理解

这一段可以直接沉淀到 [[AIGC 与 AI视频生成]]。

老师讲得很清楚：

* 现在短剧、漫剧、AI 视频内容方向非常火
* 做这类内容不是只有一个“点按钮”的岗位
* 它背后有完整的分工链

### 他提到的典型分工

* 写剧本的人
* 导演
* AE 动效组
* 剪辑和后期
* AIGC 生图师
* AIGC 视频生成师

老师的意思不是说这行完全没有门槛，而是：

* 技术门槛正在下降
* 创意、细腻度、审美、节奏感和爽点设计仍然很重要

## 七、老师对 AIGC 工具链的判断

这场直播里，老师提到的工具和平台比较多，但主线很清楚。

### 1. 生成图像和视频的主力链路

从老师的口径看，这条线大致是：

* 生图
* 再根据图生成视频
* 再结合语音、剧本、镜头拆分继续加工

他提到的高频词包括：

* 即梦
* Seedance
* Sora
* Midjourney
* Stable Diffusion / ControlNet
* Nano Banana

### 2. 老师最看重的能力变化

他特别强调：

* 以前要学很多电影镜头语言、打光、摇镜、专业术语
* 现在更先进的视频模型已经能把很多东西自动补上
* 提示词还重要，但已经不必像以前那样全靠专业术语硬写

### 3. 真实限制也很明显

老师没有把这条线讲得很轻松，他明确点了几个现实问题：

* API 很贵
* 某些平台企业版排队非常夸张
* 审核会卡住
* 版权会卡住
* 不告诉你为什么失败会很烦

也就是说，AIGC 现在很强，但远没有到“随便一做就稳定量产”的程度。

### 他现场讲到的提示词和生成细节

这一段其实很值得留：

* 常见流程不是直接“生视频”，而是：
  * 先让大模型写生图提示词
  * 图片不要太多，老师示例里控制在 `5` 张以内
  * 再让大模型基于图片写 `10~15` 秒短视频剧本
  * 再按镜头一、镜头二、镜头三去对应素材
* 他提到过：
  * 正向提示词
  * 反向提示词
  * 反向提示词主要用于规避多余元素，比如多手指这类问题
* 对某些生图工具，还会有：
  * `fast`
  * `pro`
 这种模式选择
* `pro` 更慢，但一般会更稳
* 对话式工具比传统参数型工具简单，但可控性有时不如老工作流

老师真正想讲的是：

* 今天的门槛并不是“完全不需要提示词”
* 而是提示词重点已经从纯技术参数，转向题材、结构、镜头和结果控制

## 八、老师专门区分了 AIGC 和 AGI

直播里老师专门解释了一次：

* AIGC 是 AI 生成内容
* 重点是生成文本、图片、视频、代码等内容
* AGI 则是通用人工智能那个方向

这段转写里有一些明显口误，但老师真正想表达的是：

* 找工作时，公司说自己做的是 AIGC 还是 AGI，不是一回事
* 你自己要能分清“内容生成型产品”和“更通用的模型 / agent / 平台能力”

## 九、老师现场演示的不是“从零设计美学”，而是快速把 UI 拉到可用

后半段老师花了很长时间在现场梳理这个项目的产品页面。

这部分本质上是在继续推进 [[前端新时代开发工作流]]：

* 先生成页面
* 再对页面做拼接和调整
* 再补齐登录、工作区、支付、个人中心
* 再决定是否导入 Figma 或直接转代码

### 这段的核心方法

老师一直在重复一句话：

* 不会串？那就继续打字

他的意思不是在开玩笑，而是在表达：

* 现在很多 UI 和页面问题，本质上已经变成“你会不会把需求说清楚”
* 先把大的框架做出来
* 小的地方再去局部修
* 没必要一上来就执着于完全手工抠

### 页面结构大致包括

* 登录页
* 上传头像 / 语音 / 照片的工作区
* 景点定位和素材选择
* 生成结果页
* 支付
* 个人中心 / 我的项目 / profile
* 项目官网或 H5 入口

老师想表达的是：

> 对今天的 AI 来说，把第一版 UI 快速搭出来，已经不再是最难的部分。

## 十、这场里最现实的一条线：副业、接单和变现

老师反复提到，现在 AIGC 方向不只是求职线，也是一条非常实际的副业线。

他讲的变现方式大致有几类：

* 做短剧 / 漫剧
* 前几集免费，后续付费
* 靠平台分成
* 靠粉丝积累后挂车
* 做垂直内容号
* 接 AI 图 / 视频外包单
* 给商家、电商、探店、宣传做内容

他甚至提到一些非常具体的判断：

* 接单平台可以去闲鱼、小红书、淘宝
* 如果技术和作品整理得像样，兼职一万以内不难
* 真正难的是持续产出、审美、选题和执行力

这部分本质上是在说：

* AIGC 方向已经开始从“炫技”变成“能不能形成稳定生产和变现”

### 直播里提到的一些更细的现实信息

老师其实还讲了不少很接地气的细节：

* 现在 AI 方向找工作，大体两条线最明显：
  * AIGC
  * `RAG / agent`
* AIGC 公司里会有专门的生图、生视频岗位
* 老师提到这类岗位在招聘平台上一搜很多
* 他还提到一个很现实的判断：
  * 工资不一定夸张高
  * 但对年轻人、网感好的人会更友好
* UI 背景的人也有转这条线的机会
  * 因为有些团队会混合使用 AI 生成和手绘修图

他举的变现例子也比较具体：

* 免费吸粉，后续挂车
* 先免费几集，后续付费
* 做垂类账号
* 卖自己的书、课程或周边
* 接店面图、电商图、宣传片这类外包单

这部分我第一次整理时压得比较狠，现在补回来会更完整。

## 十一、老师没有回避这条线的问题

虽然老师整体很看好 AIGC，但也讲了几个很现实的坑：

### 1. 审核

* 有些内容莫名其妙就不给过
* 平台不一定告诉你具体原因
* 你只能自己一点点试

### 2. 版权

* 经典角色和大 IP 会越来越敏感
* 迪士尼、孙悟空这类题材都可能触发限制
* 平台能否通过审核，本身就是一种风险过滤

### 3. 排队和成本

* 企业版也会排队
* API 门槛高
* 积分消耗快

### 4. 不能只会点按钮

老师不断强调：

* 不是随便点点就能做出结果
* 你得花时间研究
* 得会调提示词
* 得知道怎么拆镜头
* 得知道怎么打磨作品

换句话说，这条线不是没有门槛，而是门槛已经从“纯技术操作”转向“流程、品味、执行和打磨”。

## 十二、AWS 和今天的作业要求

这场直播虽然重点在 AIGC，但老师也把当天作业讲得很具体。

他要求大家继续补 AWS 相关链路：

* VPC
* 子网和路由表
* IGW / NAT Gateway 一类组网概念
* Lambda 部署
* 配合一个基础服务跑通流程

老师的核心口径还是老样子：

* 不要只听懂
* 要自己做
* 从春节到现在这条 AWS / 全栈线已经折腾很久了
* 再不补，后面实战就会跟不上

这一段更适合回到：

* [[06-技术就业AWS讨论｜直播笔记整理]]
* [[全栈课程作业要求]]

### 这场里关于 AWS 账号安全的提醒也不要漏

老师这里讲得很现实：

* AWS 最好按用途分开账户
* 静态 IP / 出口类用途，和纯学习用途不要混在一起
* 学习账户如果有赠金，也不要乱绑真实主卡
* 最稳妥的方式是用低余额虚拟卡
* 因为一旦误开资源，可能会直接被扣很多钱

他还特别点了：

* 弹性 IP
* ECS / EC2 一类实例
* 组网相关资源

这些东西如果没控制好，账单会很难看。

所以这场除了技术线，也是在继续强化：

> 云资源不是“先开了再说”，而是从一开始就要带着成本和风险意识去用。

## 十三、这场课最值得记住的一句话

如果把整场压缩成一句话，老师真正想推动的不是“会玩几个 AI 工具”，而是：

> 让大家从“会聊 AI”，进入“能拿 AI 跑真实项目、扛真实考核、做真实交付”的阶段。
