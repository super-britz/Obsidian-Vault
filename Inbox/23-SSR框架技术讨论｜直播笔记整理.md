# 23-SSR框架技术讨论｜直播笔记整理

## 概览

这场直播的核心，不是“教大家怎么把 `Next.js` 文档再念一遍”，而是回答四个更现实的问题：

1. [[SSR 与 CSR]] 到底解决什么问题，面试里为什么总被问
2. 为什么老师认为现在做国际化、Web3、内容站时，`SSR` 仍然是非常重要的基本盘
3. 为什么 [[Next.js 与自定义SSR]] 在真实公司里会变成“默认先用 `Next.js`，但要警惕它的边界”
4. 在什么情况下，值得自己做一套 SSR，甚至直接上 [[边缘渲染与流式渲染]]、[[Serverless架构]]

老师这场的判断非常鲜明：

* `Next.js` 仍然是默认首选
* 但只要它开始卡住你的安全、冷启动、扩展性、迁移成本，就要认真考虑别的路
* 现在做 SSR，已经不能只盯着传统 `SEO`，而要一起理解 [[AIO 与 SEO]]
* SSR 项目最终能不能扛住流量，不取决于“SSR 对不对”，而取决于部署和扩容方案

> [!note]
> 这份整理依据 [`23-SSR框架技术讨论.txt`](/Users/liangsheng/Downloads/23-SSR框架技术讨论.txt) 重构。原始转写里有不少术语误识别，我做了高概率纠正；少数不确定的框架名仍保守处理，并统一收进 [[风险与疑似错词]]。

## 高概率错词纠偏

这份稿子里有几组错词特别明显，不修正会直接影响理解：

* `textile / test text / Tesla start / text start` 基本都是 `TanStack / TanStack Start`
* `WEP js / WARP` 基本是 `Waku`
* `Jasmine / 周泰` 基本是 `Jotai / Zustand`
* `react query` 在新语境里更准确是 `TanStack Query`
* `nodejs 这是 view 的 SS 框架` 基本是在说 `Nuxt.js`
* `reserver component` 基本是 `Server Components`
* `get Server side props` 就是 `getServerSideProps`
* `cloudbot` 高概率是 `ClaudeBot`
* `setmap` 就是 `sitemap`
* `Cloudera worker` 基本是 `Cloudflare Workers`
* `Lambda edge` 更准确是 `Lambda@Edge`
* `PMTWO` 就是 `PM2`
* `Terserpack / Turbo V8` 相关口误，大概率是在说 `Turbopack` 及其配套运行时

另外有一个“`Lite plus`”类框架名，稿子里不够清楚，我没有强行下结论。

## 核心结论

* [[SSR 与 CSR]] 在面试里最重要的答案，不是“能服务端渲染”这么空，而是它主要解决两件事：[[AIO 与 SEO]] 和性能
* 老师把 SSR 框架的核心抽成两块：`data fetching` 和路由系统；其他很多东西都是附属能力
* `Next.js` 仍然是默认首选，但老师认为它近两年暴露出的冷启动、安全和平台绑定问题，已经足够让成熟团队认真评估替代方案
* SSR 带来的服务端压力不是不能解，而是部署方式不能停留在“单机 Node 服务”的思路里
* 老师的结论非常激进：对高并发 SSR 而言，[[Serverless架构]] 是最优解
* 现在国际化站点、内容站和面向 AI 检索的站点，必须把 [[AIO 与 SEO]] 一起看，而不只是盯传统搜索引擎排名
* 真正值得自研 SSR 的情况，不是“我想炫技”，而是 `Next.js` 已经成为项目的瓶颈，特别是老项目改造、冷启动、安全、扩展性这些问题

## 一、老师给出的 SSR 框架生态图

开场老师先把现在的 SSR / 全栈框架生态快速扫了一遍。

他提到的主线大概是：

* `Next.js` 仍然是 React 阵营最成熟、最完整的 SSR 框架
* `TanStack Start` 是近一年非常值得关注的新兴全栈框架
* `Waku` 这种更轻的新框架也在冒头
* `React Router` 现在也已经不只是路由库，而是有 framework mode，可以直接起 SSR 项目
* 如果是 Vue 技术栈，被迫做 SSR 时再去看 `Nuxt.js`

老师对这些框架的态度很清楚：

### 1. `Next.js` 是默认首选

原因不是因为它“最新”，而是因为它：

* 历史足够久
* 生态足够大
* 功能足够完整
* 生产实践足够多

他的原话落点大概就是：

> 你能想到的 SSR 能力，它基本都有。

### 2. 新框架进入“百花齐放”阶段

老师认为现在前端 SSR / 全栈框架已经进入一个很明显的“多方案并行”阶段。

这背后的信号是：

* 大家已经不再默认只有一个答案
* 社区开始重新审视 `Next.js` 的代价
* 平台绑定、冷启动、安全、边缘部署这些问题，正在变成一线选型问题

### 3. Vue 技术栈不是本场重点

老师基本默认讲 React 方向。

对于 Vue，他的态度非常直白：

* 如果你团队已经是 Vue 技术栈，那可以看 `Nuxt`
* 但如果不是被栈绑定，不建议专门为 SSR 选择 Vue 方案

这里可以挂到 [[技术选型方法]] 来理解：

* 框架选择不是看谁“名气大”
* 而是看你现在的团队、历史包袱、业务压力和部署形态

## 二、老师眼里 SSR 的两个核心

老师把 SSR 框架压缩成两件本质：

1. 数据预取
2. 路由与页面组织

这个抽象非常有用，因为它能把很多花里胡哨的框架能力剥掉。

### 1. 第一核心：`data fetching`

SSR 的第一件事情，就是在服务端把数据拿回来，渲染到 HTML 上，再返回给浏览器。

在 `Next.js` 语境里，老师举了两个入口：

* `App Router` 时代直接用 `fetch`
* `Page Router` 时代常见的是 `getServerSideProps`

他对 `Server Components` 的态度比较保留：

* 它和 SSR 在很多地方已经混在一起
* 对复杂项目来说，反而会增加理解和维护成本
* 不是所有项目都应该无脑上

### 2. 第二核心：路由系统

老师把 `layout / page / route` 这些都归到一个点上：路由系统。

重点有两个：

* 约定式路由，尤其是基于文件系统的路由
* 嵌套路由和布局组合

他特别强调：

* `Next.js` 这套文件系统路由很典型
* 但不是所有框架都必须这么做
* 也可以保留路由表形式

这点对后面 [[SSR 老项目迁移]] 非常关键，因为“能不能保留原路由表”，直接决定改造成本。

## 三、面试里最重要的问题：SSR 到底解决什么

老师后面把答案压得非常狠，说其实就两件事，别扯太多。

### 1. 第一目的：[[AIO 与 SEO]]

他一开始先说 `SEO`，但很快就修正为：

* 现在只谈 `SEO` 已经不够了
* 更准确地说，是 `AIO + SEO`

他的意思是：

* 以前主要面对的是 `Googlebot` 这类搜索引擎爬虫
* 现在还要面对 `ClaudeBot`、`GPTBot`、`Grok` 这一类 AI 检索体系
* 这些系统会爬你的网站、抽取内容、建立自己的检索与推荐能力

老师的判断非常激进：

* 很多团队如果今天还只看传统 SEO，说明判断已经落后了

### 2. 第二目的：性能

老师把性能放到了和搜索可见性同样重要的位置。

他认为 SSR 的性能价值，不只是“让首屏更快”，更包括：

* 让爬虫更快拿到内容
* 让 AI 爬虫在同样时间内抓到更多页面
* 提高站点在检索体系里的整体权重

这部分和 [[TTFB 与 Web Vitals]] 是直接连在一起的。

## 四、AIO 时代下，SSR 为什么更重要

这部分是这场课最有现实感的一段。

老师不是停留在“AI 会抓网页”这种抽象层，而是给了非常工程化的判断。

### 1. AI BOT 的抓取量已经不是老 SEO 那个级别

他提到：

* `ClaudeBot`、`GPTBot` 本质上也是爬虫
* 它们会抓取网页、存入向量数据库，再供模型检索与推荐
* 现在这些 AI BOT 的抓取频率和规模，已经远远不是以前的节奏

老师的项目经验口径是：

* 某些站点上 `ClaudeBot` 的抓取量可以到传统 SEO 爬虫的 `40~50` 倍
* 相比 `GPTBot` 也高很多
* 极端情况下，BOT 抓取会把服务器打到 `10000 QPS`

这就是他为什么一直把 SSR、性能、扩容绑在一起讲。

### 2. 你要维护好 `sitemap`

老师非常明确地说：

* 无论是传统搜索引擎还是 AI 爬虫
* 都要把 `sitemap` 维护好

他把 `sitemap` 解释成：

* 网站地图
* BOT 进入你网站后的入口清单
* 页面数量、层级关系、可抓取路径的结构化描述

如果没有这东西，你的网站就像一个没有地图的新国家，BOT 很难高效地走遍。

### 3. 你必须把 `Web Vitals` 做到 `good`

老师对性能指标的要求几乎是“别跟我谈折中”：

* `LCP`
* `INP`
* `CLS`
* 外加另外两个附属指标

这些指标至少都要做到 `good`，不要停留在黄区或红区。

他的判断逻辑是：

* 竞品和你内容量一样
* 但它性能比你好
* 它不光会被爬得更快，权重也更高

这就是为什么他把 SSR 的第二目的直接落到 [[TTFB 与 Web Vitals]]。

## 五、服务端压力不是反对 SSR 的理由

这一段老师情绪很强。

当有人提“SSR 服务端压力大怎么办”时，他的反应基本是：

* 这不是 SSR 的原罪
* 是你部署方案太老，或者服务端能力不够

### 1. 老师给的 Node.js 粗略口径

以一台 `4C8G` 的云主机跑普通 `Node.js / Koa / Next` 服务为例，老师给出的经验值是：

* 单机大概只能承载 `300~400 QPS`

他立刻拿这个去对比：

* `Java`
* `Go`

并强调这和真正高并发服务有很大差距。

### 2. 传统思路：K8s 自动扩容

老师承认，用传统后端视角解决也不是不能解：

* 设 CPU / 内存 / 带宽阈值
* 达阈值就横向扩容
* 把流量切到新容器

但他认为对前端 SSR 来说，这已经不是最优解。

### 3. 老师的结论：[[Serverless架构]] 是 SSR 最优解

这一点他反复说了很多次。

他的口径是：

* SSR 遇到并发压力时，最优解是 [[Serverless架构]]
* 它的横向扩容几乎是无上限的
* 扩容不需要你自己操心 Pod、阈值、冷机器怎么顶

老师还特别强调了计费视角：

* `serverless` 扩容本身不单独收费
* 核心还是按请求量计费
* 2 万个请求，无论 1 年打完还是 1 分钟打完，费用模型本质上一样

### 4. 真实口径：30 万 QPS 攻击流量

老师举了一个非常猛的实战数字：

* 他的项目曾被攻击到 `300000 QPS`
* 在他使用的方案里，这只占扩容上限的约 `20%~25%`
* 继续往上顶到 `100 万 QPS`，他认为理论上也能扛

但他也补了一句非常重要的话：

* 这不是所有云厂商的水平
* 他这里主要是基于 `AWS` 或 `Cloudflare`
* 不要把这个结果默认套到所有环境上

这块可以直接挂到：

* [[Serverless架构]]
* [[Cloudflare 与 AWS 混合云架构]]

## 六、边缘渲染与流式渲染，是老师真正想推给大家的方向

这场课里老师明显不满足于“单纯 SSR”，而是已经在往更进阶的路线推：

* `Edge Rendering`
* `Streaming Rendering`

### 1. `Edge Rendering`

老师提到的两个主方案是：

* `Cloudflare Workers`
* `Lambda@Edge`

但他的判断也非常鲜明：

* 真正做到“CDN 端执行渲染”的，他更认可 `Cloudflare Workers`
* `Lambda@Edge` 更像是分布到大洲级节点，不是他口径下的“真正 CDN 边缘执行”

这部分更适合挂到 [[边缘渲染与流式渲染]]。

### 2. `Streaming Rendering`

他后面讲自定义 SSR 时，核心就在这里：

* Node 侧用 `renderToPipeableStream`
* Edge 侧用 `renderToReadableStream`

他的目标不是“只要能渲染”，而是：

* 尽可能提早返回首字节
* 把 [[TTFB 与 Web Vitals]] 压到更好

## 七、为什么老师仍然说：能用 Next.js，就先用 Next.js

虽然这场课里老师批 `Next.js` 很狠，但他其实并没有走到“不要用 `Next.js`”。

他的标准答案是：

* 如果 `Next.js` 能满足你，优先还是 `Next.js`
* 只有当它已经成为你项目的真实瓶颈，再去考虑别的方案

这个“瓶颈”不是一句空话，而是后面这些具体问题。

## 八、老师为什么决定做自定义 SSR

这部分是整场课最重要的现实经验。

他给出的几个主要理由，基本可以直接总结成 [[Next.js 与自定义SSR]]。

### 1. 冷启动问题

老师给的口径非常直白：

* 较大的 `Next.js` 项目，`serverless` 场景下冷启动可能到 `5 秒`
* 这对首个请求、新扩容容器和流量高峰场景都非常致命

而他自定义 SSR 的结果是：

* 冷启动大约 `800ms`

他的解释也很直接：

* `Next.js` 太大、太全、执行路径太重
* 自定义 SSR 只保留自己真正需要的最小能力

### 2. 安全问题

老师把安全称为压倒骆驼的最后一根稻草。

他的课堂口径是：

* 从 `2023` 年开始，`Next.js` 安全问题明显变多
* 近两年里，他印象里有 `6` 个左右高危漏洞
* 其中有些会导致：
  * 服务器被劫持
  * 白屏
  * 授权/鉴权绕过
  * 请求污染
  * 中间件问题

他提到的最典型事件是：

* 去年 `12` 月初的高危漏洞
* 配合 `App Router` / `Server Components` 等新机制
* 可能让黑客拿到服务器执行能力
* 甚至形成 `web shell`、矿机、肉鸡

这部分他还提到一个现实问题：

* Web3、交易所、金融相关业务对这类问题几乎零容忍

### 3. 性能问题

除了冷启动，老师还补了一条“常态渲染性能”：

* 复杂页面在 `Next.js` 下可能单请求渲染就超过 `100ms`
* 自定义 SSR 目标是做到 `50~60ms`

他的逻辑是：

* `Next.js` 需要兼容太多场景
* 自研只做自己要的那条链路
* 代码量和执行量都能压得很小

### 4. 平台绑定和扩展性问题

老师对 `Vercel` 绑定的意见很大。

他认为：

* `Next.js` 有不少能力在 `Vercel` 上体验最好
* 但 `self-hosting` 下就会变得麻烦很多
* 这意味着你不是单纯在选框架，而是在部分接受平台约束

他举的例子包括：

* `Edge middleware` 或某些边缘能力，不在 `Vercel` 时支持感受差
* 某些安全特性，例如 `SRI`，版本要求和兼容性都很苛刻
* `middleware` 上下文不够全
* 更复杂的 SSR 微前端支持不好

### 5. 稳定性和服务降级问题

老师特别强调：

* 好架构不能只考虑“正常情况跑得快”
* 还要考虑出问题时怎么优雅降级

他的思路是：

* 真正 `5xx`、服务崩溃、同事骚操作把服务搞挂时
* 不能让用户只看见原始错误
* 应该在 `Nginx` 或 CDN 层直接兜底到静态 HTML

这也是他觉得自定义 SSR 价值高的原因之一：

* 更容易控制错误面
* 更容易做自己想要的兜底路径

## 九、什么时候真的值得自研 SSR

老师反复强调：不是所有公司都值得自研。

真正适合的情况包括：

* `Next.js` 已经遇到真实瓶颈
* 老项目无法承受重构成本
* 对冷启动、安全、稳定性要求特别高
* 部署环境不是 `Vercel` 优先，而是自有云、`AWS`、`Cloudflare`
* 项目长期运行，能承受后续维护成本

而不适合的情况是：

* 只是为了“看起来高级”
* 团队没有后续维护能力
* `Next.js` 其实已经够用

老师甚至直说：

* 自定义 SSR 不是写完就完
* 它会长期“拴住”你
* 后续会不断出现特性补齐、培训、排障、支持等工作

## 十、最有代表性的实战案例：老 CRA 项目改 SSR

这是全场最值钱的部分之一。

老师讲的是一个真实公司项目：

* 历史约 `8` 年
* 早年由 `create-react-app` 建出来
* 代码量大约 `30 万行`
* 是公司的核心收入来源
* 每天都在上线
* 改造时不能影响正常迭代
* 不能崩、不能停、不能出意外

他说这个问题本质上像：

> 飞机正在空中飞，你要把发动机换掉，但飞机不能降落，也不能坠毁。

### 为什么不能直接迁到 Next.js

因为一迁移就会遇到这些问题：

* 路由体系要改
* 文件目录结构要改
* 页面组织方式要改
* 数据预取方式要改
* 上线节奏要重排

对一个每天发布、30 万行代码的老系统来说，这几乎等于重构。

### 自定义 SSR 的改造思路

老师的思路非常克制：

* 不强迫整个项目改成 `Next.js` 式结构
* 保留原本的路由表
* 在现有 `Webpack` 体系上，再补一套服务端构建
* 尽量少动原业务代码

最终他给出的结果是：

* 上万个文件的项目里，真正改动的只有 `100+` 个
* 核心新增文件不到 `10` 个
* SSR 核心代码不到 `200` 行

这部分很适合挂到 [[SSR 老项目迁移]]。

## 十一、自定义 SSR 的目录结构和运行链路

老师后面开始拆他自己的项目结构。

他给出的总体划分其实很清楚：

### 1. `app`

这里放的是：

* 框架层代码
* 应用层入口
* 服务端 / 客户端组合逻辑

### 2. `src`

这里放的是：

* 正常业务代码
* 页面
* Router
* 配置
* API

老师特别强调：

* 这部分应该尽量像原来的 CSR 项目一样开发
* 不要因为改成 SSR，就强迫所有业务代码重写

### 3. `config`

这里主要是：

* `Webpack` 编译配置
* 客户端构建
* 服务端构建

### 4. 入口文件

他提到至少有两类入口：

* `server.ts`：本地开发、传统 `PM2` 部署入口
* `serverless` handler：给 `AWS Lambda` / 其他 serverless 环境使用

这部分其实已经和 [[前端架构设计指南]] 很接近了：

* 应用层
* 业务层
* 构建层
* 部署层

## 十二、如何判断一个站是不是 SSR

老师给了一个很实操的判断法：

* 打开浏览器 `Network`
* 刷新页面
* 点开 HTML 请求
* 看 `Preview` 或响应内容里，是否已经有真实渲染出来的 DOM

如果 HTML 里已经有具体内容，而不是只剩一个空壳容器，那基本就是 SSR。

他现场还演示了：

* 服务端预取的数据被灌进 HTML
* 前端再做 hydrate

## 十三、Koa、Fastify、Hono：底层服务框架怎么选

老师自定义 SSR 用的是 `Koa`。

### 1. 为什么是 Koa

他的理由是：

* 生态成熟
* 会的人多
* 足够轻
* 性能比 `Express` 好很多

### 2. `Fastify`

老师也提到：

* `Fastify` 性能更好
* 但他当年做的时候，生态和使用舒适度还没到最佳状态

### 3. `Hono`

老师后半段明显比较看好 `Hono`，因为它：

* 对 serverless 友好
* 对 `Cloudflare Workers` 友好
* 对 `Lambda` 也友好
* 很适合边缘 / worker 场景

他还给了一个很有时代感的判断：

* 以前选底层框架时，要强依赖社区经验和 Stack Overflow
* 现在有了 AI，这个权重在下降
* 你更该担心的是它在生产环境能不能扛，而不是文档少不少

## 十四、流式渲染到底在优化什么

这部分是整个自定义 SSR 讲解的技术核心。

老师用两种方式做对比：

### 1. 传统 SSR

思路是：

* 先把整页渲染成完整 HTML 字符串
* 等全部完成后，再一起返回给客户端

问题是：

* [[TTFB 与 Web Vitals]] 会比较差
* 用户和爬虫都要更晚才能看到首字节

### 2. 流式渲染

思路是：

* 渲染一点，就返回一点
* 甚至在核心数据预取没完全跑完前，先把 HTML 开头和 shell 往外吐

老师的落地方式是：

* Node 侧用 `renderToPipeableStream`
* Edge 侧用 `renderToReadableStream`

他的目标非常明确：

* 先把首字节时间压低
* 再逐步把内容流出去

这也是为什么他一直说：

* Streaming 的一个核心目的就是解决 `TTFB` 过长

## 十五、Edge + Streaming 是他想带大家走到的下一层

老师后面其实已经埋了下一节课的钩子：

* 光有 SSR 不够
* 真正更好的形态，是 `Edge Rendering + Streaming`

也就是：

* 边缘节点离用户更近
* 流式返回让首字节更早
* 两者叠加，能进一步拉性能体验和 BOT 抓取效率

这块和：

* [[边缘渲染与流式渲染]]
* [[Cloudflare 与 AWS 混合云架构]]

是天然一组。

## 十六、AI 对 SSR 工程学习方式的改变

老师最后还插了一段特别有时代特征的判断：

* 以前你问 AI “什么是 SSR”，它都可能胡说
* 现在你把整个项目丢给 AI
* 它已经能把目录结构、调用链路、数据预取、服务端入口都讲出来

这段其实特别适合挂到 [[AI开发主流程]] 去看。

对今天做工程的人来说，它的意义是：

* 学框架不再只是背文档
* 更像是“拿代码 + 问 AI + 快速定位调用链”

## 十七、这场课里最值得记住的判断

### 1. 关于选型

* 默认先用 `Next.js`
* 但不要把它当成唯一答案
* 一旦安全、冷启动、扩展性、老项目改造开始出问题，就要切换视角

### 2. 关于目标

* SSR 的核心目标不是“炫技”
* 而是为了 [[AIO 与 SEO]] 和性能

### 3. 关于部署

* 单机 Node 不是 SSR 的终局
* [[Serverless架构]] 才是老师眼里的主战场

### 4. 关于改造

* 老项目能不重构就别重构
* 最好的改造，是保留业务结构、补 SSR 能力，而不是全盘推翻

### 5. 关于架构

* 好架构不仅要快
* 还要能降级、能兜底、能预防同事失误、能抗突发流量和攻击
