# 31-前端开发相关讨论｜直播笔记整理

这场的核心不是讲某一个单点技术，而是围绕“现在前端同学怎么准备作品、怎么面试、怎么选方向”展开。老师反复强调：不要只会说概念，要有可展示、可追问、可复盘的真实作品。

相关导航：[[31-前端开发相关讨论-知识地图]]

## 一句话总结

这场课把求职准备分成三条线：

1. [[AI自动化工作流]]：用 AI 完成需求、开发、测试、上线与复盘。
2. [[Web3交易所前端]]：重点做 [[K线图]]、[[盘口]]、[[开仓和平仓]]、[[WebSocket数据流]]。
3. [[AIGC平台作品]]：围绕生成成本、模型效果、用户反馈、产出质量做真实平台感。

老师最在意的是：作品要真，细节要能答，别让面试官一问就发现只是 demo。

## 1. 工作流是安身立命之本

开场老师先回到上午布置的 [[AI自动化工作流]]。他看到群里有几位同学已经发了自己的流程，就认为这些同学已经完成了基础流程。

他强调这套流程的重要性：

- 面试时，面试官会直接问“你的工作流是不是真的用在项目里”。
- 只讲“我会自动化”不够，最好能拿出真实项目。
- 代码量、测试方式、上线流程、自动化程度，都会成为面试追问点。
- 如果只是玩具 demo，很容易被问穿。

老师举了自己的例子：他春节前后做过金融系统，代码量超过 10 万行，所以当面试官问“是不是玩具项目”时，他能比较硬气地回答。

这里的重点不是炫代码量，而是：[[作品真实性]] 比“概念正确”更重要。

## 2. 自动化测试和浏览器控制要能说细

老师提到当前自动化测试大致可以从几类能力讲：

- [[Chrome DevTools MCP]]
- [[Chrome DevTools Protocol]]
- [[OpenCLI]] / 命令行自动化链路
- [[Playwright]] 或传统无头浏览器
- [[Codex]] 这类更强的代码代理

转写里有一些词不稳定，比如 `DEVM`、`MCP CDP`、`OCI`、`raise`、`Codis`。结合上下文，大概率分别指向 DevTools MCP、CDP、OpenCLI / CLI 调用、Playwright / browserless、Codex。

面试时不要只说“我用 AI 写测试”，而要能讲清楚：

- 浏览器怎么被控制。
- 页面状态怎么被读取。
- 失败时怎么截图、定位、修复。
- 测试和开发工作流怎么串起来。
- AI 改完之后如何自测和 review。

这部分可以和已有笔记 [[浏览器控制与远程模式]]、[[VSCode自动化]]、[[AI开发主流程]] 串起来。

## 3. Codex 5.5 与“图片还原页面”

老师花了较长时间讲 [[Codex]] 新模型在前端还原上的变化。他的判断是：以前从一张图 100% 还原成页面基本不现实，现在已经接近能做出很高完成度。

他说的场景大概是：

- 给一张页面截图或设计图。
- 让模型生成页面结构、样式和图片资源。
- 使用模型自测点击和交互。
- 如果页面只做首页，大约几分钟可出效果。
- 如果要做首页、解决方案、案例、优势、关于我们等完整前端，可能一小时左右能出比较完整的版本。

老师特别强调了一点：不是所有工具都能真正“高保真”。一些海外模型会生成很强的科技感页面，但遇到中国风、小程序风、中国本土设计语言时容易跑偏。

## 4. GPT 生图、中国味与设计还原

老师对 [[GPT Image]] 的生图能力评价很高，尤其提到中文、中国城市、中国本土视觉风格的表现。他拿一张“中国元素较重”的页面举例，说海外模型容易生成淡蓝科技风，和原图差距很大。

这段里出现几个关键判断：

- 海外模型常有明显“国外味”。
- 中国小程序、政企页面、2.5D 插画、中文字体，对模型还原是难点。
- [[design.md]] 这类设计说明文件有用，但不能保证还原出复杂图像。
- 页面里的背景图、2.5D 小图、局部装饰图，往往是视觉质感的关键。

老师认为，[[2.5D插画]] 很能带来创意和视觉冲击。如果自己做设计，后续也要尽量学会做更丰富的 2.5D 视觉。

## 5. 前端设计新范式：从文档流到沉浸式交互

老师说，现在新的前端设计范式已经不只是传统 [[Flex]]、[[Grid]]、文档流布局。越来越多页面会追求沉浸式交互：

- 点击小房子后进入房子内部。
- 鼠标划过时建筑产生波浪。
- 顶部动画开始运动。
- 车充完电开走。
- 风扇转动、云朵飘动。

这类页面更像“互动场景”，不是传统信息流页面。老师的意思是：面试时可以讲，但前提是自己真的做过、调过、体验过。

## 6. 简历要按方向拆

老师上午提过，简历最好拆成三份：

- [[Web2简历]]
- [[Web3简历]]
- [[AI全栈简历]]

原因是不同公司关注点完全不一样。AI 岗会问你如何落地，Web3 岗会问钱包、合约、交易所组件，传统前端会看基础工程能力和业务经验。

老师反复提醒：不要不真实。面试官一旦感觉你东西很假，就会继续追问细节。

## 7. RAG、召回率和模型微调

老师用 [[RAG]] 举例说明“真实细节”的重要性。

如果简历写了 RAG，面试官可能会问：

- 你训练 / 接入了什么资料？
- 文档有多少？
- 召回率怎么样？
- 问法换一种还能不能召回？
- 长期记忆怎么做？
- 数据怎么入库？

老师用“平台优势”举例：如果文档里明明有平台优势，但用户换一种问法后模型回答“没有优势”，这就是召回失败。

他说后面会讲 [[模型微调]]、[[模型量化]]、32B 模型压缩、本地显卡和租卡等内容。这里的重点不是让大家马上训练大模型，而是知道这些词背后的真实工作：

- 本地模型对显卡和内存要求高。
- 不是每个人都有多张 4090 / 4080 / 50 系显卡。
- 可以租显卡，但费用可能很高。
- 也可以短时间集中使用网吧高配机器学习。
- Mac 大内存可以跑部分本地模型，但也有边界。

老师不建议现在为了学习直接花 3 万配 Windows 主机，因为硬件价格和显卡环境波动很大。他提到自己在等年底新 MacBook Pro。

## 8. 岗位判断：先 AI 全栈，再传统前端加 AI，最后看 Web3 风险

老师这场的岗位判断比较明确：

- 现在机会比 Web3 熊市尾部多。
- [[AI全栈]]、解决方案工程师、前端 + AI 的机会在变多。
- 大厂和 Web3 大厂仍有前端岗位。
- 中小创业公司纯前端岗位会越来越少，很多会被产品经理和 AI 工具吃掉。

他建议优先级大致是：

1. 先找 [[AI全栈]] / AI 落地类工作。
2. 再找传统前端 + AI 加持的工作。
3. 最后再看 [[Web3外包]] 或风险更高的方向。

但如果走 Web3，也要先过滤公司风险。尤其是排名靠后的交易所、灰色项目、团队背景不清的项目，不要只看钱。

## 9. Web3 交易所前端：先从界面和核心组件做起

老师把 Web3 交易所前端作为下午重点。他说无论 [[中心化交易所]] 还是 [[去中心化交易所]]，打开交易页面后，前端 UI 很多地方是类似的。

核心组件包括：

- [[K线图]]
- [[TradingView]]
- [[盘口]]
- [[订单簿]]
- [[开仓和平仓]]
- [[资金费率]]
- 市价单 / 限价单
- 杠杆设置
- 合约信息栏
- 钱包连接与余额展示

老师认为，前端同学最应该把 [[K线图]] 和 [[盘口]] 做扎实。尤其是盘口，面试能问出很多问题。

## 10. K线图：TradingView 是绕不开的

老师说，如果做交易所类项目，[[TradingView]] 基本躲不过。

要准备到什么程度：

- 能把 K 线画出来。
- 能找到对应数据源。
- 能按 15 分钟等时间粒度重绘。
- 熟悉 TradingView API。
- 能回答价格暴涨暴跌时图表怎么处理。

他也提到可以用其他画图方案，但面试时大概率会追问 TradingView，所以最好真的玩熟。

## 11. 盘口：面试重灾区

老师说 [[盘口]] / [[订单簿]] 是交易所前端面试的重灾区。

典型问题是：如果后端通过 [[WebSocket]] 高频推送大量买卖盘数据，前端如何处理？

可以从这些方向回答和实现：

- [[Web Worker]]：把重计算从主线程拆出去。
- [[虚拟列表]]：避免一次渲染过多 DOM。
- 节流、批处理、合并更新。
- 根据价格档位做聚合。
- 用 [[Rust]] / [[WebAssembly]] 做部分计算。
- 组件封装要有明确数据结构和更新规则。

老师提醒：这不是普通表格组件，而是金融业务里非常专业的组件。只要 K 线和盘口做得像样，面试官会认为你不是只会普通页面。

## 12. 资金费率、开仓、平仓和插针

老师解释了交易所合约里的几个关键业务概念。

[[资金费率]]：

- 用来平衡多空双方。
- 通常每 8 小时计算一次。
- 如果做多的人多于做空的人，多方可能向空方支付资金费。
- 如果做空的人多于做多的人，则反过来。
- 即便价格没动，持仓时间长也可能因为资金费率导致账户变化。

开仓方式：

- [[市价单]]：按当前价格立即成交。
- [[限价单]]：指定价格，等市场达到后成交。
- 杠杆会放大收益和亏损。

[[插针]]：

- 价格短时间快速下跌或上涨，又马上收回。
- 中心化交易所可能因为流动性、风控或作恶导致用户被强平。
- 去中心化交易所通过 [[预言机]] 和链上价格保护，某些情况下能减少插针影响。

老师强调，做交易所作品时不一定真做完整后端，但要能模拟：

- 如何开仓。
- 如何匹配订单。
- 市场暴跌时如何强平。
- 价格和仓位如何变化。

## 13. CEX 与 DEX：撮合、LP 与预言机

老师区分了 [[中心化交易所]] 和 [[去中心化交易所]] 的成交机制。

中心化交易所常见模式：

- 通过 [[订单簿]] 撮合买卖双方。
- 一个 100 元卖单可能被拆成多笔成交。
- 后台撮合系统通常不是前端同学负责，但前端要理解大概机制。

去中心化交易所还会涉及：

- 钱包连接。
- 合约调用。
- [[LP池子]]。
- [[预言机]]。
- 链上开仓和清算。

老师提到工程架构时，大意是：Node.js 可以做基础 API / SSR / BFF，但真正高频、金融性、撮合类后端常会用 Go 等语言。前端同学可以用 Node.js 模拟，但面试时要知道工业场景不是直接拿 Node.js 扛撮合核心。

## 14. Web3 手写题：钱包与合约

如果找 Web3 外包或交易所前端，老师认为手写题常见就两类：

1. 连接钱包并展示余额。
2. 给定合约地址，完成代币 approve，并查询 token 信息。

对应双链：

- [[钱包连接]]
- [[approve 风险]]
- [[ERC20]]
- [[ethers]]
- [[viem]]
- [[wagmi]]

老师的态度是：这类题并不难，但必须真做过。没做过就容易在面试现场卡住。

## 15. AIGC 平台作品：不要只说“调模型”

如果不想做 Web3，老师建议把 [[AIGC平台作品]] 做细。

面试官可能会问：

- 你调了哪些模型？
- 哪些模型效果好？
- 遇到过什么问题？
- 用户生成后不满意怎么办？
- 有没有基于用户反馈做调整？
- 成本如何统计？
- 花了多少 token / 钱，产出了多少可用视频？
- 是否有质量评估和英雄榜？

老师举了一个公司内部 AIGC 平台的例子：每天统计谁花了多少钱、生成了多少视频、最终有多少能被剪辑使用。这个例子说明，平台作品不能只停留在“我会调用模型”，还要有运营指标和成本意识。

可关联已有笔记：[[AIGC 与 AI视频生成]]、[[AI开发主流程]]、[[模型授权与成本]]。

## 16. 课堂作业与行动项

老师最后给的行动项很明确：

- 没有完成 [[AI自动化工作流]] 的同学，先什么都别碰，把工作流搞好。
- 想走 Web3 / 交易所方向的同学，做 K 线、盘口、socket、开仓、平仓、强平模拟。
- 想走 AIGC 方向的同学，把 AIGC 平台做细，尤其是模型选择、成本统计、反馈调整、可交付结果。
- 每个人最终都要给出自己的侧重点和作品。

老师的底层判断是：他以后帮同学推荐机会时，需要知道每个人到底适合什么方向，以及有没有能拿得出手的作品。
