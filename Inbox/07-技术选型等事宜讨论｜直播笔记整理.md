# 07-技术选型等事宜讨论｜直播笔记整理

## 概览

这场直播不是单纯“盘点框架”的课，而是在把课程从：

* [[06-技术就业AWS讨论｜直播笔记整理]]
* [[前端架构设计指南]]
* [[技术选型方法]]
* [[Agent 与 Skill]]
* [[前端技术雷达]]

这几条线正式串起来。

老师最想讲清楚的底层逻辑是：

1. 技术选型不是“看谁火就用谁”
2. 前端架构第一步不是框架，而是缓存、离线、路由和部署形态
3. AI 可以写实现，但做不了项目上下文里的最终决策
4. 所以后面真正值钱的是：**会判断、会取舍、会解释为什么这样选**

> [!note]
> 这份整理按主线重构。文字稿里有大量口音词和转写错误，凡是我不够确定的地方，都放进了 [[风险与疑似错词]]，没有硬猜成结论。

## 一、开场先续上了前一轮作业

老师一开始没有直接讲框架，而是继续接着上一轮 [[Cloudflare Tunnel]] 和 `EC2 + 域名` 作业说。

他主要强调：

* 前面已经搭过 `EC2`
* 现在域名也买了
* 接下来要把它通过 `Cloudflare Tunnel` 之类的方式真正带出来
* 目标不是只有一个 `IP`
* 而是把：
  * 入口
  * HTTPS
  * 权限控制
  * 访问体验

接在一起

老师还顺带讲了很多关于代理 / VPS / 家庭住宅 IP 的经验判断：

* 太便宜的 VPS 通常速度一般
* 不要既想 `6~7 刀/月`，又想又快又纯净又不封
* 如果普通代理不稳，才会进一步考虑更贵的路线

这一段真正想表达的是：

> 你前面做的 AWS / 域名作业，不是“有一台机器”就结束，而是要形成一个 **真实能访问、能交作业、能继续开发** 的入口。

## 二、老师对作业执行力还是非常不满意

老师开场情绪很强，重点其实还是：

* 作业交得太费劲
* 很多人执行得不够
* 听懂和做出来完全不是一回事

他反复强调：

* 你不折腾作业，后面很难真正上手
* 该晚睡就晚睡，先把事做了
* 工作以后很多事情也不是等你“完全想明白了”才开始

这和整场课后半段的技术选型主题是连着的：

> 不是知道名字就行，而是要真的把这些东西跑进项目。

## 三、前端架构第一步不是框架，而是先回答几个问题

老师讲“前端架构设计指南”的时候，第一步不是 `React` 还是 `Vue`，而是先问：

* 要不要缓存？
* 要不要离线？
* 要不要 `Service Worker`？
* 是 `SPA`、`MPA` 还是混合形态？
* 页面白屏时间怎么处理？
* 路由到底是真是假路由？

这也是他为什么一直说：

> 前端架构不是“布几个文件夹”，而是一开始就要做系统级取舍。

## 四、缓存和离线能力：这场里反复强调的基础题

老师在这场对缓存题非常上火，因为他认为这是面试里最基础的一类问题。

他点到的关键词包括：

* 强缓存
* 协商缓存
* 本地缓存
* `IndexedDB`
* `Cookies`
* `LocalStorage`
* `Service Worker`

他的核心意思是：

* 强缓存 / 协商缓存是高频面试题
* 本地缓存不是只有 `LocalStorage`
* 真要做复杂缓存和离线能力，`IndexedDB` 和 `Service Worker` 就会进入主线

### `PWA` 和 `Service Worker`

老师的讲法很典型：

* `PWA` 当年想把 `H5` 应用做得更像原生 App
* 虽然那条大路线没完全走成全民主线
* 但 `Service Worker` 作为能力被留下来了

它带来的价值包括：

* 离线访问
* 资源预缓存
* 断网下继续打开
* 秒开体验

但老师也强调代价：

* 维护成本高
* 版本更新难
* 如果策略搞不好，用户可能一直看不到最新版本

### `Workbox`

老师明确点了：

* 真要做 `Service Worker`
* 最值得直接记住的工具就是 `Workbox`

> [!important]
> 这场课里，`Workbox` 是老师明确点出来的简历 / 面试关键词。

## 五、老师怎么划分前端架构形态

老师这场给的结构很经典：

1. `SPA`
2. `MPA`
3. `MPA + SPA`
4. 同构 / SSR 一类形态

### `SPA`

老师的判断是：

* 最普遍
* 最好招人
* 开发相对简单
* 但白屏时间长是典型问题

而且他反复说：

* 你不能只会用 CLI
* 至少要理解构建链
* 要知道白屏时间怎么处理

### `MPA`

他把它看成更传统的多页方式：

* 每个路由是真路由
* 页和页切换时更多内容会重新渲染
* 更像早期 Java / PHP 的 Web 系统

### `MPA + SPA`

老师很重视这种混合形态，因为它会把真实服务端路由和前端假路由揉在一起。

### 同构 / SSR

老师在这段其实已经开始把前端往 `Node.js` 服务渲染那边推了：

* 前端不只是“浏览器里的页面”
* 有时要自己在 `Node` 层处理渲染
* 要同时考虑模板、路由、服务端接管方式

## 六、真假路由：老师认为很多人平时根本没想明白

这场课里，老师专门点出来一个非常经典的问题：

* `SPA` 常常是“假路由”
* `MPA` 是“真路由”

他的意思是：

* 前端改成 history 路由后，URL 看起来很真
* 但服务端未必真有这个路由
* 一刷新就可能 `404`
* 所以部署时常常要给 fallback
* 再让前端重新接管路由

老师的抱怨很明确：

* 很多前端写页面写了几年
* 但根本不知道自己代码最后部署到哪里
* 也不知道刷新为什么会 `404`

> [!summary]
> 这段最值得记的不是“假路由”这个词，而是：
>
> **前端路由一定要和真实部署入口一起理解。**

## 七、老师为什么还在反复强调“手配 Webpack”

老师这场对“只会脚手架”的态度非常强硬。

他的意思是：

* 你如果只会 `create xxx app`
* 对构建链一无所知
* 本质上就只是个工具使用者

他点的基础能力包括：

* `Vue` 原理
* `React` 原理
* `Webpack` 原理
* 手配 `Webpack`
* HTTP 原理

这场课里老师给的口径是：

> 手配 Webpack 在老前端语境下，已经是非常基础的工程能力。

更客观一点的说法应该是：

* 现在未必每个项目都手写 `Webpack`
* 但 **理解构建链** 仍然是基础，不然你很难理解旧项目和很多构建问题

## 八、包管理工具：老师这场非常明确地站队 `PNPM`

老师这里其实给了一条很完整的包管理演进线。

### 1. 旧时代依赖问题

老师在讲早期依赖体系时，想表达的是：

* 旧的依赖树容易出现层层嵌套
* 会有依赖重复安装
* 容易产生“幽灵依赖”式的问题

### 2. `Yarn`

老师的理解是：

* `Yarn` 曾经很好地解决了部分旧依赖问题
* 更偏软链接和扁平化思路

### 3. `PNPM`

老师明确推它的原因包括：

* 省空间
* 有 `pnpm store`
* 用硬链接组织依赖
* 对 `monorepo` 特别友好

### 4. `Yarn 2+`

老师的态度很清楚：

* 新版 `Yarn` 把很多 `PNPM` 的思路学过去了
* 但配置复杂
* 既然这样，不如直接上 `PNPM`

### 5. 在 serverless 里的坑

老师还补了一个很有实战味道的点：

* `PNPM` 早期在某些 serverless 场景下不一定很顺
* 因为硬链接和受限文件权限环境之间可能有兼容问题
* 现在已经比以前好很多，但原理要知道

> [!important]
> 这场课里，包管理的现实结论基本就是：
>
> **团队默认优先 `PNPM`，尤其是你要做 `monorepo` 时。**

## 九、构建工具：`Webpack` 没死，但 `Vite` 更像新默认

老师对这一块的判断很现实。

### `Webpack`

* 依然是老牌王者
* 很多公司旧工程还在用
* 不是说它已经没人用了
* 但这几年明显疲软

### `Vite`

* 最早和 `Vue` 生态关系很深
* 新项目越来越常见
* 黑盒程度更高
* 上手更快

老师真正想表达的是：

* 旧项目里看见 `Webpack` 很正常
* 新项目里 `Vite` 越来越像默认项
* 但你不能因为新项目都用 `Vite`，就完全不理解 `Webpack`

## 十、老师把浏览器自动化和测试也放进了选型视野

这场课里，老师提到了一类越来越重要的工具方向：

* `Playwright`
* `Puppeteer`
* 以及“自然语言 + 代码控制浏览器”的新工具

他的主线是：

* 以前主要靠 `Puppeteer` 这类传统自动化
* 现在开始出现更偏 agent 化、更适合陌生页面的浏览器自动化方案

老师想说的不是“测试工具换代了”，而是：

> 浏览器已经不只是调试工具，而是越来越像执行入口。

## 十一、老师怎么看前两年很火的拖拽式工作流

老师中段花了一段时间专门吐槽拖拽式 workflow。

他给出的判断很强：

* 愿景很好
* 是想让不会编程的人也能做自动化和 agent
* 但对程序员来说，很多事直接写代码反而更快
* AI 能直接生成和拼接代码后，这类工具的吸引力下降很快

他甚至用了非常直接的比喻：

* 这类东西像“时代的电话亭”

> [!warning]
> 这是老师很强的阶段性判断。  
> 更客观的说法是：
>
> * 低代码 workflow 没有完全消失
> * 但对会编程的人，价值确实被代码型 agent 框架和大模型能力压缩了很多

## 十二、Apple Intelligence、Claude Max、电脑控制

老师这场还补了一条 AI 观点线：

* Apple Intelligence 后面会越来越强
* Claude Max 已经在往“控制电脑”方向走
* 曾经很火的“电脑控制工具”很多是过渡产品

他的真实意思是：

* 官方系统级能力一旦成熟
* 很多第三方电脑控制 / workflow 工具的独特性会变弱

这和前面讲的低代码工作流、浏览器控制、agent 框架，其实是同一条判断：

> “让 AI 去动工具” 这件事，会越来越被官方能力接管。

## 十三、`Agent` 和 `Skill`：这场讲得特别清楚

这场课一个最值得单独保留的部分，就是老师把 `agent` 和 `skill` 讲开了。

### `Agent`

老师的定义是：

* 能理解目标
* 能做决策
* 能调用工具
* 能持续朝目标前进

### `Skill`

老师的定义更像：

* 被调用的具体能力
* 比如：
  * 浏览器 skill
  * 写代码 skill
  * 部署 skill

### 老师给的类比

* `agent` 像“人”
* `skill` 像“人会的技能”

### 这场给的最小 agent 结构

老师大致把它拆成：

* 模型
* 提示词
* 记忆系统

再往后扩：

* `RAG`
* loaders
* 向量检索
* tools / skills
* workflow

> [!important]
> 这场课里，`agent` 和 `skill` 的分层，是最值得单独留卡片的一条。

## 十四、老师这场实际推荐的 agent 开发方向

老师这场不是让大家继续拖拽面板，而是明确要大家：

* **用程序方式开发自己的 agent**
* 不只会用工作流面板

他明显带大家接触的第一个 AI 框架，高概率是在说：

* `Mastra`

原因和它的特征很吻合：

* TypeScript 友好
* 有 agent / workflow / cloud 这类模块
* 支持模型接入
* 支持提示词
* 支持记忆
* 能继续扩成 RAG 和更复杂能力

### 这场作业的新增主线

老师明确想让大家做的是：

* 用程序方式开发一个最小 agent
* 有余力再继续往深处扩

> [!note]
> 文字稿里“部署到 corner / corner 上”的词不稳定。  
> 更稳妥的理解是：**把这个 agent 真正跑到一个可访问环境里**，而不是只在本地演示。

## 十五、老师为什么带大家刷“前端技术雷达”

后半场老师明显是在看一类技术榜单 / 趋势页，然后逐个点评。

他的目的不是“看热闹”，而是让大家训练：

* 看趋势
* 看成熟度
* 看生态
* 看社区
* 看适不适合当前项目和团队

这也是这场课题目里“技术选型”的核心。

老师想让大家形成的习惯是：

> 不是别人说火就跟着用，而是自己要能说清楚：
>
> * 为什么选它
> * 为什么不选另一个
> * 它会给团队带来什么收益
> * 它可能踩什么坑

## 十六、这场里老师明显偏好的几类工具

下面这些是我根据上下文整理出的高确定性偏好。

### 1. UI 组件库

老师口中的：

* `top one`
* “爹一样的 UI 组件库”
* 比 `MUI` 更简单
* 和 `Tailwind` 集成紧
* 主题特别好改

高概率是在说：

* `shadcn/ui`

### 2. 数据库 / BaaS 平台

老师提到那个平台的特征是：

* `Postgres`
* 验证
* API
* 边缘函数
* 实时订阅
* 存储
* 每个项目一个数据库

高概率是在说：

* `Supabase`

而且老师明显认为：

* 后面课程和作业会真的用到它

### 3. 授权 / 认证框架

老师提到一个他明显在认真用的认证框架：

* 做授权验证
* `Node.js` 生态
* 全开源
* 很完善

高概率是在说：

* `Better Auth`

### 4. 设计 / 原型工具

老师提到一个：

* “cursor for designers”
* 开源
* AI-first
* 可视化编辑

高概率是在说：

* `Onlook`

但他的判断是：

* 方向有意思
* 课程当前主线暂时不需要重点投入

### 5. serverless / 后端框架

老师明显反复提到一个轻量方向，高概率是在说：

* `Hono`

他的口径是：

* 轻
* serverless 友好
* 在 Cloudflare / AWS 这条线上很合适

## 十七、前端框架趋势：老师真正想让大家抓的是“路线变化”

老师不是单纯在说某一个框架赢了，而是在说前端框架这几年在追求什么。

### 他反复提到的方向

* 去 runtime
* 去沉重虚拟 DOM
* 更细粒度响应式
* 框架更轻
* 保留熟悉的开发体验

### 这段里高频出现的名字

转写混得比较乱，但明显涉及：

* `React`
* `Svelte`
* `Solid`
* `SvelteKit`
* `SolidStart`
* `HTMX`

老师真正的判断大概是：

* `React` 依然是主轴
* 新框架在不断挑战“体积、运行时、diff、响应式”这些老问题
* 这些新路线你不一定马上生产上用
* 但你做前端，不能完全不懂

他还特别提到：

* 海外团队更愿意试新路线
* 国内团队更保守一些

> [!warning]
> 这部分更适合记成“趋势判断”，不要记成：
>
> * 某个新框架已经全面赢了
> * 或者 `React` 快不行了

## 十八、移动端：老师并没有轻易抛弃 RN

老师在移动端部分讲得很现实。

### 1. `React Native`

他的态度是：

* 热度在变
* 排名没有以前稳
* 但它还没有彻底死

### 2. 老师自己的做法

他明确说：

* 自己这些年一直还在用 `RN`
* 只中间稍微切过一点 `Flutter`

### 3. 对新框架的判断

老师承认一些新框架：

* 理念很强
* 双线程
* 懒加载
* 复杂交互更有吸引力

但他更担心：

* 生态不成熟
* 组件不够
* 关键时刻踩坑

所以他的核心标准还是：

> 选移动端框架时，不只看理念，还要看生态和长期维护性。

## 十九、老师最后真正想训练的，是“为什么选它”

老师后面其实把一个很典型的面试题讲出来了：

* 你为什么选这个框架？
* 你选择它之后遇到过什么坑？
* 团队怎么接受它？
* 发现不适合时怎么调整？

他很明确地说：

* “别人用我也用” 不是答案
* “我看它挺火” 也不是答案

这说明这场课的真正落点不是工具，而是：

> **你要能为自己的技术选择负责。**

## 二十、这场课最不适合直接照抄照做的地方

### 1. 代理 / VPS / 封号经验

老师很多说法来自他的长期经验，但更稳妥的执行原则还是：

* 正规
* 可控
* 成本和预期匹配

### 2. 对低代码 / workflow 工具“已经死了”的判断

这是他很强的阶段性判断。  
更客观地说：

* 它们对程序员的吸引力下降了
* 但并不等于彻底没有价值

### 3. 对国内 / 海外框架使用习惯的绝对化表述

比如：

* 中国全员 `Vue`
* 海外都不用框架

这些更适合记成趋势观察，不适合当绝对事实。

### 4. 国行设备开启某些 AI 能力的技巧

这里明显带有强环境依赖和阶段性技巧，不适合直接照做。

## 二十一、适合放在顶部的一段摘要

> 这场直播真正讲的不是“今天装哪个框架”，而是：
>
> * 前端架构从缓存、离线、路由、构建、同构开始就要做选择
> * 包管理、构建、测试、组件库、BaaS、移动端和 agent 框架都要进入统一的选型视野
> * AI 会帮你写，但不会替你承担技术选型的后果
> * 所以后面真正值钱的，是“会做判断的人”，而不是“只会照着教程搭的人”
