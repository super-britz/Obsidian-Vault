# 24-多领域议题研讨会｜直播笔记整理

## 概览

这场直播不是单线课程，而是一场把几条主线硬串在一起的高密度研讨，主要分成四段：

1. 企业环境里的 [[企业落地判断]]、AI 工具落地阻力和账号成本经验
2. [[Solidity]] 延伸：用 `Remix` 写最小红包合约

4. 以太坊标准、[[ERC20]]、[[EVM]]、`gas`、账户模型和面试常问点

这场最有价值的地方，不在于某个单点知识，而在于老师把这些东西放在了一条很现实的职业链路里：

* 企业 AI 落地不只看工具强弱，还看组织、合规和登录体系
* Web3 前端不只要会接页面，还要能看懂合约、钱包和标准
* AIGC 不只是“会生成”，而是要能落到真实项目、真实变现和真实交付

## 这场里需要先纠正的错词和概念

### 高概率错词

* `cloud` 基本都是 `Claude / Claude Code`
* `Slardar / solid / solid课` 基本都是 `Solidity`
* `remit` 基本是 `Remix`
* `message center` 是 `msg.sender`
* `message value` 是 `msg.value`
* `pay able` 是 `payable`
* `UIINT256 / UIN7` 是 `uint256`
* `UIDT` 是 `USDT`
* `1320 / EIP120` 这一段其实都在说 `ERC20`
* `DF` 这一段其实是在说 `DApp`
* `easers / easy` 基本是 `ethers / ethers.js`
* `Vargan / Wagon` 是 `wagmi`
* `Vmware / Vimo` 是 `viem`
* `net js / nicejs` 结合上下文更像 `Next.js`
* `AIDC / AIDP` 基本都是 `AIGC`

### 需要纠正的几个概念

* `EIP` 才是以太坊改进提案的大集合，`ERC` 是其中偏应用接口和标准的一类，不是反过来
* `ERC20` 不是“不能找零”，而是链上按最小单位整数记账，再通过 `decimals` 展示成用户看到的小数
* “有白皮书就不是骗子”这个说法不成立，白皮书只是最低限度的项目说明，不是信用担保
* 红包示例里的随机数只是教学用伪随机，不能直接拿去做真实抽奖或资金分配
* 跨区买账号、二手家庭号、中转站号这些只能算老师的省钱经验，不能当标准操作手册

## 核心结论

* 大型国企、大厂和“带中国/国家字样”的组织，AI 落地最大的问题不是模型本身，而是合规、登录、数据和组织链路
* 只会“注册账号、会用 Claude”已经不够了，市场更看重你能不能把 AI 放进真实工作流
* `Solidity` 语法本身不难，真正难的是业务规则、资金流向、安全和 `gas`
* DApp 前端不能只会页面，至少要理解钱包、标准、合约地址、ABI、确认块和交易状态
* 现在 React DApp 生态里，`wagmi + viem` 的存在感比单独讲 `ethers.js` 更强
* AIGC 视频方向已经不是概念题，老师把它直接放进景区、寺院、短剧和企业项目的变现场景里看
* Web3 面试喜欢问标准编号和行业黑话，但很多题本质上是在确认你有没有做过真实钱包和合约交互

## 一、00:08-05:09 企业 AI 落地与账号成本

### 1. 国企和大厂不是不能用 AI，而是很难直接上云端工具

老师开场举了中国电建、设计院这类案例，结论很直接：

* 这些组织不是完全不用 AI
* 但直接把现成的云端 AI 工作流塞进去，成功率很低
* 真正卡住它们的往往是：
  * 登录体系
  * 本地部署要求
  * 数据和合规约束
  * 内部流程过长

他的实际判断是：

* 活往往要从甲方主体里拆出来
* 放到外围公司、承接公司、外包团队里先跑
* 这样才更现实

这和 [[企业落地判断]] 是一致的：

* 不是“会不会”
* 而是“能不能在组织里被允许”

### 2. 如果必须本地化，老师给的是“本地模型兜底”思路

他现场提到的思路大致是：

* 代码模型可以考虑千问系代码模型
* 再配一个 `Gemma` 一类模型做补充
* `Claude Code` 这种工作流未必完全不能装，但登录和环境会更麻烦

这里更适合理解成：

* 企业受限环境下的替代方案
* 不是老师认为“本地模型已经全面追平”

这点和 [[模型授权与成本]]、[[开发环境要求]] 可以对着看：

* 本地模型能跑，不等于主力体验最好
* 真要稳定交付，还是要看整体环境

### 3. 大厂 AI 推进的真实问题是“上面要 KPI，下面不买单”

老师对大厂的描述很尖锐，但主线很清楚：

* 大厂很多已经自研 CLI、自研模型、自研内部工具
* 但内部一线团队未必真心愿意用
* 于是就会出现：
  * 为了完成 KPI 推 AI
  * 结果做出一堆不好用的内部流程
  * 甚至出现“拥抱 AI 但体验很差”的尴尬状态

这一段不要记成“所有大厂都不行”，更准确的理解是：

* 组织越大，推进成本越高
* 工具强不强只是很小的一部分

### 4. 求职市场已经进入“只会基础 AI 工具不算优势”的阶段

老师反复在强调：

* 以前会注册、会开账号、会用基础 AI 工具，已经算加分
* 现在这已经不够了
* 企业更看重的是：
  * 你能不能把 AI 接进业务
  * 你能不能解决企业里的实际问题
  * 你能不能适配对方的环境限制

这和 [[AI开发主流程]] 的逻辑也是一致的：

* 会聊，不等于会落地
* 会调用，不等于能交付

### 5. 账号全球比价是经验，不是标准答案

老师花了一段时间讲：

* 订阅价格存在地区差异
* 可以通过本地 Apple ID 或地区切换比价
* 不建议买家庭号、中转号、来源不明的二手号

但这里一定要加一层修正：

* 这只能记成他的省钱经验
* 不能记成官方认可方案
* 也不能忽略 [[设备与账号安全]]、封号、扣款失败和条款风险

## 二、05:09-23:17 红包合约实战、求职穿插与作业

### 1. 这段课的目标不是“让 AI 帮你一把梭”，而是让你看懂最小合约

老师在写红包合约前先强调了一件事：

* AI 现在很强
* 但如果你完全不懂代码，AI 出错时你会被直接埋掉

这段课真正的目标不是造一个多复杂的产品，而是让大家明白：

* 合约最小结构怎么搭
* `payable`、`msg.sender`、`msg.value` 在哪用
* 状态变量、映射、构造函数和转账逻辑怎么串起来

### 2. 红包合约的最小设计

老师现场写的红包合约，结构很简单，核心变量大致包括：

* 发红包的人地址
* 红包总金额
* 红包数量 `count`
* 是否均等红包 `isEqual`
* 是否已领取的记录 `mapping(address => bool)`

换句话说，这个示例其实就是一个最小版的资金分发合约。

### 3. 构造函数里的关键点

老师的构造函数逻辑大致是：

* 传入红包个数
* 传入是否均等
* 用 `require(msg.value > 0)` 保证部署时真的打钱进来
* 用 `msg.sender` 记录发红包主体
* 用 `msg.value` 初始化总金额

这里正文里最值得记住的是这些原生概念：

* `uint256`
* `payable`
* `msg.sender`
* `msg.value`
* `require`

### 4. `getBalance` 和 `claim` 只是练语法，但很适合入门

老师自己也说了：

* 获取余额其实可以直接看合约余额
* 这里单独写一个方法，更多是练语法

`claim` 的主逻辑大致是：

* 先判断红包数量还有没有
* 再判断当前地址有没有领过
* 再判断还有没有余额
* 如果只剩最后一个红包，就把剩余金额全部转走
* 如果是均等红包，就按剩余人数均分
* 如果不是均等红包，就按一个伪随机逻辑算金额
* 最后标记已领取，并把 `count` 扣掉

### 5. 随机红包这一段要特别纠偏

老师这里用了时间戳一类方式做“随机数”示例。

这只能理解成：

* 为了课堂演示“随机红包怎么写”

不能理解成：

* 这就是安全的链上随机

更准确地说：

* 这种写法只是伪随机
* 真做抽奖、资金分配、博弈类产品，不能直接照搬

### 6. 老师对合约难度的判断很现实

他这场的核心判断其实很好记：

* 简单合约非常像“带钱的后端逻辑”
* 真正复杂的不是语法
* 而是金融规则、杠杆规则、结算规则和业务背景

这和 [[21-区块链课程等讨论｜直播笔记整理]] 是一致的：

* 合约代码量不一定大
* 但规则价值很高

### 7. 中途穿插的求职信息，也值得单独记一下

老师中间插入了两个很现实的案例：

* 有同学拿到火币 `HTX` 方向的 offer，案例里展示的是北京线下工作、税后 `32000`、薪资以 `USDT-TRC20` 发放
* 还有一个快 `40` 岁的学员，在非常传统的出口家电公司，靠 AI 相关理解拿到了不错的岗位和薪资

这两段真正想说明的不是“所有公司都这样”，而是：

* [[交易所求职]] 仍然有机会
* 传统行业也开始需要“懂 AI 又懂业务”的人

同时也要保留一个修正：

* 老师对“只要按时发工资就行”的价值判断非常个人化
* 这部分更适合记成他的职业立场，不适合直接当统一择业标准

### 8. 这节课布置的作业很明确

这场和合约实战直接对应的作业是：

* 做一个红包功能
* 把它部署到本地链
* 再和上次的商店类作业一起继续推进

老师对目标的要求也很清楚：

* 不是让大家把合约写得花哨
* 而是先把最小链路跑通

## 三、23:17-38:05 前端工程、AIGC 项目与线下安排

### 1. 课堂从“会调 ABI”推进到“别让调用全是 any”

老师后面切到了前端调用合约的工程问题。

他想解决的核心痛点是：

* 直接拿 ABI JSON 然后 `new Contract`
* 很多调用全是 `any`
* 参数和返回值没有类型提示

所以他演示的是：

* 用 ABI 类型生成工具产出带 `factory` 的调用封装
* 这样在前端里调合约时能拿到更明确的类型签名

结合他提到的 `factory`、`ethers v6`、自动生成签名这些上下文，这一段高概率是在讲 `TypeChain` 这一类工具。

### 2. 现在 React DApp 前端更常见的是 `wagmi + viem`

老师这段虽然口误很多，但主线很好抓：

* 以前很多人直接讲 `ethers.js`
* 现在 React 生态里更常见的是 `wagmi + viem`
* 原因不是 `ethers.js` 不能用，而是：
  * React hook 生态更完整
  * 连接钱包和状态管理更顺
  * 业务集成层体验更现代

这部分可以和 [[钱包封装库]]、[[20-Web3知识及作业安排｜直播笔记整理]] 对着看。

### 3. `SPA` 还是前端底座，`Next.js` 这类框架是下一层能力

老师后半段把工程能力分成了几层：

* `SPA`
* `MPA + SPA`
* 同构 / `SSR`
* `Next.js` 一类更完整的工程框架

他真正想强调的是：

* 现在找工作不能只会做静态页面
* 你至少要理解 [[SPA 与 MPA]]、[[SSR 与 CSR]]、[[Next.js 与自定义SSR]] 这些基底
* 真到企业项目里，很多 AI、Web3、管理后台、内容站都会连到这层

### 4. 线下项目的安排已经很像“多项目并行”

老师给出的线下规划大致是：

* 用一套前端底座做一个 `AIGC` 项目
* 再做一个 `DEX` 最小项目
* 可能再补一个管理后台或部分业务逻辑

他想用这些项目把大家真正拉到：

* 合约
* 前端
* 模型调用
* 业务逻辑
* 工程集成

这条完整链路上。

### 5. AIGC 这段最有价值的是“寺院 / 景区祈福视频”变现场景

老师提到他最近在推进的日本方向项目，其中一个很典型的想法是：

* 用户在线下景区或寺院扫码
* 上传照片或提供个体信息
* 系统自动生成一条带个人元素的祈福视频
* 再按“香火钱”或场景化收费模式变现

这段真正值得记的是：

* AIGC 的价值不只是“生成炫技视频”
* 而是能不能嵌进真实线下场景和付费场景

这和 [[AIGC 与 AI视频生成]] 可以直接连起来。

### 6. `Happy House` 这一段更适合记成“待观察工具”

老师现场提了一个国内的视频生成产品 `Happy House`，大意是：

* 号称免费部署
* 走文生视频 / 图生视频方向
* 目标上想对标 `Seedance`

但他自己当场也没有完全研究明白，所以更准确的记录方式应该是：

* 这是一个待观察的国内视频生成工具
* 不要现在就把它记成成熟结论

### 7. 短剧和传统企业两个方向，都被老师看成新岗位来源

老师在这场里给出的判断是：

* AI 短剧平台和内容平台已经需要程序员团队做生成平台、稳定性和工程化
* 传统出口、家电、外贸公司也在问 AI 客服、机器人、海外自动化运营怎么做

这意味着他看的是同一件事：

* AI 没有把岗位简单抹平
* 而是在把岗位重组成“更懂工程 + 更懂业务”的形态

## 四、39:23-59:58 以太坊标准、基础概念与面试重点

### 1. 面试喜欢问标准编号，但本质是在试探你有没有真实交互经验

老师先从 `ERC20` 讲起，然后立刻跳到几个面试常见标准：

* `EIP-747`
* `EIP-1193`

他想表达的是：

* 面试官未必真要你背诵全部编号
* 但如果这些最常见的交互标准你完全没概念，就很容易暴露出没做过钱包和合约接入

### 2. `EIP-747` 这段的正确理解

老师举的例子是对的，术语要修正得更准确：

* `EIP-747` 对应的是让 DApp 请求钱包“关注 / 添加”某个 token 资产
* 用户不用自己手抄合约地址再手动添加
* 这样钱包交互体验会更顺

换句话说，这一题本质问的是：

* 你知不知道 DApp 如何改善“钱包里没有这个币”的用户体验

### 3. `EIP-1193` 这段的正确理解

老师这里想讲的是钱包 Provider 标准。

更准确地说：

* `EIP-1193` 是 JavaScript 侧以太坊 Provider 的标准接口
* 它约束钱包扩展怎样把 Provider 暴露给网页
* 也约束 DApp 怎样与钱包通信

所以这一题背后在问的是：

* 你是否知道前端和钱包到底怎么连起来

### 4. `EIP` 和 `ERC` 的关系，这场里必须纠偏

老师中间自己也意识到前面说反了。

更稳妥的记法是：

* `EIP` 是总的改进提案体系
* `ERC` 是其中偏接口和应用标准的一类
* `ERC-20`、`ERC-721` 这些，本质上也是在 EIP 流程里被讨论和沉淀出来的标准

所以如果面试里有人把 `ERC-1193` 这种说法直接讲出来，你至少要知道：

* 他把概念混了

### 5. `ERC20` 的真正重点

老师这段课真正想讲的是：

* `ERC20` 是最常见的同质化代币标准
* 它定义的是一组接口，不是唯一实现
* 真做项目时，应该优先用成熟、被审计过的实现，而不是自己野生手撸

更准确地说：

* 原生 `ETH` 不是 `ERC20`
* `USDT`、`USDC` 这类部署在以太坊上的 token 才是典型 `ERC20` 语境
* 钱包里添加 token 时，本质上是根据合约地址识别资产

这一段和 [[ERC20]]、[[ERC 标准]]、[[MetaMask]] 可以一起看。

### 6. “世界计算机”这一段是 Web3 基础课的底座

老师后半段把以太坊的底层抽成了几层：

* 区块链是账本和存储层
* 矿工 / 验证者是执行和打包的一部分
* `EVM` 是智能合约执行环境
* 智能合约是跑在链上的程序
* 全网共享的变量状态就是“世界状态”

它更适合和这些笔记一起看：

* [[Web3基础概念]]
* [[EVM]]
* [[智能合约]]
* [[区块]]

### 7. `DApp`、智能合约和图灵完备

这一段有几个地方转写会直接影响理解：

* 老师说的 `DF` 实际上是 `DApp`
* `solid` 实际上是 `Solidity`

这里的主线是：

* `DApp = decentralized application`
* 智能合约不是以太坊发明的词，但以太坊把它工程化、规模化了
* `Solidity` 是图灵完备语言，所以能写条件、循环、变量和复杂逻辑

老师举的“抵押资产、到期自动结算”这个例子，其实就是在解释：

* 为什么智能合约会比口头协议更适合自动执行

### 8. 前端看交易成功，不能只看“发出去了”

老师讲区块和确认块这一段时，有个细节非常重要：

* 交易发出去，不等于业务已经最终稳定成功
* 真正稳妥的前端交互，应该等待确认
* 有时甚至要等多个块确认

所以在前端里更合理的状态机应该是：

* 用户签名
* 交易广播
* 交易上链
* 等待确认
* 最终再提示完成

这点和 [[数据上链]] 直接连着。

### 9. 账户模型、`nonce` 和 `gas` 都是面试高频点

老师这段把几个面试常问点放到一起讲了：

* 外部账户 `EOA` 有私钥
* 合约账户没有私钥，靠代码驱动
* 两类账户地址外观可能一样，但控制方式完全不同
* `nonce` 在链上是很重要的交易顺序和状态概念
* `gas = gas limit * gas price`

他说得比较口语化，但你真正该记住的是：

* 不要在面试里把 `gas` 只说成“手续费”
* 更准确地说，它是执行交易或合约调用时消耗资源的计价机制

### 10. `gas` 这一段最值得保留的经验提醒

老师举了一个很好的反例：

* 自己演示私钥时，被同学用更高的 `gas` 抢先转走了钱

这段最应该沉淀到笔记里的不是故事本身，而是两件事：

* 私钥绝对不要在公开环境里展示
* 前端和链上交易都要理解“更高优先费会影响打包顺序”

这和 [[设备与账号安全]]、[[Web3背调与风险]] 可以一起看。

### 11. `EVM` 这一段提醒的是“链上不是普通后端”

老师最后简单提到 `EVM` 的几个特点：

* 它是堆栈式虚拟机
* 有不同的存储区域
* 遍历、循环、大映射操作都要非常小心

这里的潜台词是：

* 链上执行和普通服务端逻辑很不一样
* 写得不慎，成本和性能都会出问题
