# 05-全栈课程相关讨论｜直播笔记整理

## 概览

这场直播的核心，不是在讲某个单独技术点，而是在把课程从“会用 AI”继续推向：

1. 设计稿到代码的完整链路
2. 多 agents 和自动化工作流
3. [[全栈课程路线]]
4. [[双云架构]]
5. [[Serverless架构]]
6. `EC2 + 弹性 IP + 稳定出口 + Google/Gemini 前端工作流`

老师整场最想传达的现实判断是：

* 传统纯前端的很多基础活已经被 AI 抽走
* 现在很多普通背景开发者必须尽快补全栈
* 课程后面会围绕 Cloudflare、AWS、Serverless、MCP、EC2、自动化持续推进

> [!note]
> 这份整理以 [`全栈课程相关讨论.txt`](/Users/liangsheng/Downloads/%E5%85%A8%E6%A0%88%E8%AF%BE%E7%A8%8B%E7%9B%B8%E5%85%B3%E8%AE%A8%E8%AE%BA.txt) 为准，文字稿里有不少发音 / 转写错误，涉及工具名和协议名时需要结合上下文理解。

## 一、老师开场在强调什么

老师一开始情绪很强，核心其实是三件事：

* 到课率不够
* 执行力不够
* 前面布置的作业很多人没有认真做

这决定了整场课的风格：

* 不是慢慢铺概念
* 而是边催作业、边推进课程方向
* 强行把大家拉进“能交付、能落地”的状态

## 二、课程在往哪几个阶段推进

老师这场明确说出前两层：

1. 先学 AI 的基本使用
2. 再学 AI 工作流

结合整场内容，课程实际上已经继续往这些方向推进：

* 多 agents / 自动化
* [[全栈课程路线]]
* [[双云架构]]
* [[Serverless架构]]
* 更完整的云端交付和工程链

## 三、“小龙虾”只是过渡，不是终点

老师这一场非常明确：

* 控制电脑本身不是新事物
* 真正新的地方，是现在能用自然语言 + 大模型去控制
* 一旦官方模型和系统下场，很多中间层拼装工具会很快被替代

他举的方向包括：

* `Claude` 已经开始接电脑控制
* 苹果智能会逐步接自己的办公软件和系统能力
* 将来很多“控制电脑”的能力可能会内建在官方产品里

这段真正想表达的是：

> 别把某个中间工具当主角，真正该学的是“AI 调度工作流”的思维。

## 四、前面作业真正要跑通的链路

老师对大家没交作业很生气，但同时把第一轮主线说得非常清楚：

* 根据自己的想法生成设计稿
* 用 export 或内置 MCP 把设计稿导出来
* 导进 Figma 或直接进入开发链路
* 再交给代码工具还原成代码

这场里他反复强调：

* 很多工具自己已经带 MCP
* 不一定还要额外去折腾三方中转
* 产品自己就能快速把 UI 画出来
* 设计稿不是终点，它会继续进入 Figma、代码和后续工程流

他还提到更进一步的玩法：

* 热力图
* 用户注意力区域分析
* 页面变量和视觉探索

也就是说：

> 设计稿已经从“交给前端看的图”，变成了 AI 工作流里的中间资产。

## 五、MCP、skills、ACP 到底怎么分

这一场非常重要的一点，是老师试图把几层能力分开。

### MCP

老师给的理解是：

* MCP 是上下文协议
* 类似 AI 的 USB 接口
* 让 AI 可以像插件一样调外部能力

他反复提到 MCP 能做的事包括：

* 读数据库
* 操作文件
* 调 API
* 操作浏览器
* 操作 GitHub
* 操作 Figma
* 调本地程序和云服务

### skills

老师说的 skills 更像：

* 专门用来操作某类软件的能力层
* 比如 Slack、VS Code、Electron 应用

他特别强调：

* 浏览器比较好操作
* VS Code、Slack 这类 Electron 应用更难
* 更适合用对应的专门 skills

### ACP

ACP 在这场里的意思更接近：

* 用自然语言去调用 code / CLI / 编辑器
* 通过对话驱动代码工具

老师讲的意思大概就是：

* ACP 操作代码工具
* MCP 操作外部资源
* skills 操作具体软件

> [!warning]
> 这几个词在转写里有一些混用，正式笔记里更适合保留“老师想表达的分层”，而不是逐字硬翻。

## 六、自动化的重点不是“全自动开发”，而是多 agents

老师这场很明确地说：

* 现阶段他的硬性作业重点不是“全自动开发”
* 而是先把多 agents 搭起来
* 先把重复性的信息工作自动化

他举的典型例子包括：

* 早报
* 前端技术更新
* 币圈大新闻
* 推特热点
* 技术文章收集
* 视频选题和二创剧本

他的底层逻辑是：

* 以前这些事要自己花大量时间翻
* 现在起床、上厕所的时间就可以先跑完
* 真正值钱的，是把时间留给高价值的开发和判断

## 七、为什么现在必须补全栈

老师这一场的职业判断非常强：

* 纯前端的很多基础活已经不值钱
* AI 很快就能把“写组件、发请求、渲染页面”这类活秒掉
* 如果学历、年龄、背景不占优，就更得尽快往全栈走

他反复强调：

* 名校、年轻、履历强的人，传统路径还有空间
* 但大多数普通背景开发者，更现实的是：
  * 全栈
  * 云
  * AI 工作流
  * 自动化

也就是说：

> 对很多普通开发者来说，全栈已经越来越像必要项，而不只是加分项。

## 八、双云架构：Cloudflare + AWS

这场课最重要的技术主轴，就是 [[双云架构]]。

老师的判断很明确：

* 如果以后想做远程工作、海外语境下的项目
* 更值得优先熟悉的是 Cloudflare + AWS
* 国内单一云服务不是完全没用，但不是这套路线上最关键的主角

从这场口径看，它们的分工大致是：

### Cloudflare

负责：

* 边缘入口
* DNS
* 前端托管
* 边缘层 serverless

关键词：

* Waiting Room
* Pages
* Workers

### AWS

负责：

* 数据中心
* 安全入口
* 微服务分发
* 数据库、缓存、队列、对象存储
* 更重的后端执行层

关键词：

* WAF
* Shield
* AppSync
* Lambda
* CloudWatch
* SQS / SNS / 死信队列
* RDS / Aurora
* ElastiCache
* ECS / ALB

## 九、为什么老师强推 Serverless

老师这一场对 [[Serverless架构]] 的态度非常明确：

* K8S 已经很传统了
* 对这门课和大多数同学的目标来说，先学弹性云和 Serverless 更现实

他给的理由主要有：

* 更便宜
* 更适合创业和中小团队
* 更适合这套 AI + 自动化工作流

他还专门做了一个认知转换：

* 以前要买机器、SSH、SCP、手动布服务
* 现在很多时候不需要这样了
* “云函数”只是其中一种
* `Serverless` 的概念比“云函数”更大

## 十、架构图到底在讲什么

这场里老师其实就是在口头展开那张 [[Cloudflare 与 AWS 混合云架构]] 图。

主线可以压成这样：

1. 用户先进入 Cloudflare edge
2. 大流量时先进 Waiting Room
3. 纯前端托管在 Pages
4. 边缘服务跑在 Workers
5. 深层数据和核心服务进 AWS
6. AWS 用 WAF / Shield 做安全
7. AppSync 负责 GraphQL 一类网关
8. Lambda 负责 Serverless 微服务
9. CloudWatch 负责日志
10. RDS / Aurora / ElastiCache / S3 / 队列体系负责数据和异步
11. 更重的常驻服务再上 ECS / ALB

这张图真正想帮大家形成的是：

> 前端、边缘层、后端、异步、数据库和安全，不再是散着记，而是放进一条完整的交付链。

## 十一、接口层：GraphQL / tRPC / gRPC 的关系

这一段口述和转写有点混，但能确认几件事：

* GraphQL 长期必须补
* 在 Web3 语境里，一些接口协议会非常常见
* 当下作业不要卡死在“某个协议名”上
* 更重要的是把接口层和 Worker、MCP、云服务一起跑通

更稳妥的记录方式是：

* 长期要补：GraphQL / gRPC 这类能力
* 当下先做：更容易让你把作业跑起来的那条接口链

> [!note]
> 文字稿里 `GraphQL / tRPC / gRPC` 有混用和转写误差，建议后续结合你实际作业栈再统一。

## 十二、今天的作业到底是什么

这场课的作业已经明显进入完整工程链。

### 1. Cloudflare 侧

* 注册 Cloudflare
* 买域名
* 绑定到 Cloudflare
* 用它的 CI/CD 构建前端项目

老师建议：

* 域名不要去阿里云买
* 去 GoDaddy 一类地方买
* 便宜域名就够

### 2. 前端项目

* 用 React 起项目
* 老师建议直接用 `Vite`

### 3. Workers 侧

* 部署一个 Worker
* 用 `export default` 起服务
* 用 Worker 去请求 `DeepSeek` 或 `OpenAI`

### 4. 接口层

这场转写最稳妥的理解是：

* 接一个你当前能跑通的接口层
* GraphQL / tRPC 都在老师口中出现过
* 不要因为名词卡住主任务

### 5. MCP 侧

* 选 1 到 2 个 MCP
* 用它解决一个实际问题
* 不要只是“装了”，而是要真的用起来

### 6. 提交物

这场里反复提到的交付内容包括：

* 域名
* 项目链接
* GitHub 地址
* 项目文档
* 自动化测试结果
* 测试报告
* 中途遇到的问题和心得
* 录屏 / 视频

也就是说，这次不是“做个 demo 看看”，而是：

> 域名 + Cloudflare + React + Worker + 接口 + MCP + 文档 + 测试

这整条链都要开始熟起来。

## 十三、AWS MCP 最大的风险：别装错

这场里老师专门停下来讲了一件事：

* AWS 相关的 MCP 很多
* 不能随便装
* 尤其不要装来路不明的个人版

他给出的原则很明确：

* 先问 AI 哪个是官方的
* 认官方标识
* 认官方认证

也就是说：

> AWS MCP 不是越多越好，而是越官方越稳。

## 十四、为什么老师一直逼大家做 EC2 作业

这一场里，[[EC2 与稳定出口]] 被讲得比之前更彻底。

老师的核心逻辑是：

* 很多人担心 `Claude`、`Gemini` 一类服务被封
* 市面上的共享代理、共享住宅 IP 不稳
* 最稳的方式之一，就是自己搞一台 EC2

他把 EC2 的作用讲成两层：

### 1. 部署机器

* 你可以把项目布到上面
* 作为你云端工作环境的一部分

### 2. 稳定独享出口

* 配弹性 IP
* 配出网
* 得到一个稳定独享的固定出口

老师口语化把它说成：

* “家庭静态住址 IP”

更稳妥的理解其实是：

* 用 EC2 + 弹性 IP 模拟出稳定独享的固定出口
* 它不等于真正居民宽带
* 但在实际使用里，已经非常接近“长期稳定的独享美国出口”

## 十五、EC2 的几个现实细节

这场里老师补了很多具体经验：

* AWS 学习账户通常就够起步
* EC2 本身可能几乎不花钱
* 真正要花的是弹性 IP 和网络侧配置
* 一个月通常也就几刀到十刀左右

他还特别强调：

* 地区要选对
* 最好选美国
* 最好选免税州
* 俄勒冈州是他特别提到的一个选择

更底层的意思是：

> 不是所有“美国区”都一样，地区和税费会直接影响你后面的成本和体验。

## 十六、Google / Gemini 家族为什么被他讲得这么重

这场里老师对 [[Gemini 前端工作流]] 的评价非常高。

他强调的优势包括：

* 对前端特别友好
* 和 Chrome 生态整合强
* 可以调试浏览器本身
* 可以暂停 Chrome 再去看问题
* 从 IDE、模型、浏览器控制台到前端调试，几乎是一整套

他还给出很强的判断：

* 在还原 Figma / Sketch 到 HTML 这件事上，这条线非常强
* 某些前端场景下，几乎像是“为前端而生”

## 十七、这条 Gemini / Chrome 路线的前提条件

老师同时也讲得很现实，这套东西不是随便就能顺畅跑的。

他反复强调的条件包括：

* 美国 IP
* 英文系统
* 英文浏览器环境
* 美区账号体系
* Gmail / location 等环境信息要对

他还演示了一种激活浏览器能力的思路：

* 先在纯英文环境下
* 用英文命令激活浏览器操作
* 点 `let's go` 一类的引导入口
* 激活以后再回到中文对话

更稳妥的结论是：

> 这条线非常强，但对环境洁净度和一致性的要求也非常高。

## 十八、为什么老师一直说“IP 不行，啥都白扯”

这场里，老师几乎一直在讲：

* 网不好
* IP 不稳
* 地区不对
* 代理不行

那很多能力就根本跑不起来。

这包括：

* Chrome 侧能力
* Gemini 侧能力
* 浏览器控制
* 登录体验
* 调试体验

所以这场一个非常现实的底层结论就是：

> 在这类工作流里，网络和出口不是辅助条件，而是基础设施。

## 十九、老师后面还想继续讲什么

这场课临近结束时，还提到后面准备继续往下讲：

* VPC
* 子网
* 安全组
* NAT Gateway
* Endpoint

他的意思是：

* 把服务布到云上不等于结束
* 真正落到线上后，还有一整套网络和安全配置要理解

这也说明：

> 这门课已经不只是“前端加一点后端”，而是在往完整云上工程系统走。

## 二十、这场课最值得执行的版本

如果把整场直播压缩成真正该执行的版本，大概就是：

1. 补上前面的设计稿作业
2. 搞清楚 MCP / skills / ACP 各自干什么
3. 接受“纯前端基础活正在被抽走”这个现实
4. 开始把脑图切到 [[双云架构]] 和 [[Serverless架构]]
5. 先把 Cloudflare + React + Worker + 接口 + MCP 跑通
6. 认真把 EC2 和稳定出口这条线补起来
7. 有能力再去做多 agents 和更完整的自动化
8. 记住最终目的不是“装工具”，而是“形成可交付的工作流”
