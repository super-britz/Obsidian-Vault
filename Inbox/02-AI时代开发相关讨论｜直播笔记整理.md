# AI时代开发相关讨论｜直播笔记整理

## 概览

这场直播的重点不是讲某一个技术点，而是在讲一套新的 [[AI开发主流程]]：

1. 用 AI 做创意发散
2. 把创意变成 UI / 设计稿
3. 把设计稿接进代码生成链路
4. 借助 [[MCP工具栈]]、浏览器控制和自动化测试，把开发流程串起来
5. 最后进入部署、运维、日报 / PR / 周报等更完整的工作流

老师的判断非常明确：

* AI 不只是聊天工具，而是研发流程中的执行层
* 会不会用 AI，不只是会提问，而是会不会把整条流程跑通
* 以后更值钱的角色，是懂业务、懂架构、懂流程控制、会审结果的人

## 核心结论

这场直播最值得保留的结论有几条：

* 现代开发流程已经从“人手工串联”转向“AI 参与全链路”
* 真正重要的是把 AI 接进创意、设计、代码、测试、部署、运维和汇报
* 面试也越来越偏向真实问题诊断，而不是只背理论
* AI 会接管大量标准化动作，但 [[企业落地判断|架构和业务理解]] 暂时仍然离不开人
* 个人要先学会手动跑通流程，再进入自动化和“AI 看 AI 干活”的阶段

## [[AI开发主流程]]

### 1. 创意生成

老师认为第一步仍然是先有想法，但现在可以先用 AI 做创意发散。

他提到：

* 文字类 AI 适合做产品创意和方向发散
* 可以打开 `Deep Research / 深度思考` 之类能力，让结果更长、更像研究报告
* 这一步不是直接写代码，而是先把“要做什么产品”想清楚

### 2. 设计稿生成

老师重点提到：

* `Figma Make` 可以把文本创意直接转成 UI 图
* Figma 的价值不仅在于出图，更在于它和团队协作、开发衔接得更紧
* 生成结果虽然有时会带一点“AI 味”，但足够用来快速起版
* 如果想要更漂亮的 UI，他明显更偏向 Gemini 相关能力

> [!note]
> 他的核心思路是：先用 AI 快速出一版，再人工继续改，而不是一开始全部手工做。

### 3. 设计稿转可编辑资产

老师讲了一个桥梁步骤：

* 用 `HTML to Design` 之类插件
* 把代码或生成结果再导入 Figma
* 让原本不能编辑的生成结果，重新进入团队可协作的设计流程

### 4. 通过 [[MCP工具栈]] 接入外部能力

这一段是整场直播很核心的部分。

老师把 MCP 理解为：

* 让 AI 以标准化方式调用外部能力
* 不只是聊天，而是可以接 GitHub、浏览器、设计稿、本地文件、脚本、地图、表格等
* `mcp.so` 可以理解成 MCP 的索引 / 市场入口

### 5. 代码生成

老师的意思是：

* 有了 Figma 链接、MCP 和设计稿之后，代码生成会明显加快
* `Claude / Claude Code` 仍然是他非常推崇的主力
* 某些客户端工具会比 VS Code 插件更强
* 新工具对 Mac M 系列芯片更友好

### 6. [[自动化测试]]

老师认为浏览器控制和自动化测试是下一阶段的关键：

* AI 可以直接控制浏览器
* 自动点页面、跑流程、分析性能
* PRD 还可以继续喂给自动化测试工具，生成测试脚本和测试报告

### 7. 部署与运维

老师希望大家感受一条完整链路：

* 基础部署到云服务器
* 能给出可访问链接
* 再逐步进入远程运维和自动化上线

### 8. 汇报与协作

老师还提到后面连这些都能被 AI 接进流程：

* 发日报
* 发周报
* 生成 PR
* 整理上线通告

## [[MCP工具栈]]

老师重点提到的 MCP 方向包括：

* `GitHub MCP`
* `Chrome / Chrome DevTools MCP`
* `Figma MCP`

### GitHub MCP

他的意思是：

* AI 可以更直接接进仓库、项目、代码管理动作
* 很多原本手点的流程，会被自然语言任务取代

### Chrome / Chrome DevTools MCP

这是他最强调的一个实战点。

他认为这个特别适合：

* 分析页面性能问题
* 看火焰图
* 定位 Long Task
* 看布局和错误信息
* 直接输出性能问题和对应建议

他还把这和 [[面试变化]] 直接联系起来：

> 以后面试更可能是给你一个网址，让你指出问题并给解决方案，而不是只背性能优化概念。

### Figma MCP

他提到：

* 设计稿链接可以直接交给 Claude 或代码工具
* 让设计稿直接进入组件生成链路
* 设计稿不再只是给人看，而是越来越像 AI 的输入源

## [[AI工具栈]]

这场直播里出现的工具方向，大致可以分成几类：

* 创意与研究：ChatGPT、Deep Research 一类能力
* 设计与 UI：Figma、Figma Make、Gemini
* 代码生成：Claude / Claude Code、AI Studio 一类入口
* 浏览器与测试：Chrome DevTools、浏览器控制、自动化测试工具
* 云端与部署：AWS、EC2、阿里云等基础环境

老师的底层判断是：

> AI 不是一个工具，而是一套平台化工作流。

## [[自动化测试]]

老师对自动化测试的理解不是传统意义上的“只写测试用例”，而是：

* 让 AI 控制浏览器
* 跑功能流程
* 分析性能
* 根据 PRD 生成测试脚本
* 输出测试报告

他的一个关键观点是：

* PRD 不只是产品文档
* PRD 还可以成为自动化链路的上游输入

## [[企业落地判断]]

老师对企业落地有几个强判断：

* 创业公司更容易快速采用这套流程
* 大公司不是不用，而是推进更慢、更重视合规和安全
* AI 会替代大量标准化、流程化、重复性的研发动作
* 最后留下来更值钱的人，是懂业务、架构、规范、验收的人

## 老系统怎么接入

这一段也很重要。

老师的思路是：

* 先让 AI 读现有业务逻辑
* 把整理后的业务知识沉淀进 `.docs`、`.ai`、`.claude` 这类目录
* 后续 AI 开始工作前先读这些沉淀好的上下文
* 老业务人员再审一遍，修正理解错误

这背后的方法其实是：

* 先做知识沉淀
* 再做流程复用
* 再做自动化执行

## [[作业要求]]

老师给这次作业的目标非常明确：

* 先做一个小而完整的产品
* 不求大，但要把链路跑通
* 今天先手动串流程
* 节后再进阶到“让机器人 / AI 自动执行流程”

适合保留的方式是：

* 把高概率错词统一纠正

## 个人可执行版本

如果把这场直播压缩成值得执行的版本，可以记成：

1. 先用一个小产品把创意、设计稿、代码、测试、部署串起来
2. 先学会手动跑一遍，再考虑自动化
3. 重点补 MCP、浏览器控制、设计稿转代码这些桥梁能力
4. 不要只记工具名，要记住每个阶段的输入和输出是什么
5. 真正值钱的，仍然是业务理解、架构判断、审结果能力
