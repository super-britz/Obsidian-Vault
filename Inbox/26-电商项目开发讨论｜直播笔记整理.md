# 26-电商项目开发讨论｜直播笔记整理

## 概览

这场直播虽然标题是“电商项目开发讨论”，但真正讲的已经不只是“把一个商城项目搭起来”，而是在拿一个最小电商 Web3 项目，去演示一整条 AI 自动化研发工作流：

1. 用 [[Hardhat]] 初始化合约工程
2. 用 `Vite + React + TypeScript` 初始化前端工程
3. 先由人把架构和 UI 边界定好
4. 再生成 `.cloud` 规则、`PRD -> specs -> tasks`
5. 然后把编码、review、上下文压缩、记忆、守护、测试和安全扫描串起来

这场最有价值的地方，不是“又做了一个电商 Demo”，而是老师把几件事说得非常清楚：

* AI 不适合替你拍脑袋定架构，架构和边界最好先由人定
* 自动化工作流真正值钱的，不是“会生成代码”，而是 `init -> PRD -> spec -> task -> code -> review -> 测试 -> 守护`
* 复杂项目里，最影响交付质量的不是模型名，而是需求颗粒度、上下文控制和项目记忆怎么存

它和这些笔记是直接连着的：

* [[25-Web3项目开发会议｜直播笔记整理]]
* [[前端新时代开发工作流]]
* [[Skill 设计模式与任务编排]]
* [[AI开发主流程]]
* [[前端架构设计指南]]
* [[Web3前端开发三步走]]

## 这场里需要先纠正的错词和概念

### 高概率错词

* `headhatch / headhatch / head project` 基本都是 `Hardhat`
* `mosh and Easter` 基本是 `Mocha + ethers`
* `await` 在前端初始化语境里基本是 `Vite`
* `red typescript` 基本是 `React + TypeScript`
* `telegram 加这个 red` 高概率是在说 `Tailwind + React`
* `一台房子测试网` 基本是 `以太坊测试网`
* `TRP / HDP 网络配置` 高概率是默认网络 / `HTTP` 或 RPC 配置的转写漂移
* `cloud / codis / coats` 这一段大多是在讲 `Claude`、Codex 一类编码工作流工具混着说
* `UIDT / UIDC` 是 `USDT / USDC`
* `1320` 基本都是 `ERC20`

### 需要纠正的几个概念

* `.cloud` 不是官方唯一标准答案，更像老师自己围绕项目上下文做的一套工作流组织方式
* `PRD -> spec -> task` 的重点不是“格式漂不漂亮”，而是颗粒度是否适合 AI 真正执行
* 样式和 UI 不是不能自动化，而是老师更建议在主编码工作流之前先定好界面，避免边写逻辑边改样式
* 项目记忆在开发场景里更适合“可 review、可删、可审计”的结构化文档，而不是只靠向量记忆
* 小项目默认不一定需要多 agent 并行；粒度足够细时，单 agent 串行反而更稳

## 核心结论

* 新项目最适合的顺序不是“先让 AI 全自动写”，而是“人先定架构和页面边界，再把 AI 接进执行流程”
* 老师把项目拆成了两个独立仓库：合约端和前端端；对这种双仓库 Web3 项目，自动化工作流必须先学会识别项目结构
* `.cloud`、`PRD`、`specs`、`tasks` 这些文档不是附属物，而是整个自动化流程的骨架
* 需求拆得过粗，AI 就容易跑偏；拆得足够细，单 agent 串行往往就够用
* 项目长久记忆最好落在项目目录里，用结构化文档维护，不要完全依赖外部记忆工具
* “守护”是这类工作流真正绕不开的问题：不是流程设计得出来就万事大吉，还要考虑 VS Code 卡住、网络断开、会话挂死时怎么办
* 测试和安全扫描不一定要混进主编码流程里，拆成独立 skill 或独立流程反而更稳

## 一、00:11-03:37 先把电商项目骨架搭起来

### 1. 合约端先用 [[Hardhat]] 起项目

老师开场先搭的是合约工程，主线很清楚：

* 新建项目目录
* 用 `pnpm` 初始化
* 选 `TypeScript` 模板
* 选更熟悉的 `Mocha + ethers` 测试路线

这段最值得记住的是：

* 老师更推荐 `pnpm`
* 不是因为其他包管理器不能用
* 而是对大多数本地开发机来说，`pnpm` 够顺、够省事

### 2. 新项目不是直接开写，而是先把示例代码清干净

老师在 `Hardhat` 初始化后，第一件事不是写业务，而是：

* 看看默认的 `counter` 示例
* 决定删不删
* 初学者可以保留做参考
* 熟了的人就直接清掉，准备一个干净骨架

这个判断很现实：

* 初学阶段，示例文件有学习价值
* 真做项目时，示例代码往往会成为干扰项

### 3. 远程部署之前，基础配置要先准备好

老师这里强调了几样“别等 AI 帮你猜”的基础项：

* 钱包私钥
* RPC URL
* 测试网配置
* 环境变量

他真正想表达的是：

* AI 可以帮你补配置
* 但这些关键前提你不能完全不懂
* 不然你连它在花谁的钱、往哪发交易都说不清

### 4. 这一段其实已经在强调“AI 不负责替你决定基础工程边界”

老师说得很直接：

* 项目骨架、人该清掉的代码、人该准备的配置，最好先由人处理
* 等基础架子对了，再让自动化工作流接进来

这和 [[前端架构设计指南]] 的逻辑很一致：

* 先定义问题和边界
* 再让 AI 加速

## 二、03:37-11:19 再把前端工程独立出来

### 1. 这是一个双仓库项目，不是单体一锅炖

老师这里的结构非常明确：

* 一个仓库放电商合约
* 一个仓库放电商前端

也就是说他想做的是：

* 合约端独立
* 前端端独立
* 最后再通过 workspace 或工作区把它们放在一起协作

这个设计其实就是在给后面的自动化流程做准备：

* 自动化工具先要知道你在操作哪个仓库

### 2. 前端初始化走的是 `Vite + React + TypeScript`

虽然转写里有不少错词，但主线很清楚：

* 前端用 `Vite`
* 技术栈是 `React + TypeScript`

老师中间还高概率提到了：

* `Tailwind + React`

但这段他的重点不在“前端库名”，而在于：

* 先把前端基本架子定出来

### 3. 架构阶段尽量别偷懒交给 AI

老师这一段最值得记的是态度：

* 他对“让 AI 直接定架构”是不太放心的
* 尤其新项目、真实项目、接别人的屎山项目时更是如此

他的真实判断是：

* 架构准备阶段，最好人坐在电脑前
* 这一步不是自动化流程最适合接管的地方

### 4. UI 最好先画好，再进主工作流

老师给了一个很明确的建议：

* 页面、样式、UI 边界先画好
* 再把编码工作流接进来

原因不是 AI 画不了样式，而是：

* 一边定样式一边写逻辑，工作流更容易跑偏
* 如果 UI 都定完了，后面自动化流程会更顺

这点可以直接接到 [[前端新时代开发工作流]]：

* 先定设计和实现边界
* 再让自动化链路推进

## 三、11:19-23:56 `.cloud` 初始化：工作流真正的起点

### 1. 老师把 `.cloud` 看成整个项目的“项目说明书 + 执行规则”

这一大段的核心其实只有一句话：

* 没有 `.cloud`，后面的自动化工作就没有指导

老师这里把 `.cloud` 的作用理解成：

* 项目结构说明
* code style
* 测试约定
* git flow
* 安全规则
* 子目录规则

换句话说，它不是“又一个文档目录”，而是：

* 让 AI 进入项目后先读的上下文入口

### 2. 他不喜欢把规则写得太死

老师现场改命令定义时，反复做了一件事：

* 把过度死板的分支和条件删掉

比如：

* 项目结构探测不要写太死
* 工种不要事先框死
* 前端 / 后端 / 合约这些层次，不要提前替 AI 把路堵满

他的理由其实很现实：

* 规则写太死，工作流反而变笨
* 该还给 AI 的判断能力就还给 AI

### 3. 单项目和 monorepo 的处理方式不一样

老师这里点得很清楚：

* 如果是单项目，就没必要把规则拆得像大型 monorepo 一样复杂
* 如果以后要兼容 monorepo、合约、RN、前后端多模块，再去统一拆分

这意味着 `.cloud` 的设计不是固定模板，而是：

* 跟项目结构强绑定

### 4. 这套流程里实际有三个自定义命令角色

虽然转写里命令名有些漂移，但老师这场的动作链很清楚，核心大致是三步：

* `init`：分析当前项目，生成 `.cloud` 和规则
* `PRD`：读取需求文档，生成 `specs / tasks`
* `AI`：读取 task，按顺序执行编码与修改

它们合起来，其实就是老师自己定义的最小自动化流水线：

* 先建项目规则
* 再拆需求
* 最后执行任务

### 5. 这一段其实在讲“工作流也要做产品设计”

老师反复在改：

* init 的话术
* 项目结构探测逻辑
* 生成的规则层次

这说明一个很重要的判断：

* 自动化工作流本身也需要产品设计
* 不是从网上抄一套 prompt 就能稳定跑

这和 [[Skill 设计模式与任务编排]] 是直接一致的：

* 不是 skill 越多越强
* 而是流程设计得合不合理

## 四、23:56-35:58 从 PRD 到 specs 再到 tasks

### 1. `PRD -> specs -> tasks` 才是老师真正想演示的主线

项目骨架准备完之后，老师立刻进入：

* 初始化 PRD
* 根据 PRD 生成规格
* 再把规格拆成可执行任务

这段最重要的不是某个命令名，而是：

* 产品需求必须先成形
* 不能让 AI 直接从一句模糊描述开写

### 2. PRD 的来源可以有很多种

老师这里列了几种来源：

* 直接打一段文字
* 现成文档
* 浏览器里抓取需求
* 本地项目里的 PRD

但不管来源是什么，后面都要落到一件事：

* 把需求变成结构化、可迭代、可 review 的材料

### 3. specs 不要拆得太粗

老师在这段里反复强调：

* specs 拆分粒度很重要
* 如果功能点太大，后面 task 也会太大
* task 太大，AI 执行就容易超上下文，或者做着做着就跑偏

他的偏好非常清楚：

* 尽量按功能点、按编号、按小任务拆分

这不是为了格式好看，而是为了：

* 让 AI 真能一段一段做完

这里还有一个很实际的补充：

* 如果 PRD 本身就很简单
* 那它最后拆不出很多层级任务，其实是正常的

老师这场的电商需求就属于：

* 功能不复杂
* 重点是把流程跑通

### 4. 开放性问题要么前置问清楚，要么后面人工补答

老师这里给了两条路线：

* 一种是先生成，再由人看开放问题再补
* 一种是边生成边对话确认

他自己的偏好更偏第一种：

* 先生成
* 再 review 和补答

但也承认第二种更稳：

* 前期沟通成本大
* 后面返工会更少

### 5. specs 更适合放在独立目录，不要硬塞进代码仓库

老师中间明显改了一次自己的做法，最后更偏向：

* 不要把 docs/specs 生硬塞进前端代码仓库里
* 更适合给它一个单独目录或单独项目

这样做的好处是：

* 需求文档和代码职责更清楚
* 前后端 / 合约仓库之间更容易复用
* review 和版本管理更顺

所以他后面才会把它调整成类似：

* `一灯shop/specs`

这种独立位置，而不是继续往 `front-end` 仓库里塞。

### 6. 这场电商 PRD 里最典型的开放问题

老师现场举出来的这些问题，很适合直接沉淀成笔记：

* 使用 `USDT` 还是 `USDC`
* 测试网上有没有可用的稳定币地址
* 商品是否限制库存
* 已购买商品展示范围是什么
* owner 添加商品是走脚本还是走后台页面
* 钱包接入是否要 UI 组件

这说明一个很重要的事实：

* 很多返工不是代码写错
* 而是这些开放问题一开始没问清楚

### 7. 这场最后把支付代币切成了 `USDC`

老师中间在 `USDT / USDC` 上来回纠结，最后更偏向：

* 测试里直接用 `USDC`

原因非常实际：

* 更省事
* 获取路径更明确

这里顺手还踩了一个非常值得记的坑：

* 他一度把测试币转到了合约地址而不是自己的钱包地址

这个细节很值钱，因为它说明：

* 链上地址和用途必须分得非常清楚

### 8. “购买记录只靠前端传地址”这块有安全问题

老师这里点到了一个很重要但没完全展开的问题：

* 如果购买记录只靠前端传地址给合约或后端
* 理论上就可能被伪造

他的方向判断是：

* 后续应该补签名验证

这点非常值得单独记下来，因为它说明：

* 即使是最小电商 DApp，身份和所有权校验也不能只靠前端说了算

## 五、35:58-50:30 执行阶段：一灯 AI、上下文和谁来守护

### 1. 这场后半段已经不在讲“怎么写功能”，而在讲“怎么让流程跑下去”

老师这里开始关心的核心问题变成了：

* specs 在哪
* tasks 在哪
* 命令怎么读取
* 上下文怎么切
* 谁来盯着它别挂

这就是自动化项目真正的难点：

* 不是第一步跑起来
* 而是一路别死

### 2. 上下文管理是这类工作流的硬约束

老师这里给出了很清晰的策略：

* 每个任务尽量只吃自己的上下文
* 到 80% 以上时就考虑压缩上下文
* 每个 task 做完后，尽量开新会话或清理上下文

他的逻辑非常实用：

* 任务没做完不能乱切
* 任务做完就不要把旧垃圾带进新任务

这和 [[AI开发主流程]] 里强调的“上下文不是越长越好”是一致的。

这里还有一个非常具体的执行习惯：

* 每次重新跑新需求前，最好先清掉旧会话记忆
* 不然模型很容易把上一次需求里的开放问题和结论带进来

老师现场就反复踩到了这种“旧上下文污染新任务”的坑。

### 3. 真正难的是“谁守着它别挂”

老师把“守护”单独提出来，说明这已经是实战痛点：

* VS Code 可能卡住
* 会话可能不动
* 网络可能断
* 电脑可能死机

所以他给出的守护方案是很现实的一套：

* 远程桌面
* SSH
* TMUX
* 手机接入电脑
* 能兜底就行

这跟 [[浏览器控制与远程模式]] 是一条线上的：

* 自动化不只是会跑
* 还要有人能接管

### 4. 这里顺手点了一个判断：理论上固定流程不容易断，现实中工具还是会莫名停

老师说得很实在：

* 按 task 一路往前走的流程，从逻辑上不复杂
* 但工具层还是会出现“没报错、就是不动了”的情况

所以“守护”这个词不是多余设计，而是：

* 真实交付里必须预留的一层

### 5. “守护”不是只为了继续跑命令，也是为了能人工接管

老师这里的真实需求不只是“看着它别挂”，还包括：

* 如果自动化工具卡死
* 如果编辑器不响应
* 如果某个软件弹窗

那人要能马上接回电脑继续打字、继续推进。

所以他提到的远程桌面、SSH、TMUX、手机接入，本质上都在解决一件事：

* AI 流程失灵时，人要能无缝接手

## 六、57:19-01:06:58 项目记忆、Agent/Skill、多 agent 和测试安全

### 1. 这场对“项目记忆”的判断很鲜明

老师不太看好把开发记忆全丢给外部向量记忆工具。

他的理由非常工程化：

* 向量记忆适合聊天
* 开发记忆更需要精确、结构化、可 review

所以他的偏好是：

* 直接把长久记忆写进项目目录
* 比如写进 `specs` 旁边的结构化文档

这样做的好处是：

* 能看见
* 能删
* 能提交
* 能 review

### 2. `.cloud`、`specs`、附加记忆文档要同生命周期维护

老师这里给的实际建议可以压成一句话：

* 项目记忆要和项目本身一起活着

也就是说：

* 不要把关键决策、踩坑、架构变化只寄托在某个外部聊天记忆里
* 最稳的是把它们写成项目内的文档

老师这里甚至给出了一个非常直接的倾向：

* 与其引入一堆外部 memory 工具
* 不如老老实实在 `specs` 旁边加一个记忆文档

因为这样最符合开发场景里的：

* 精确
* 可 review
* 可删除
* 可提交

### 3. 这里再次区分了 [[Agent 与 Skill]]

老师后半段对这两个概念又展开了一次：

* `skill` 更像能力包、规则包、脚本包
* `agent` 更像带独立上下文的执行者

而他这场的判断是：

* 小项目默认没必要一上来多 agent 并行
* 只要你把 task 粒度控制好，单 agent 串行就够

### 4. 多 agent 不一定更好

老师这里给出的判断非常重要：

* 并行确实快
* 但也更复杂
* 当前这个电商项目的任务粒度还没有大到必须上 agent team

换句话说：

* 多 agent 是一种能力
* 不是默认必须开启的配置

### 5. 测试和安全扫描，他不想直接塞进主编码流程

老师后面把测试和安全扫描拿出来单独讨论，态度很鲜明：

* 主编码流程里先别塞太多附加环节
* 测试和安全可以拆成独立 skill 或独立流程

这个判断其实很稳：

* 主流程负责把功能做出来
* 测试和安全更适合独立执行、独立 review

这和 [[自动化测试]]、[[前端自动化测试]] 是直接连着的。

### 6. 技术测试与安全扫描更像“企业附加能力”

老师对安全扫描的态度是：

* 这是后期加分项
* 老板一看会觉得你这套东西更专业
* 但它不一定要跟每个 task 强耦合

更适合记成：

* 这是项目后期的独立质量保障链路

### 7. 公司级 skill 只有在“内部约束很强”时才值得做

老师后面提到公司内部封装框架、内部习惯、内部规范时，给了一个很实际的结论：

* 不是所有流程都值得专门做成 skill
* 只有当公司里有很多特殊框架、特殊要求、特殊顺序时，才值得额外做公司级 skill

这说明他对 skill 的态度是：

* 能少则少
* 真有必要再封装

## 七、这场里最值得沉淀的工作流判断

### 1. 架构先于自动化

老师最核心的判断之一就是：

* 架构、项目边界、前后端拆分、合约位置、人先定
* 自动化接的是“执行层”

### 2. PRD 拆分质量决定返工成本

这场反复证明了一点：

* PRD 不是文书工作
* 是自动化代码质量的上游输入

### 3. 记忆必须可见、可审、可删

老师这里明显更偏向：

* 项目内结构化文档

而不是：

* 黑盒式外部记忆

### 4. 单 agent 串行是默认稳态

只要 task 颗粒度足够细：

* 单 agent 串行就能完成大部分项目

### 5. 守护是工程问题，不是 prompt 问题

真正让流程失败的，经常不是 prompt 写坏了，而是：

* 工具挂了
* 网断了
* 编辑器卡了
* 会话没续上

### 6. 100% 全自动交付还不现实

老师最后的判断也很诚实：

* 现在复杂业务要做到 100% 丝滑交付，还是有难度
* 但做到 80%~85% 已经非常能打

这也是这场最适合保留的一句话：

> 先把 80% 的自动化链路跑稳，再靠人把剩下 20% 缝起来。
