# 10-前端开发事宜讨论｜直播笔记整理

## 概览

这场直播本质上是把：

* [[03-前端开发相关讨论｜直播笔记整理]]
* [[08-前端项目相关讨论｜直播笔记整理]]
* [[09-AI技术应用研讨会｜直播笔记整理]]
* [[前端架构设计指南]]
* [[全栈课程作业要求]]

这几条线进一步往“实际开发怎么落地”推进。

老师这场重点不是继续讲某一个新工具，而是在讲：

1. 如果 UI 已经被 AI 生成出来了，前端下一步到底该怎么接
2. 前端和后端在 AI 提示词上的差异到底在哪里
3. 组件库和脚手架生成物到底该怎么取舍
4. 当天作业为什么要同时把 `Hono + Lambda + VPC + GitHub Token + 数据库` 串起来

> [!note]
> 这份整理按主线重构。文字稿里有不少口音词和转写漂移，尤其集中在 `Better-T-Stack`、组件库名字、`React Router`、`Hono`、`Tailwind` 一类术语上。凡是不够确定的，我都统一挂到 [[风险与疑似错词]]，没有硬改成“绝对结论”。

## 核心结论

这场直播最值得记住的结论有几条：

* 如果你已经拿到了可用的 `HTML / CSS`，很多时候没必要再绕回设计稿链路
* 前端提示词更像“拆结构、抽组件、抽样式”，后端提示词更像“压上下文、比方案、审架构”
* 脚手架给你的初始栈不一定是最终答案，组件层往往要自己清理和重组
* AI 能把前端 UI 快速拉到可用，但你仍然要负责选型、清理、抽象和验收
* 当天的 AWS 作业不是额外插入，而是在给后面真实项目做承上启下的准备

## 一、老师先把“UI 够不够用”这件事说得很现实

老师这场开头其实在说一件很朴素的事：

* 美感本来就没有统一标准
* 有人会觉得这套 UI 漂亮
* 有人会觉得丑
* 但对于大部分接私活、做小项目、快速出活的前端来说
* 只要能把风格统一、明暗主题补上、交互做顺，就已经够用了

他的态度非常明确：

* 不要把“审美完美主义”当成一切工作的前提
* 先把一个“能交付、能上线、能继续开发”的 UI 搭出来更重要

老师还顺带提了一句很有信息量的经历：

* 疫情那会他有一段时间就是专门给团队画 CSS
* 每天的工作几乎就是精细地写页面样式
* 所以他并不是不重视样式，而是今天他更倾向于把 AI 当成“先把 80% 搭出来”的工具

## 二、如果代码已经有了，就别总想绕回 Figma

这场最核心的一句判断其实是：

> 如果我已经拿到代码了，我为什么还要把它再绕回 Figma？

老师把当前常见的几条做法讲得很清楚：

### 方案一：导到 Figma，再走 MCP，还原回代码

这是可以做的，适合：

* 想在设计稿里继续精修
* 想人工微调结构和视觉
* 想借助 Figma 插件再做公共样式抽离

### 方案二：既然已经拿到 `HTML / CSS`，就直接转成前端组件

这是老师更偏好的方式，原因包括：

* 少一次来回折腾
* 还原更直接
* 已经接近 `100%` 视觉结果
* 后面拆组件、转 TypeScript、接路由、补业务逻辑更顺

他的意思不是说 Figma 没价值，而是：

* 如果目标是继续开发
* 那代码比图更接近真实交付

这一段和 [[前端新时代开发工作流]]、[[前端架构设计指南]] 是能接起来的。

## 三、老师反复强调：还原不好，很多时候不是工具问题，而是提示词问题

这一场又把 [[AI开发主流程]] 里的一个关键点讲深了一层：

* 不是“丢一张图进去”
* 不是“给我还原页面”
* 而是你得把前端任务拆成 AI 能稳定执行的结构

### 他给出的前端提示词思路

老师现场基本就是按这种逻辑在写：

* 页面里有 `header / footer`
  * 先抽成公共组件
* 公共资源和公共样式先抽离
* 业务页面样式放到页面自己的层
* 通用组件样式放到 UI 组件层
* 遇到文本框、按钮、卡片这种组件要单独封装
* 业务组件要复用
* 图片资源放到明确的目录
* 原来脚手架里没用的组件和垃圾代码不要保留

老师真正想强调的是：

* 前端提示词不是让 AI 猜
* 而是要把“结构、归属、复用、抽离”一次讲清楚

### 他对“还原不了”的态度很直接

老师这场几乎一直在说：

* 还原不是还原不了
* 更多是你没把需求说清楚
* 没把组件怎么拆、样式怎么归、资源放哪、公共层怎么处理讲清楚

这和 [[08-前端项目相关讨论｜直播笔记整理]] 里讲“先拆页面、再拆组件、再抽公共样式”是同一条方法线。

## 四、Figma 这条线并没有被否定，但老师更偏向极简工作流

老师这场没有否定 Figma，反而提了两种更细的用法：

### 1. 直接粘贴到 Figma

如果你想继续在设计协作里看它、改它、补它，那就把结果粘过去。

### 2. 使用专门抽离公共样式的插件

他提到了一类 `Figma DTC` 之类的插件，作用大致是：

* 帮你总结公共颜色
* 抽离重复样式
* 帮你把页面里的设计 token 再整理一遍

但老师个人非常明显偏向极简派：

* 不愿意装太多插件
* 能少一层中间步骤，就少一层
* 生成公共样式这类事，他很多时候宁愿直接交给 AI 去做

所以这场里一个很明显的信号是：

> 前端的目标不是“把工具堆满”，而是“把流程变短”。

## 五、这场真正进入了前端实战项目初始化

老师这场正式把项目拉到了新的脚手架里，核心其实就是：

* 用一套现代 monorepo 初始化
* 再把前端页面和后端能力一点点填进去

### 现场选择的大致技术栈

从老师现场口径看，核心大致是：

* 路由：[[React Router]]
* 后端：[[Hono]]
* 运行环境：`Node.js`
* 数据库：`PostgreSQL`
* 接口层：[[tRPC]] / `oRPC`
* monorepo：[[PNPM 与 Monorepo]] 这条线

### 老师讲 ORM 那一段其实也很重要

虽然转写有些漂移，但上下文基本能看出来，他在比较的是：

* 一个更轻、更接近 SQL、性能更强、适合 serverless 的 ORM
* 一个更偏 schema-first、自动化体验更强、GUI 和开发体验更完整的 ORM

老师的重点不是背名字，而是让大家形成这种判断：

* 是想更轻量、更贴近 SQL
* 还是想更自动、更省心
* 以及它和 serverless、金融系统、复杂查询之间的关系

这一段更适合回链到：

* [[全栈课程路线]]
* [[技术选型方法]]

## 六、老师专门花了不少时间讲 tRPC / oRPC / 区块链 RPC 不要搞混

这场里这部分信息量挺大，容易被忽略。

老师其实在讲三层不同概念：

### 1. tRPC / oRPC

这一层是应用开发里的：

* 前后端共享类型
* 前端像调函数一样调后端
* 少掉一大堆 `fetch + any + 手写类型` 的成本

### 2. 区块链里的 RPC

这一层是：

* 前端或客户端去请求链节点
* 通过 JSON-RPC 去拿链上数据

### 3. 其他容易混的词

老师顺带提到：

* `WebRTC`
* `gRPC`

他想表达的核心是：

* 这些词都带 `RPC`
* 但语义和场景完全不同
* 如果连这个都混着说，后面做全栈和 Web3 会很容易乱

## 七、老师对付款能力也做了一个很实用的判断

这场里脚手架里带了一个付款 provider。

老师的态度很务实：

* 这类平台适合做月付 / 年付 / 免费档那种 SaaS 订阅面板
* 如果你的项目正好就是标准 AI SaaS
* 那这种模板会很省时间

但他同时也明确说：

* 当前这个项目的实际收款方式和它不完全一致
* 如果最后是 `PayPal` 或加密支付
* 那这个模板组件未必直接能用

这段的重点不是这个平台本身，而是：

> 现在很多脚手架已经把“收费面板”也做成了可插拔模块，但你还是得看实际支付链路是否匹配。

## 八、脚手架给你的不是最终项目，而只是“能开始”的项目

老师这场一个很强的态度是：

* 脚手架初始给你的东西不一定该全留
* 里面有些是模板产物
* 有些是示例代码
* 有些是你这个项目根本不需要的组件层

所以他现场做的事情并不是“照单全收”，而是：

* 先看结构
* 再删默认 UI
* 再把自己的页面塞进去
* 再决定哪些公共层保留
* 再把业务页面、公共组件和样式抽出来

### 他明确要求大家做的一件事

把项目结构和开发规范写进 `CLAUDE.md / cloudmd` 这一类上下文文件。

原因是：

* 模型每次都会先读这些项目规则
* 你不写，后面它每次都得重新猜
* 你写了，后面它会稳定得多

这一点其实已经非常像 [[AI使用原则]] 和 [[企业落地判断]] 里的知识沉淀思路了。

## 九、这场有一条很值钱的线：前端和后端提示词不是一回事

这部分我建议直接看 [[前后端提示词差异]]。

老师这场讲得很清楚：

### 前端为什么相对更直接

* 视觉目标是固定的
* 页面长什么样，大体是确定的
* 你甚至可以把整张图、整段 HTML 都扔进去
* 重点是抽结构、抽组件、抽样式、抽资源

### 后端为什么复杂得多

* 同一个接口有很多合法实现
* 方案之间差异很大
* 你得跟 AI 讨论优劣，而不是只让它写

老师举的例子很典型：

* 某个支付限额接口
* 如果你设计成 `POST`
  * 可能要考虑二次提交
  * 要不要 token
  * 要不要 Redis
* 如果你设计成 `PUT`
  * 逻辑可能又是另一套

他要表达的是：

* 后端不是“能写出来就行”
* 而是“哪种写法更合理、更稳、更适合当前架构”

### 他说后端提示词至少要做这些事

* 先压缩上下文
* 让 AI 分析当前后端架构
* 让 AI 讲优点和缺点
* 让 AI 分析云端部署是否合理
* 再决定接口具体怎么写

### 他还提到自动化测试

老师专门说：

* 可以给编码工具挂 [[Playwright]] / 浏览器类 MCP
* 让它自己测接口、自己跑浏览器、自己生成测试文件

所以这场对后端的核心要求，不是“直接生成”，而是：

> 你得懂怎么和 AI 讨论架构，而不是只会让它吐代码。

## 十、组件库选型这段，不是闲聊，实际上很关键

这场有很大一段都在围绕“为什么这个项目里有这一层组件、那一层组件、到底要不要保留”。

老师真正想讲的是：

* 组件库是分层的
* 有的是底层无头组件
* 有的是在底层上包了一层可直接使用的组件
* 还有的是再往上包一层项目模板或生成器

他最关心的是：

* 这层封装到底有没有必要
* 你是不是只是多叠了一层复杂度
* 这个项目到底适合重组件体系，还是适合轻量方案

### 他对组件库的几个判断

* 有些团队爱用大型组件库
* 有些大型组件库体验很好，但更重、更贵，部分高级组件收费
* 有些底层组件更轻，但很多东西要自己补
* 对文本框、输入框、表格这类组件，中国团队和国外默认习惯也不一定一样

老师的态度是：

* 如果项目和团队已经选好了生态，就跟着生态来
* 但不要在自己不需要的地方叠太多层

这一段我单独沉淀成了 [[前端组件库选型]]。

## 十一、老师对当前这一套“暴力还原”是怎么评价的

老师很坦白：

* 他这次演示是偏“暴力”的
* 是为了让大家先看到结果、先把活推进
* 如果真要做得更稳、更漂亮
* 最好还是走标准玩法

他说的标准玩法大致就是：

* 接 MCP
* 或走更细的 Figma / 设计稿链路
* 或者用 AI 一步步抽离组件和样式

他的“暴力”玩法优势是：

* 快
* 够直接
* 结果能马上看见

缺点也很明确：

* 图片资源可能没拿全
* 有些背景和细节会偏
* 有些组件抽离不到位
* 还得手工调一轮

这段其实是在给大家划边界：

* 这个方法能拿来推进
* 但别把它误解成“任何场景都应该这么干”

## 十二、这场的 AWS 作业要求非常具体，而且很关键

这场最后讲的作业，不是“顺手补一个小练习”，而是老师明确说：

> 这个作业做不出来，后面的项目就干不了了。

### 作业主线

老师要求大家做一个非常简单但完整的东西：

* 用 [[Hono]] 开发一个接口
* 再做一个简单页面
* 页面上输入 `GitHub Token`
* 调接口去拿 GitHub 的用户信息
* 把拿回来的信息存进数据库

老师还提到：

* 至少要把最基础的增删链路跑通

### 部署要求

这部分更重要：

* 用 `AWS SAM` 部署到 [[AWS]]
* 把接口部署到 [[Lambda]]
* 数据库和 Lambda 放在同一个 `VPC`
* 子网和路由要自己配明白
* 最后再配一个 `GitHub Action`
* 以及最基本的 `IAM` 权限

### 他专门提醒的学习方式

老师这次特别强调：

* 这个作业先别用 MCP 一句话把子网全建完
* 你可以问 AI
* 但最好自己把 VPC、子网、安全组、Lambda 关联这条链手工搞明白

因为他的逻辑很明确：

* 如果现在让 MCP 直接替你做完
* 你看似快了
* 但后面一碰云服务细节就会彻底懵

所以这作业的真正目标是：

* 把前几节的 AWS 基础
* 和当前这个项目开发链路
* 真正串成一条线

## 十三、这场其实也在继续强化一个现实判断

老师这场好几次都在讲：

* 纯前端基础活越来越不够
* 你不往全栈、后端、云、自动化去补
* 很快就会被时代推着走

他不是在否定前端，而是在强调：

* 前端已经不再只靠“页面写得不错”就够了
* 你得懂组件、架构、接口、部署、测试、云

所以这场最底层还是那句老话：

> 不是 AI 让前端消失了，而是 AI 让“只会一小段前端”的价值迅速缩水了。

## 十四、课程安排的推进

老师最后把课程节奏又往后推了一步：

* 今天把这套前端实战和作业要求讲清楚
* 明天继续讲更复杂的后端 / 老系统一类话题
* 把 `SPA / MPA / MPA + SPA` 这条线补完
* 下周进入 [[Web3基础概念]] 和 Web3 项目
* 再往后接 serverless、自研服务和线下课

这意味着：

* 当前这场不是独立的一节
* 而是课程正式从“能用 AI”进入“拿 AI 做项目和交付”的中间过渡节点
