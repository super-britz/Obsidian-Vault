# 06-技术就业AWS讨论｜直播笔记整理

## 概览

这场直播不是单纯讲 AWS 控制台怎么点，而是在把课程继续往：

1. [[技术就业判断]]
2. [[AWS基础部署链路]]
3. [[Serverless架构]]
4. [[VPC 与子网]]
5. [[IGW 与 NAT Gateway]]
6. [[Cloudflare Tunnel]]
7. `AWS MCP + GitHub MCP + Cloudflare MCP`

这几条线一起推进。

老师最想强调的现实变化是：

* 纯前端越来越不够
* 部署、云、网络和 AI 工作流越来越接近日常基本功
* AI 会压平很多“纯操作型能力”
* 真正更值钱的是行业理解、交付能力、学习速度和协作能力

> [!note]
> 这份整理按直播主线重构，文字稿里有不少发音和转写错误，已经尽量按上下文校正。高风险口头建议和待核词汇统一收进了 [[风险与疑似错词]]。

## 一、老师开场先提醒了什么

老师一上来先讲的不是架构，而是 **AWS 费用风险**。

他的核心提醒包括：

* AWS 如果配置不当，费用可能很高
* 学习账户和长期使用账户最好分开
* 学习阶段先想办法控制扣费风险

老师还特意点了三个很多人会混的概念：

* `VPS`
* `VPC`
* `VPN`

他想表达的是：

> 如果连 `VPS / VPC / VPN` 都分不清，后面的云部署和网络概念很容易越学越乱。

## 二、从旧式前端部署到 serverless

老师用一种“旧时代 -> 新时代”的方式来讲这节课。

他提到的旧路线大致是：

* `Node.js`
* `GraphQL`
* `React`
* `webpack`
* `Docker`
* `Jenkins`

这对应的是以前典型的流程：

* 本地开发
* 打包
* 上 Docker
* 交给 Jenkins
* 部署到服务器

而他现在更想推的是 [[Serverless架构]]。

他顺带建立了一组“服务化”名词：

* `FaaS`
* `BaaS`
* `PaaS`
* `SaaS`
* `DaaS`

老师真正想表达的是：

> 现在很多数据库、函数、平台和后端能力，本质上都在变成“服务”，你不再需要像以前那样手里一直抱着整台机器。

## 三、老师怎么解释 serverless

老师对 serverless 的解释虽然很口语，但重点很清楚。

### 为什么它便宜

* 以前像是一人一台机器
* 现在更像多人共享底层资源
* 平台尽量把机器填满，提高利用率

### 为什么它适合很多业务

* 长尾业务适合
* 小流量项目适合
* 不需要整天维持一台空转机器

### 它的缺点是什么

* 不是持久化环境
* 不是独占环境
* 往往是无状态的
* 长时间没人访问会被回收

老师特别强调了一个词：

* `冷启动`

他明确说这会是后面面试里的重点词。

> [!important]
> 这场课里，`冷启动` 是老师明确点名的 serverless 面试关键词。

## 四、BFF：为什么以前前端和后端中间还会夹一个 Node 层

老师用 `BFF = Backend for Frontend` 来解释一段老思路。

他的意思是：

* 前端未必直接请求真正后端
* 常常会先请求一层 `Node`
* 由这层 `Node` 做：
  * 接口编排
  * 字段裁剪
  * 返回结构重组

老师举的典型例子是：

* 页面只想拿用户头像 URL
* 后端却返回一大坨 JSON
* 于是前端团队会在 Node 层把返回数据重新整理

他在这场课里的判断是：

* 以前很多这类活靠 BFF / Node 中转层扛
* 现在很多场景开始改成 serverless 去接这一层

也就是说：

> 以前很多 “前端 -> Node BFF -> 真后端” 的中间层逻辑，正在逐渐被 serverless 形式替代或稀释。

## 五、老师真正开始讲的 AWS 主线

这场课真正涉及到的 AWS 主线，可以压缩成一条：

**代码 -> `sam deploy` -> `CloudFormation` -> `IAM Role` -> `API Gateway` -> `Lambda` -> `VPC / 子网 / 安全组` -> 数据库 / 缓存 / 队列 / 存储**

这里边反复出现的核心产品包括：

* `AWS Lambda`
* `CloudFormation`
* `IAM Role`
* `API Gateway`
* `VPC`
* `Subnet`
* `Security Group`
* `Aurora / RDS`
* `S3`
* `ElastiCache`
* `SQS`
* `CloudWatch`
* `Secrets Manager`

如果把这节课压缩成一张部署脑图，最适合看的就是 [[AWS基础部署链路]]。

## 六、CloudFormation 和 Role：老师觉得最不能忽略的两层

老师这里反复提的“confirmation / formation / foundation”，高概率都是在说 `CloudFormation`。

他的意思是：

* 代码部署到 AWS 上，不是简单“扔到一台机子里”
* 平台会创建并绑定很多资源
* `CloudFormation` / stack 这一层就是在做资源编排

而 `Role` 则是另一条核心线：

* AWS 里很多资源访问和部署都是靠角色驱动
* `CI/CD` 想部署 Lambda，需要 `Role`
* 权限不通，很多流程就跑不起来

老师给的现实结论是：

> 你要是配不好 `role`，基本就玩不转 AWS。

## 七、VPC、子网、安全组：这场课里最重要的基础设施脑图

老师反复强调：

* 企业服务不会让核心代码和数据库直接裸露在外网
* `Node.js` 服务、数据库、内部资源通常都会放在 [[VPC 与子网]] 里

### `VPC`

* 一个更大的虚拟私有网络
* 用来装应用服务、数据库、内部资源

### `Subnet`

* VPC 里的分区网络
* 老师这场课里反复举的典型分法是：
  * `public subnet`
  * `private subnet`
  * `database subnet`

### `Security Group`

* 本质上就是防火墙
* 不只是“建了资源就能互通”
* `Node / DB / Redis` 等资源能不能通，往往取决于：
  * VPC
  * 子网
  * 安全组

老师很想让大家记住一句话：

> 云上“连不通”，很多时候不是代码问题，而是网络和权限模型没有配对。

## 八、API Gateway：外部怎么安全进入内部服务

老师对 `API Gateway` 的讲法非常明确：

* 它是独立产品
* 它不等于 VPC
* 它负责接外部请求
* 再去连接 VPC 内部的服务

也就是说：

* 外部用户不会直接敲进你的内部 Node / Lambda / 数据库
* 流量通常先过 `API Gateway`
* 再由它把请求转给内部资源

## 九、私有子网怎么出网：这场课最值得单独记的一段

这场 AWS 内容里，最关键的一段其实是：**私有子网里的服务，怎么“出网”**。

老师举的场景包括：

* 私有子网里的服务想访问公网
* 想发 webhook
* 想访问 GitHub 或其他外部服务
* 但又不想把内部服务直接暴露出去

这时就会进入 [[IGW 与 NAT Gateway]] 这条线。

老师真正反复讲的是三种方式：

* `Internet Gateway`
* `NAT Gateway`
* `Endpoint`

他最想让大家搞懂的是：

### `Internet Gateway`

* 用于双向公网访问
* 常见于 `public subnet`

### `NAT Gateway`

* 用于私网单向出网
* 私网服务能访问公网
* 公网不能反向主动访问它

### `Endpoint`

* 更像私有访问 AWS 服务
* 不一定非要走公网

> [!summary]
> 老师这段最值得记住的不是所有名词，而是这句：
>
> * `IGW`：双向公网访问
> * `NAT Gateway`：私网单向出网
> * `Endpoint`：私有访问 AWS 服务

## 十、Lambda 进入 VPC 后，复杂度为什么会一下上来

老师虽然没有把 Lambda 单独讲成一节，但主线已经说出来了：

* Lambda 如果只是简单 HTTP 处理，会比较轻
* 一旦要访问数据库、缓存、内部服务
* 就往往需要进入 VPC

但一旦进去，就会跟着带出更多配置：

* 绑定 VPC
* 绑定子网
* 绑定安全组
* 再考虑出网路径

所以老师这场课的底层逻辑其实是：

> 你不能只会“把代码部署上去”，还得理解代码在云网络里到底站在哪。

## 十一、S3、CloudFront、Aurora、ElastiCache、SQS：老师顺手帮大家做的区分

老师虽然没有展开每个产品的深度，但区分关系是清楚的。

### `Aurora / RDS`

* 关系型数据库

### `ElastiCache`

* 缓存
* 老师这里主要按 Redis 语境在说

### `SQS`

* 队列
* 用来做异步任务和排队

### `S3`

* 对象存储
* 放图片、文档、静态文件

老师后面又特意强调：

* `S3` 本身是存储
* 如果想更像 CDN，要在前面挂 `CloudFront`
* 这样才更像完整的静态资源分发链路

## 十二、Cloudflare 作业：这场课真正要交的东西

这场课落到作业时，老师的方向很明确：

1. 买一个自己的域名
2. 解析到 `Cloudflare`
3. 接上前面做过的 `EC2` 作业
4. 使用 [[Cloudflare Tunnel]] 把服务带出来

老师这里反复说的 `channel / turn / turnnel`，高概率都是在说：

* `Cloudflare Tunnel`

他强调的用途包括：

* 把本地 `127.0.0.1` 服务暴露给外部访问
* 把 EC2 上的服务通过 Cloudflare 接出来

也就是说，这节课的作业不是只学概念，而是：

> **域名 + Cloudflare + Tunnel + 现有服务入口**

## 十三、老师要求的作业提交物

老师这一场对交付物讲得很直白：

* `GitHub` 地址
* `IP / 域名 / 可访问入口`
* 视频讲解

他还建议：

* 最好准备一个自己的在线文档
* 每周作业都持续放进去

这说明他很在意的不是零散截图，而是：

* 连续作业记录
* 可访问结果
* 代码仓库
* 讲解表达

## 十四、MCP / AWS / GitHub：老师后半段在展示什么

老师后半段明显在演示这样一种工作方式：

* 用 AWS MCP 查 VPC
* 查子网配置
* 查 SQS 队列
* 让 AI 直接读和操作一部分云资源信息

他想传达的判断非常强：

* 以前很多 AWS 查配置的活要自己翻半天
* 现在很多事可以直接“打字问”
* 但你自己还是得懂概念

同时他还给了 GitHub 这边的典型接法：

* `Settings`
* `Developer settings`
* `Personal Access Token`
* 再交给 GitHub MCP

然后 AI 就能继续做：

* 查仓库
* 拉分支
* 提 PR
* 做部分自动化操作

## 十五、老师对就业的判断：AI 抽平操作能力后，还剩什么

老师这场关于就业的判断非常强烈。

他反复强调：

* 现在技术“纯操作型能力”被 AI 抽出来很多
* 单纯比写代码速度，已经很难构成长期优势
* 真正更值钱的是 [[技术就业判断]]

他给出的关键词包括：

* 垂直行业知识
* 学习能力
* 毅力
* 情商
* 主动成长

老师还明确提到：

* 纯前端不好找
* 全栈、部署、云、Web3、AI 工作流更值得补
* 国内岗位里学历筛选仍然普遍存在

> [!warning]
> 这里更适合记成“老师的现实判断”，不要记成绝对规律。  
> 尤其是关于学历、裁员比例和“谁都不牛”的表述，情绪和立场都很强。

## 十六、这场课里不适合直接照抄照做的部分

### 1. AWS 学习账户的支付建议

老师强调了“控制风险”这件事，这个方向对。  
但具体支付方式和账户处理上，更适合走 **正规、可控、可回收** 的方案。

### 2. GitHub Token 全权限、不过期

直播里是为了图快。  
正式执行时，更稳妥的是：

* 最小权限
* 按用途拆 token
* 设置有效期

### 3. “AI 都能做了，所以基础不用学了”

老师真正想说的是 AI 会加速很多事。  
但从这场课本身也能看出来：

* 不懂 `VPC`
* 不懂 `子网`
* 不懂 `Role`
* 不懂 `IGW / NAT`

AI 也很难帮你真正排清楚问题。

## 十七、这篇最值得保留的一段摘要

> 这场直播真正讲的，不只是 AWS，而是：
>
> * 为什么部署、云网络、serverless 正在成为前端 / 全栈求职的重要组成部分
> * 为什么 `CloudFormation / IAM Role / API Gateway / Lambda / VPC / Subnet / Security Group` 这条线值得尽快建立脑图
> * 为什么 `IGW / NAT Gateway / Endpoint` 是理解“私网怎么出网”的关键
> * 以及在 AI / MCP 时代，为什么“会操作”已经不够，更重要的是懂概念、懂行业、懂交付
