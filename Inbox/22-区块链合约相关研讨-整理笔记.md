# 22-区块链合约相关研讨

延伸导航：[[22-区块链合约相关研讨-知识地图]]

## 概览

这节课是一堂偏实操的 [[智能合约]] 入门课，主线很清楚：

- 先用 [[貔貅合约]] 讲风险识别。
- 再从一个最简单的“存钱/取钱”合约切入，讲 [[Solidity]] 基础、转账单位、取款逻辑和前端交互。
- 然后延伸到 [[MetaMask]]、[[本地链]]、[[Ganache]]、[[EVM]]、[[Remix]]、[[Truffle]]、[[Hardhat]]、[[ethers]]、[[The Graph]]。
- 最后把重点落到 [[数据上链]] 的几种实现路径和课后作业。

一句话总结：

> 这节课不是只教“怎么写合约”，而是把“怎么识别风险、怎么在本地跑起来、怎么连接钱包、怎么部署、怎么读链上数据、怎么做一份最小可运行作业”连成了一条线。

## 课程主线

### 1. 先建立风险意识：什么是 [[貔貅合约]]

时间段：`00:01 - 02:24`

- 老师用“存钱/取钱”的最基础合约举例，说这种合约如果设计成“只能进不能出”或“只有 owner 能提走钱”，就可能被包装成典型的 [[貔貅合约]]。
- 他把这类骗局类比成：
  - 先拉你进老师群、荐股群；
  - 先让你赚一点，建立信任；
  - 再让你把本金打进某个合约；
  - 表面承诺高收益，实质上可能只允许项目方提币。
- 这节课一上来反复强调：
  - 世界上没有白掉的馅饼；
  - 对方给你的“利润”，很多时候套的是你的本金；
  - 合约代码虽然短，但短不代表安全，短也足够骗人。

老师给出的第一条判断标准：

- 如果一个合约不能开源，基本可以默认风险很高。
- 至少要去 [[区块链浏览器]] 看它是不是开源合约。
- 要去浏览器里的 `Contract` 页面看代码，而不是只听项目方宣传。

老师举例：

- 以 `USDT` 之类稳定币为例，虽然是世界级资产，底层代码行数也未必很多。
- 钱最终就是被映射进一段代码逻辑里。

本段顺手布置了一个方向性的作业：

- 把你们自己的测试合约部署到测试链。
- 部署后做合约验证和开源。

### 2. 从“只有 owner 能提钱”改到“指定地址可接收”

时间段：`02:24 - 08:30`

- 老师继续沿用上午的例子：
  - 当前合约只能管理员提钱；
  - 这不合理，因为正常逻辑应该是“谁存的谁能取”。
- 他说更合理的版本应该往 `mapping` 方向走：
  - 记录每个地址存了多少；
  - 提款时做对应校验；
  - 不是你的钱就不能取。

随后他演示了一个过渡版思路：

- 先不直接写完整 `mapping` 账户体系；
- 先给合约加一个收款目标地址；
- 让合约支持“把钱转给某个地址”。

这里有几个关键点：

- 目标地址要写成 `address payable`。
- 老师提到了 `transfer`、`transferFrom` 等转账写法，但转写里有些地方不够清楚。
- 他强调：现在很多新写法更推荐底层 `call`，但课堂里先按基础逻辑理解即可。

关于单位转换，他反复提醒：

- `1 ETH = 10^18 wei`
- 人类输入的“1、60、6000”不能直接拿来当链上金额。
- 前端或后端必须做单位换算，比如：
  - 输入 `60`
  - 实际上传给链的是 `60 * 10^18`
- 这和现实里的“元角分”很像，链上有自己的最小单位。

老师顺手拿一个真实业务场景做类比：

- 他在做一个 [[跨境支付]] 相关平台顾问；
- 平台会设置每日操作限额；
- 可能通过海外卡或海外通道把钱合规汇回国内；
- 这里的重点不是业务细节本身，而是提醒大家：
  - 链上金额和业务展示金额往往是两套单位体系；
  - 前端表单不能直接把展示值原封不动送上链。

### 3. 钱包、多链与测试网：先把 [[MetaMask]] 玩明白

时间段：`07:00 - 13:45`

- 老师建议学习顺序：
  - 先把 [[MetaMask]] 这类 EVM 钱包玩明白；
  - 再去碰 [[Solana]]；
  - 不要一上来把所有链、所有钱包、所有概念混在一起。

他说现在钱包生态在互相抢入口：

- EVM 钱包开始支持更多链；
- 一些非 EVM 钱包也在补充 EVM 能力；
- 但从学习角度，前期最好先把主线分开。

他提到几个实践点：

- `MetaMask` 现在对 `Solana` 的支持比以前强，但对 `devnet` 的支持和默认展示仍然可能不直观。
- 如果默认网络列表里没有某条测试链/开发链，可以自己添加自定义 RPC。
- 添加自定义网络时要填：
  - RPC URL
  - Chain ID
  - 货币符号

他还提到：

- 做钱包接入时，很多平台/SDK 对 EVM 支持更成熟；
- 某些钱包其实只是“在 EVM 侧被支持”，不是在原生链生态里完全丝滑；
- 如果公司产品既要支持 EVM、又要支持波场、又要支持 Solana，就得认真看开源项目是怎么封装钱包连接层的。

这段里还穿插了一些 DEX/CEX 的行业常识：

- DEX 和 CEX 的一个大差别是是否强依赖实名/KYC；
- DEX 里很多场景是“连钱包就能做操作”。

### 4. 在浏览器 IDE/VM 中演示：为什么明明转出了，目标账户却没加钱

时间段：`13:45 - 24:31`

这部分的实操味很浓，核心是：

- 老师在浏览器 IDE 里重新部署了合约；
- 先存钱；
- 再从合约里向指定地址转钱；
- 中间遇到了一个很迷惑的环境问题。

几个关键观察：

- 编译器/IDE 会警告：
  - `transfer` 这类写法不是最推荐的；
  - 更建议用 `call`；
  - 原因之一是 `call` 更底层，也更灵活。
- 老师强调新版 UI 很容易误导人：
  - 存钱参数和取钱参数显示不够直观；
  - 容易把“金额参数”和“附带 value”搞混。

他在演示里专门区分了两类账户：

- [[外部账户]]（EOA）
  - 有私钥
  - 由用户控制
- [[合约账户]]
  - 没有私钥
  - 钱只能按代码逻辑移动

也就是说：

- 地址长得都像普通地址；
- 但合约账户本身不能像人一样直接签名；
- 合约里的钱要不要动、往哪动，全看代码。

课堂里的“坑点”非常值得记：

- 某个 VM/环境下：
  - 合约余额明明减少了；
  - 目标地址余额却没正确显示；
  - 老师一度怀疑环境或 UI 有问题。
- 后来切换了 VM/执行环境之后，转账显示恢复正常。

这一段的经验总结非常适合单独记成一条：

> 当链上结果看起来“违反常识”时，先不要立刻怀疑你的业务逻辑；先检查 VM、节点、浏览器状态、网络环境、编译器版本和 UI 显示层。

老师明确说：

- 课堂里这种情况更像环境问题，不像合约逻辑本身的问题。
- 学习阶段不要被工具链的偶发问题拖死。

### 5. 引出 [[本地链]]：为什么初学者要先学 [[Ganache]]

时间段：`24:31 - 30:24`

这部分是全课非常重要的一段，因为老师正式把大家从“在线演示环境”拉回到“可控的本地环境”。

核心观点：

- 初学合约和钱包交互时，本地链非常重要。
- 因为：
  - 不会突然断网；
  - 不依赖外部节点稳定性；
  - 交易、区块、日志都更容易观察；
  - 适合反复试错。

老师推荐的工具是 [[Ganache]]：

- 下载后本地会给你一组测试账户；
- 常见默认是 `10` 个账户；
- 每个账户里有很多测试 ETH；
- 很适合教学和面试演示。

导入钱包时，他专门提醒了两个容易混淆的动作：

- `添加账户`
- `导入账户`

他的意思是：

- 用私钥导入测试账户更适合这节课的场景；
- 助记词和私钥之间的关系要弄清楚；
- 同一组助记词可以派生出多个地址；
- 不要误以为“在钱包里再点一次创建账户”就等于恢复了原来那个地址体系。

本地网络配置要点：

- 自定义 RPC 通常填本地地址，例如：
  - `http://127.0.0.1:7545`
- `Chain ID` 常见填：
  - `1337`
- 货币符号：
  - `ETH`

他在口头里有过端口口误，但课堂主线很明显是：

- 以 [[Ganache]] GUI 常见的 `7545` 为主；
- 不要被一时口误带偏。

这部分还顺手讲了 [[区块]] 的基本概念：

- 区块链本质上就是“块形成链”。
- 块里最直观的内容之一就是交易记录。
- 每条链的出块时间不同。
- 出块时间不同，本质上和挖矿/共识算法有关。

老师还提到了 [[创世块]]：

- 你什么都没干时，系统里也会先有一个初始块；
- 这是整条链的起点。

他让大家重点观察本地链界面里的这些对象：

- 账户
- 私钥
- 交易
- 区块
- 日志
- gas

这段的学习建议可以总结成一句：

> 本地链不是“简化版玩具”，而是把黑盒过程可视化的第一层台阶。

### 6. 从 [[区块]]、[[日志]] 到“链上数据库”的理解

时间段：`29:28 - 30:24`

老师说有些人把区块链叫成“区块链数据库”，原因是：

- 很多人会拿链上来存日志；
- 日志本质上也是数据；
- 只是链上存储非常昂贵。

区块里可以看到的典型内容：

- from/to
- gas price
- 交易消耗
- 属于哪个 block
- 合约相关信息
- event/logs

老师反复强调一个现实判断：

- “能不能上链”和“值不值得上链”是两回事。
- 链上适合放最关键、最需要不可篡改留痕的内容。
- 不适合把大块普通业务数据、图片、视频原样往上怼。

### 7. [[Solidity]] 提升部分：重载、library、import、标准化思维

时间段：`30:24 - 38:13`

这部分开始进入语法和标准。

#### 7.1 函数重载

- 老师说 [[JavaScript]] 没有真正意义上的重载，更像覆盖/重写。
- 但 [[Solidity]] 里函数是可以重载的。
- 原因不是“语言魔法”，而是：
  - 编译后每个函数都对应不同的函数选择器；
  - 参数签名不同，最终选择器就不同；
  - 所以虽然源码里函数名一样，底层并不是同一个函数。

#### 7.2 library 与 `using ... for ...`

- 老师用 JS 的 `prototype` 做类比：
  - 可以把一个方法“挂”到某类对象上；
  - 于是这个类型的实例就都能调用该方法。
- 在 Solidity 里，`using A for B` 有点类似这种扩展思路。

#### 7.3 import

- `import` 的基本感受和前端开发很像；
- 只是导入的是合约、接口或库。

#### 7.4 [[ERC 标准]]

这部分是全课里很适合做概念卡片的内容。

老师的表达可以整理成：

- 以太坊生态里有大量标准；
- 常见的有：
  - [[ERC20]]
  - [[ERC721]]
  - [[ERC1155]]
  - [[ERC3525]]
- 标准不是凭空来的，而是有人提出、讨论、沉淀出来的。

他讲了一个“标准委员会”的理解方式：

- 先有人提出自己的金融/资产抽象；
- 先给出接口；
- 再经过长期讨论和接受；
- 最终才有可能成为广泛采用的标准。

关于 [[ERC20]]，老师的课堂重点是：

- 它是“同质化代币”的标准；
- 强调的是代币如何转、如何授权、如何交互；
- 它不是现实世界找零逻辑的完整映射。

他还专门区分了：

- 公链原生币
- 部署在 EVM 上的 ERC20 Token

这两者不是一回事。

### 8. 最重要的安全提醒之一：`approve` 不要乱点

时间段：`37:04 - 38:30`

这是老师整场课里风险提示最明确的一段之一。

核心意思：

- 你钱包里的代币如果要让某个合约代为操作，通常要先 `approve`。
- 一旦授权额度过大，甚至是“无限授权”，这个合约理论上就能把你的钱全花掉。

老师说的实操判断标准非常直接：

- 如果一个页面上某个操作实际只值 `1 U`；
- 弹窗却要你授权“无限额度”；
- 这就必须高度警惕。

要点整理：

- `approve` 不是“点一下没事”的按钮。
- `approve` 是在给合约开支配你代币的权限。
- 学会看授权对象、授权额度、授权资产。

老师还举了真实踩坑例子：

- 有同学明知有风险还想“试试”；
- 结果账户资金被清空。

这部分建议你后面单独拆成：

- `[[approve 风险]]`
- `[[无限授权]]`
- `[[钱包弹窗审查]]`

### 9. 合约调合约：从高级方式到低级 `call`

时间段：`38:42 - 42:53`

老师把“合约调合约”分成三种思路：

#### 9.1 直接 import 后 new

- 如果两个合约代码都在自己手里；
- 最简单就是 `import` 后直接使用。

#### 9.2 继承 + `is` + `super`

- 如果是继承关系；
- 可以通过 `is` 继承；
- 再通过 `super` 调父合约逻辑。

#### 9.3 通过合约地址 + 接口调用外部合约

- 这是更常见的真实场景；
- 因为你不可能永远和别人的合约代码放在一起；
- 所以通常是：
  - 拿到对方的合约地址；
  - 拿到对方公开的 interface 或 ABI；
  - 再进行接口化调用。

老师把它类比成一种“强制类型转换”的感觉：

- 地址本身只是地址；
- 通过接口视角去看它，才知道有哪些方法可以调。

#### 9.4 低级 `call`

老师虽然不建议一上来就死啃它，但还是讲了原因：

- `call` 更底层；
- 更灵活；
- 很多时候也更省 gas；
- 所以现在很多地方都会推荐使用。

要点：

- `call` 可以转钱；
- 也可以调方法；
- 若需要转钱，就用 `{value: ...}`；
- 若只是调函数，可以只编码函数和参数。

#### 9.5 `staticcall`

- 这是“只读调用”；
- 只能读，不能改状态；
- 更像静态查询。

### 10. `msg.sender` 为什么会变

时间段：`41:59 - 42:53`

老师讲的是一个非常基础但特别容易在业务里踩坑的点：

- 你让账户 A 调合约 A；
- 合约 A 再去调合约 B；
- 那么在合约 B 里看到的 `msg.sender`，通常已经不是最初那个外部用户，而是“上一个调用者”，也就是合约 A。

这意味着：

- 你在多层调用链里做权限判断时，不能想当然地认为 `msg.sender` 一直是最终用户；
- 很多业务主体要不要“固定身份”，要事先设计好。

### 11. `receive` / `fallback`、附言、16 进制数据与 [[数据上链]]

时间段：`42:53 - 46:07`

这是下半场最关键的实践主题。

老师在讲“收钱时的回调”时，顺带把“最简单的数据上链方式”也讲出来了。

他的逻辑可以整理成：

- 如果 `msg.data` 为空，并且合约实现了 `receive()`：
  - 那么收到钱时会走 `receive()`
- 如果 `msg.data` 不为空：
  - 通常会落到 `fallback()` 这样的逻辑里

而这跟“数据上链”的关系在于：

- 你在转账时可以附带数据；
- 钱包端往往只接受 16 进制数据；
- 这些数据会跟随交易一起上链；
- 链上任何人都能看到。

老师讲了两种主要上链方式：

#### 11.1 通过转账附言/附加数据上链

- 例如在钱包里转 ETH 时附加十六进制数据；
- 然后通过交易详情把这段数据读出来。

#### 11.2 通过合约 `event log` 上链

- 在合约里 `emit` 一个事件；
- 事件参数进入日志；
- 前端或索引服务再去读日志。

他强调：

- 链上存数据很贵；
- 容量受限；
- 所以上链通常只上最重要的摘要、标识或关键记录。

老师还提到现实应用想象：

- 政府数据上链
- 犯罪记录上链
- 婚姻记录上链

他的观点非常朴素：

- 从技术动作上说，并不神秘；
- 就是把关键数据写进交易或日志；
- 难的不是“能不能写”，而是“业务该写什么、写多少、怎么读回来”。

他甚至举了一个偏社会事件的例子：

- 有人把某些长期需要保留的证据材料写到了链上；
- 目的就是让记录难以消失。

这个例子虽然偏情绪化，但折射出的技术点是：

- 链上不可篡改和长期可检索，是现实里很多人愿意为其付费的原因。

---

### 12. 合约不能像普通程序那样随便删

时间段：`46:22 - 47:08`

老师很明确地说：

- 部署后的合约不是你想删就删。
- 原来大家会提到某些“销毁/自毁”语法；
- 但现在相关机制已经越来越不适合作为常规设计手段。

课堂里更重要的结论是：

- 当你升级合约时，很多时候不是“把老合约删了”；
- 而是：
  - 部署新合约；
  - 把交互入口切过去；
  - 必要时处理数据迁移和兼容。

这也是为什么：

- 合约地址变更是大事；
- 升级方案一定要慎重设计。

### 13. `try/catch`、[[Assembly]] 与日志索引

时间段：`47:12 - 48:51`

老师带过了几块提升点：

#### 13.1 `try/catch`

- Solidity 有 `try/catch`
- 但写法和 JS 不完全一样
- 课堂里强调的是：
  - 它没有你熟悉的那套完整前端语境
  - 要适应合约语言自己的异常处理方式

#### 13.2 内联汇编 / Assembly

- Solidity 也能下沉到底层汇编风格
- 主要价值在于：
  - 极致性能
  - 极致 gas 优化
- 但这不是入门期重点

#### 13.3 日志 `indexed`

这是一个很实用的点：

- 事件参数如果不做 `indexed`
  - 读出来可能是一大坨数据，前端自己筛
- 如果做了 `indexed`
  - 某些字段更容易被检索和过滤

这其实就是在给后面的前端索引、日志读取做准备。

时间段：`49:08 - 52:38`

老师在命令行里演示时提到：

- `Truffle` 已经不是最前沿的主流生产工具；
- 官方也在提示迁移；
- 但对教学来说，它仍然有价值：

他提到的常用命令可以概括为：

- `truffle init`
- 编译
- 部署

项目结构的重点目录：

- `contracts`
- `migrations`
- `test`

课堂里还提到一个小坑：

- VS Code 里 Solidity 扩展可能有多个版本或多个提供方；
- 用官方的就够了；

### 15. 课堂示例合约：`InfoContract`

时间段：`52:12 - 55:18`

老师现场让大家做了一个非常小的示例合约，核心结构大概是：

- 一个字符串字段
- 一个年龄字段
- 一个 `setInfo` 或类似方法，用来设置字段
- 一个 `sayHi` 或读取方法
- 一个事件 `emit`，在设置成功后打日志

这类合约的教学价值非常高，因为它同时覆盖了：

- 状态变量
- 写操作
- 读操作
- 事件
- 部署
- 前端调用

部署脚本的课堂思路是：

- 在 `migrations` 里写 JS
- `require` 这个合约
- 在部署函数里 `deployer.deploy(InfoContract)`

老师非常强调 [[ABI]]：

- 编译后会生成 JSON
- 里面包含：
  - 合约方法
  - 参数
  - 事件
  - 网络部署信息
- 前端如果没有 ABI，基本没法正常调合约

课堂可直接记成一句：

> 前端调合约至少要有两样东西：合约地址 + ABI。

### 16. 前端怎么调合约：`provider + signer + contract`

时间段：`58:38 - 01:02:06`

这段已经接近一个完整的最小 DApp 流程。

老师做的是一个原生页面，大意是：

- 用一个简单页面去连接钱包；
- 用 [[ethers]] 去实例化合约；
- 调读方法；
- 发起写方法；
- 等待交易上链；
- 再读取结果。

课堂里的最小流程：

1. 检查是否安装 `MetaMask`
2. 拉起钱包连接
3. 拿到 `provider`
4. 拿到 `signer`
5. 获取当前地址和网络
6. 用 `new ethers.Contract(address, abi, signer)` 实例化合约
7. 读方法直接调
8. 写方法需要签名、花 gas、等待确认

老师还提醒了两个实操坑：

- 不能直接把 HTML 当本地文件双击打开就指望钱包正常连；
- 应该起一个本地 server 再访问页面。

另一个很现实的问题是：

- 如果浏览器里同时装了多个钱包扩展；
- 它们都可能争抢注入对象；
- 最终你得明确这次到底连哪个钱包。

在本地链场景里，完整操作链路是：

- 先把本地网络切到 `localhost` / `1337`
- 然后钱包里就会出现本地测试余额
- 再发起 `setInfo("老袁", 20)` 之类的交易
- 钱包弹窗确认 gas
- 确认后交易上链
- 事件日志出现
- 再读回最新状态

老师顺手强调了一件非常现实的事：

- 真正工作里，纯技术当然重要；
- 但更值钱的是你能不能快速理解业务。

也就是说：

- “会写”只是门票；
- “理解产品和链上业务怎么跑”才更值钱。

### 17. 本节课最终作业：做一个“完整的数据上链平台”

时间段：`01:02:41 - 01:07:26`

这是全课最需要落地的一部分，我按“必做/选做”重写如下。

#### 必做 1：十六进制数据上链工具

要做一个小页面，至少包含：

- 文本输入
- 16 进制编码
- 16 进制解码
- 把待上链数据转成十六进制

老师的要求本质上是：

- 让你意识到钱包附言/交易数据通常是十六进制；
- 先把“数据怎么变成能上链的格式”搞明白。

#### 必做 2：通过转账附言做一次数据上链

流程：

- 在测试链上准备测试币和 gas
- 用钱包发起一笔转账
- 在交易里附加十六进制数据
- 交易上链后，用 [[ethers]] 或第三方 API 把这段数据读回来

老师的点评是：

- `ethers` 也能读
- 但三方 API 往往更快

#### 必做 3：通过合约日志做一次数据上链

流程：

- 写一个最简单的合约
- 合约里只做一件事：
  - `emit` 一个事件日志
- 把合约部署到测试链
- 前端调用这个合约，把数据写进日志
- 再通过索引服务把日志读出来

老师这里点名了 [[The Graph]]：

- 它本质上是一个“把链上日志和数据更容易取回来”的索引层工具。

#### 选做：通过代币转账附加数据

老师还给了一个进阶思路：

- 先在测试链上把 `ETH` 换成一点 `USDT/USDC`
- 再做一次代币转账
- 看是否能带上附加数据或做相关研究

这块被他明确说成选修/研究项，不是主作业。

#### 强制要求

- 只要部署到测试链，合约就必须验证并开源。
- 这是“上道”的最低标准。

#### 学习顺序

- 先做 EVM
- 有精力再把同类流程迁到 [[Solana]]

## 我帮你提炼出的高频结论

- [[貔貅合约]] 的第一判断标准之一是：是否开源、提款逻辑是否只对 owner 开放。
- 链上金额一定要注意单位换算，`ETH` 和 `wei` 不能混。
- 学合约时，本地链是第一站，别急着直接冲测试网。
- [[Ganache]] + [[MetaMask]] 能帮你把账户、区块、交易、gas、日志全看明白。
- [[Solidity]] 的重载、library、事件、接口、ABI 都要结合“最终怎么被前端调用”来理解。
- `approve` 是安全高危区，尤其要警惕无限授权。
- 合约调合约要理解 `interface`、`call`、`staticcall`、`msg.sender` 的变化。
- [[数据上链]] 最实用的两条起步路径：
  - 交易附带十六进制数据
  - 合约 `emit event`
- 链上数据默认公开且昂贵，不是什么都值得上链。
- 前端调合约的最小心智模型就是：
  - `provider`
  - `signer`
  - `address`
  - `ABI`
