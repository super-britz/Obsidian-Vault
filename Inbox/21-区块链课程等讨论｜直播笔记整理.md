# 21-区块链课程等讨论｜直播笔记整理

## 概览

这场直播虽然标题是“区块链课程等讨论”，但真正的主线其实很清楚，基本分成三段：

1. 课程安排和老师最近在做的 AI / Web3 / 日本方向项目
2. 老师对 `CEX / DEX / 合约岗 / 前端岗` 的现实判断
3. 正式进入 [[Solidity 与合约基础]]，用 `Remix` 带大家看最小合约、`gas`、`payable`、`memory / storage` 和一个小银行示例

这场最有价值的地方，不只是“开始写 Solidity”，而是老师把这门语言在整个 Web3 工作流里的位置讲得很现实：

* 前端不一定转合约
* 但前端必须能看懂合约
* 真正难的不是语法，而是安全、手续费和通证经济

> [!note]
> 这份整理依据 [`21-区块链课程等讨论.txt`](/Users/liangsheng/Downloads/21-区块链课程等讨论.txt) 重构。转写里有大量口音词，尤其集中在 `Solidity`、`Hardhat`、`Remix`、`require`、`storage`、`payable` 一类术语上。我把高概率错词统一收进 [[风险与疑似错词]]。

## 核心结论

* 想进中心化交易所，底子仍然是工程化、性能、AI 使用能力和内推
* 想做去中心化方向，必须把 `公链 + 钱包 + 合约 + 合约交互` 这条链打通
* `Solidity` 语法本身不难，难的是安全、`gas` 成本和金融模型
* 合约代码量未必大，但它直接决定资金怎么流、规则怎么跑
* 短期内很难直接从零切到合约岗，更现实的路径是先进入 Web3 公司再内部转岗

## 一、课程安排与开场信息

老师先交代了最近的课程安排：

* 清明当天上午照常上课
* 会请一位在交易所工作的老师来讲大约 `4` 节 `Solidity` 课
* 这位老师原来也是从课程体系里成长起来的

他借这个例子强调的其实是：

* 技术岗还是要敢冲
* 这几年如果方向对、动作快，成长速度会非常明显

## 二、开场夹带的 AI / 工具 / 副业信息

这一段虽然不属于区块链正文，但信息量很大：

### 1. Claude / AI 工具使用体验

老师提到最近 `Claude Code` 一类工具的几个变化：

* 权限模式比早期更细
* 对话框交互体验更顺
* 一些 `code review` 可以桥接来做

这段更像是在给大家传递一个信号：

> 以后 Web3、前端、AI 开发不是分开的，工具能力会成为通用底座。

### 2. 垂直 AI 已经能走到硬件设计

老师现场举了一个很夸张的例子：

* 输入一个“自动给花浇水的小机器人”
* 工具能给出硬件清单
* 能给出电路图
* 还能给出 3D 布局

他想说明的是：

* 现在 AI 的价值越来越依赖垂直领域知识
* 不是只有写网页和写代码这一条路

### 3. 老师自己现在的多线收入结构

老师讲了自己目前同时在做的几条线：

* 正常授课
* 跨境支付相关项目
* `DEX` 架构项目
* 企业培训
* 日本机房和中转站
* 日本方向保就业班
* 一些 AI / App 创业尝试

这一大段的核心不是“炫”，而是在讲：

* 现在不要只给自己留一条路
* 代码能力之外，最好逐渐形成第二、第三条收入线

## 三、老师对 Web3 岗位的现实判断

### 1. 想进中心化交易所

老师的判断非常直接：

* 交易所岗位仍然值得去
* 但要靠工程化、性能、AI、项目经验这些基本功
* 这个圈子非常吃内推

也就是说：

* 不是说“会一点链上词汇”就能进
* 还是得把工程实力打牢

### 2. 想做去中心化方向

老师把这条线概括成：

* 公链原理
* 钱包
* 钱包封装库
* 钱包与合约交互
* 合约本身

这基本和 [[20-Web3知识及作业安排｜直播笔记整理]] 讲的路线是接得上的。

### 3. 想直接找合约岗

老师给的结论很现实：

* 短期学一点 Solidity，直接去找合约工作，基本不现实
* 更现实的路径是先进入一家纯 Web3 公司
* 再通过业务熟悉和信任积累申请转岗

原因是：

* 合约岗太独立
* 安全要求高
* 很多公司不会轻易把它交给新人

## 四、老师为什么反复强调“前端也要懂合约”

老师的立场不是“人人都要转合约”，而是：

* 你可以不写很多合约
* 但你不能完全看不懂

他强调的现实原因有两个：

### 1. 跟合约同学打交道不能完全露怯

如果一点不懂：

* 讨论业务逻辑时接不上
* 调合约时听不懂
* 很容易被当成“只会接页面的人”

### 2. 合约岗在公司里地位很高

老师反复讲了一个感受：

* 合约代码未必多
* 但它控制规则和钱
* 所以在很多公司里就是“心头肉”

## 五、区块链最小组成再次被强调

这场直播再次把最小概念图拎了一遍：

* [[公链与代币]]
* 矿工 / 节点
* 区块链浏览器
* 钱包
* 合约

并且继续用两条主线举例：

* 以太坊：原生代币是 `ETH`
* Solana：原生代币是 `SOL`

## 六、老师对 Solidity 的判断：语法简单，业务不简单

老师的结论非常明确：

* `Solidity` 是大家接触到的最简单的一批编程语言之一
* 语法和 `TS / Java / C-like` 语言很像
* 真正难点不在语法，而在：
  * 安全
  * `gas`
  * 通证经济设计

他说得很直白：

* 很复杂的合约代码也就两三百行
* 但就是这几百行会决定钱怎么流
* 这比“代码量大”重要得多

## 七、为什么合约难点在金融模型

老师这场把“通证经济”这件事讲得更具体了。

他的意思是：

* 代码少，不代表业务简单
* 合约设计的是整套系统怎么运转
* 代币怎么产生、怎么流通、怎么奖励、怎么回收，这些都在合约规则里

他举了一个“Web3 大学”的例子：

* 用户先拿真实资产换成链上资产
* 再换成平台自己的 token
* 之后用 token 买课、卖课、拿奖励

这里就引出了一个典型概念：

* `learn to earn`

也就是：

* 学习本身能变成获得 token 的行为

所以老师才说：

> 简单项目的通证模型你可以慢慢摸，但复杂金融模型不是只会写代码就能做。

## 八、Remix：这场课的 Solidity 入门环境

老师没有先带大家本地配环境，而是先走 `Remix`：

* 在线 IDE
* 自带编译、部署、测试、插件区
* 对新手最友好

他讲的最小流程是：

1. 新建 `.sol` 文件
2. 写 `pragma`
3. 写 `contract`
4. 编译
5. 部署
6. 拿到交易哈希和合约地址

这和普通前端理解“部署一个接口”最大的不同是：

* 合约地址本身就像新的调用入口
* 业务逻辑真正写到链上去了

## 九、最小 Hello 合约：老师讲的几个关键点

老师带大家写了一个最基础的字符串合约，大意是：

* 顶部先写 `pragma`
* 定义 `contract`
* 在里面声明一个字符串状态
* 用 `constructor` 初始化

这里他特别强调了几个概念：

### 1. 状态变量不是普通变量

在老师的说法里，它更像：

* 世界状态树里的一个状态

核心差别在于：

* 这个值不是只存在你电脑里
* 而是要进链上状态
* 改它要花钱

### 2. 读和写的成本不同

老师的说法可以压缩成一句：

* 读通常不花钱
* 写状态要花 `gas`

### 3. 构造函数也会消耗 gas

构造函数只执行一次，但一样要花钱。

## 十、gas 是怎么理解的

老师给了一个非常形象的类比：

* `gas price` = 油价
* `gas limit` = 你最多打算烧多少油

最终：

* `gas` 成本 = 单价 × 用量

同时他强调了一个很关键的现实问题：

* 如果 gas 不够，执行可能失败
* 失败不代表你一分钱都不花
* 高峰期手续费可能会很夸张

## 十一、编译版本和 breaking changes

老师一直提醒大家：

* Web3 生态的版本变化非常激进
* `Solidity` 不同编译器版本报错会不同
* 你让 AI 帮你修的时候，一定要把具体编译器版本一起给它

这是这场里非常值得记下的一句经验：

> 在链上环境里，很多错误不是“代码不会写”，而是“版本不对”。 

## 十二、部署后的理解方式：哈希、地址、链上浏览器

部署完成后，老师重点让大家记三样东西：

* 交易哈希
* 合约地址
* 测试链浏览器

他的意思是：

* 哈希能查这次部署
* 地址能作为后续调用入口
* 浏览器能看到链上执行结果

如果换成前端能听懂的话，可以近似理解成：

* 合约地址很像你以后要请求的“后端地址”

当然它不是 HTTP URL，但老师就是想让大家先把“调用入口”这个概念建立起来。

## 十三、Solidity 里这节课点到的高频语法

老师这场其实已经把最常用的一批语法扫了一遍：

### 数据类型

* `string`
* `bool`
* 有符号 / 无符号整数
* `bytes`
* `address`

其中最值得记的是：

* `address` 是 Web2 里通常没有的类型

### 常见修饰符

* `public`
* `private`
* `internal`
* `external`
* `view`
* `pure`
* `constant`
* `payable`

老师对 `payable` 的解释最直白：

* 只要涉及转钱、收钱，这个函数就得带它

### 全局对象

* `msg`
* `block`

尤其要记：

* `msg.sender`
* `msg.value`

### 常见结构

* 数组
* `mapping`
* `struct`
* `interface`

## 十四、memory 和 storage：这场最该听懂的概念之一

老师把这两者类比成：

* `memory` 更像按值传递
* `storage` 更像按引用传递

他想让大家记住的是：

* `memory` 内的改动很多时候只在当前函数生效
* `storage` 会真的改到链上状态

这也是为什么这部分必须理解：

* 一旦你改的是链上状态，就不再是普通变量问题，而是成本和数据持久化问题

## 十五、require / assert / 自定义 modifier

老师这场也把错误控制扫了一遍：

* `require`
* `assert`
* 自定义 `modifier`

老师的解释方向是：

* `require` 更像对外部输入、余额、权限等做判断
* `assert` 更像内部逻辑不该出错的地方

同时演示了自定义修饰符，比如：

* `onlyAdmin`

意思就是：

* 把权限判断提出来复用

## 十六、Hardhat、console.log 和审计公司

老师顺着调试讲到了两件事：

### 1. `Hardhat`

他提到：

* 一些调试能力可以借助 `Hardhat` 一类工具
* `console.log` 也是通过相应工具链来补的

### 2. 审计公司地位很高

老师借审计公司展开了一层行业现实：

* 合约审计在 Web3 是高价值岗位
* 审计报告会影响项目能不能上线、能不能买保险
* 所以合约安全不是“写完就行”

## 十七、小银行例子：充值、提现、余额

课上最后带了一个最小银行示例：

* 记录 `owner`
* 存钱函数使用 `payable`
* 通过 `msg.value` 收钱
* 提现前先判断余额

老师顺便指出了一个非常有 Web3 味道的问题：

* 如果只允许 `owner` 提现，那就可能变成“貔貅盘”

这句话其实很重要，因为它说明：

* 合约不只是语法
* 它和业务可信度、项目模式、用户风险直接相关

## 十八、单位与测试环境的细节提醒

老师最后还提醒了几个细节：

* `ETH` 和 `wei` 要分清
* 小数单位一旦看错，就可能以为自己取错钱
* `Remix` 自带测试账户和测试余额
* 演示环境里的数值和真实链上成本感受不完全一样

## 十九、这场课真正想让前端记住什么

压缩成一句话就是：

> 我们不一定现在就转合约，但必须把“合约怎么写、怎么编译、怎么部署、怎么收费、怎么和前端接起来”这一整套最小链路看明白。

所以这场课并不是要把大家培养成合约工程师，而是：

* 先别怕合约
* 能看懂
* 能对接
* 能和合约同学正常沟通
