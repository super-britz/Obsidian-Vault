# 11-前端及业务等问题讨论｜直播笔记整理

## 概览

这一场内容被分成了两段：

* 前半段是课程内的前端架构与 AWS 部署讨论，穿插了 MCP、Amazon Q、页面还原和工具链的现场演示
* 后半段是偏业务讨论和对接细节，涉及互助内容平台、会议表达策略、以及接口对接与环境配置

## 核心结论

* `SPA / MPA / MPA+SPA` 不是概念题，而是直接决定“路由由谁接管”和部署时的 `fallback` 规则
* 长时间任务不能放在 `Lambda` 里，要把状态落在 `S3 / Redis / 数据库`
* `IoC / DI / AOP` 的价值是“把依赖交给容器”，减少耦合与手动装配
* MCP 的使用门槛不在“是否能用”，而在“装在哪、如何让工具识别到”
* 页面还原能做到 80% 就很接近交付，但必须补主题、资源与真实路由

> 这场转写里出现了不少“工具名、框架名”漂移，我把不确定的统一挂在了 [[风险与疑似错词]]。

## 一、课程节奏与作业语境

老师先交代了本周节奏：

* 本周仍是前端架构主线
* 下周会进入 Web3 相关内容

## 二、MPA / SPA / BFF / core 的划分

老师把一个服务端项目拆成三类用途：

* `MPA`：真实路由，负责渲染页面
* `BFF`：专门面向前端的接口层
* `core`：偏定时、脚本、批处理的后端能力

对应的结构建议是：

* 后端按模块分目录
* TypeScript 最终编译成 JS
* 如果有前后端同构需求，可以用 `webpack` 来统一构建

相关主题可回链：

* [[SPA 与 MPA]]
* [[前端架构设计指南]]

## 三、长任务与状态管理

老师用“超长任务”为例强调：

* `Lambda` 是无状态、短时执行
* 长任务要交给 `ECS` 或更稳定的常驻进程
* 任务状态必须落到 `S3 / Redis / 数据库`

他的意思是：

* 不是所有任务都适合 `Serverless`
* 状态和生命周期必须“外置”

## 四、AWS 部署相关细节

### 1. S3 + CloudFront 是最常见的面试组合

老师强调面试高频点：

* `S3` 负责静态托管
* `CloudFront` 提供 CDN
* `CloudFormation` 是另一件完全不同的事

### 2. Lambda Layer 与打包流程

打包思路：

* 把依赖抽到 `Layer`
* `layer/nodejs` 下放 `package.json` 和 `node_modules`
* 应用包保持轻量

### 3. SAM / API Gateway / 自定义域名

他现场的流程大致是：

* `SAM` 构建 + 部署
* `template` 中定义 API Gateway、VPC、资源
* 需要上传能力就补 API Gateway 配置
* 自定义域名绑定后用 `CNAME` 指向 Cloudflare

## 五、IoC / DI / AOP 的直观解释

老师用了“商店送糖”的比喻来解释 `IoC`：

* 传统方式是你去找服务
* `IoC` 是服务找你
* 容器 (`container`) 里提前装好服务
* 你只需要在构造函数里声明依赖

`DI` 被解释为：

* 构造注入是最常见方式
* 其他注入方式只是形式不同

`AOP` 被解释为：

* 在业务逻辑外层做切面
* 不破坏原有类的结构

这段逻辑背后的目的是：

* 让路由层和 service 层“无直接引用”
* 通过容器统一加载和注入

## 六、前端部署与路由 fallback

老师强调了“真假路由”的现实问题：

* `Nginx` 下经常遇到 `404`
* 需要 `history fallback` 把路由指回 `index.html`
* 部署到 `S3` 时也要配置静态网站托管的 `index.html` 和错误页回跳

这与 [[SPA 与 MPA]] 的“路由由谁接管”是同一条线。

## 七、Amazon Q 与 MCP 的现场演示

老师现场展示了 `Amazon Q`，强调它的功能点：

* 帮助理解 AWS 架构
* 在终端或 IDE 内提供对话式助手
* 能给出配置和最佳实践建议

同时也演示了 MCP 的安装与使用逻辑：

* 有些 MCP 是“全局”安装
* 项目内如果找不到，需要告知工具去全局查
* 有时需要重新加载窗口或更新设置文件

相关主题可回链：

* [[MCP工具栈]]

## 八、页面还原与日常修复的细节

这一段几乎是“还原工作流实战”：

* 先修主题和背景色，再修组件
* 图片资源用 `PNG` 更稳，复杂图案不推荐 `SVG`
* `MCP` 可以还原设计稿，但也要人工修细节
* 不要给工具“完全权限”，避免错误回滚
* 有 `Git` 才能稳定撤销，否则容易还原到意外版本

老师也给了一个实际的修复清单：

* 先修报错
* 再补主题切换
* 最后校对输入框、按钮、图像资源

## 九、业务讨论片段（后半段）

这段更偏业务策略讨论：

* 互助板块需要引入 C 端用户、资深玩家、KOL/KOC 和门店人员
* 不建议回避风险话题，因为社媒会放大问题
* 会议表达上尽量放在附件，不展开细节

同时夹杂了一段对接讨论：

* 部署后配置内网域名和 `context path`
* `API auth` 前缀需要和服务端一致
* SDK 接完先本地跑通 `TOKEN` 校验
* 老项目要规划新前缀，避免影响旧接口
* 未接成功时先验证“是否成功接入”，再谈业务功能
