# 全栈课程路线

这页记录的是老师这场直播里对“为什么现在必须补全栈”的总体判断。

## 老师的核心判断

* 纯前端的很多基础活正在被 AI 抽走
* 对大多数普通背景开发者来说，全栈正在从“加分项”变成“必要项”
* 以后真正值钱的，不只是页面实现，而是整条交付链路的理解能力

## 在这门课里，全栈意味着什么

不是单纯“多会一门后端语言”，而是开始理解这些东西：

* 前端页面和交互
* 接口层
* GraphQL / tRPC / gRPC 一类协议层
* 云服务
* Serverless
* 数据、缓存、消息队列
* 自动化、MCP、浏览器控制

## 为什么老师现在强调它

他的逻辑大致是：

* AI 已经把很多基础页面活抽出来了
* 云服务和自动化开始变成日常工具
* 远程工作和海外语境里，双云架构越来越常见

## 这场课里更明确的两条原则

### 先跑人工链路，再逐步自动化

老师这场讲得很直白：

* 先把设计、部署、接口、测试这条链亲手跑通
* 再把重复劳动交给多 agents 和自动化
* 不要一上来就把希望都压在“全自动开发”上

### 不要把中间工具当主角

他的判断一直很一致：

* 某些电脑控制工具、拼装方案只是阶段产物
* 真正要学的是工作流和调度思维
* 官方工具一旦把能力内建，中间层很快就会被吃掉

## 这场课里“全栈”具体意味着什么

从这场课的口径看，全栈不是简单“会点后端”，而是至少开始理解这些东西：

* 设计稿如何进入开发链路
* MCP、skills、ACP 这类工具层怎么分工
* Cloudflare 和 AWS 分别负责什么
* Serverless 为什么替代“买机器 + SSH + 手动布服务”
* EC2 为什么会被当成稳定出口的一部分
* 为什么美国区网络和环境条件会直接影响工具可用性
* 自动化和多 agents 最后怎么服务交付

## 这页最重要的一句话

全栈在这里不是“多学点后端”，而是从“会写代码”升级到“会交付系统”。
