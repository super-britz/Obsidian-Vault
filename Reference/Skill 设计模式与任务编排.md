# Skill 设计模式与任务编排

这页主要从 [[13-AIGC第二季提效讨论｜直播笔记整理]] 拆出来，记录“企业级 AI 自动化”里最值钱的那部分方法论。

## 一句话理解

不是 skill 越多越强，而是：

* skill 怎么分角色
* 怎么串流程
* 怎么 review
* 怎么把结果接进任务编排和验收

## 老师强调的最小链路

企业级自动化最小可以理解成：

1. 拉需求
2. 出设计
3. review
4. 拆 task
5. 写代码
6. 跑测试
7. 验收和上线

## 这场里点到的五类设计模式

### 1. 工具包装器

作用是：

* 把某个库、平台、接口包成 skill
* 让 agent 像“会这个工具的专家”一样工作

### 2. 生成器

作用是：

* 根据模板、规则、结构生成稳定产物
* 例如需求文档、代码骨架、配置文件

### 3. 审核员

作用是：

* 不负责直接生产
* 负责按清单打分、挑错、补边界

这本质上就是“AI review AI”。

### 4. 先提问再行动的角色

意思是：

* 不先直接执行
* 先澄清边界、权限、上下文、特殊约束

复杂系统特别需要这一层。

### 5. 流水线

这是最关键的一层：

* 不是一个 skill 单打独斗
* 而是前一个产物进入下一个步骤
* 最后形成企业级工作流

## 老师自己的实践偏好

老师更看重的是：

* Token 消耗要可控
* 流程不要为了“高级感”无限套娃
* 编排结果要和业务产出成正比

所以他更偏向：

* 需求
* design
* review
* task
* 自动化测试

这条更短、更稳的链路。

## 最值得保留的判断

* 单模型直出复杂系统风险很高
* AI review AI 是第二季的核心
* 人最终要做的是验收和兜底
