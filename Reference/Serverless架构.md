# Serverless架构

老师这场课非常偏向 Serverless 和弹性云路线。

## 老师为什么偏向它

主要有几个原因：

* 更便宜
* 更适合创业公司和中小团队
* 更适合和现在的 AI 工作流结合
* 不一定还要维持传统“买机器 + SSH + SCP”的思路

## 老师在课里举的典型组件

### Cloudflare 侧

* Workers
* Pages

### AWS 侧

* Lambda
* AppSync
* Event Scheduler

## 它和传统部署的差别

传统方式更像：

* 买机器
* SSH 上去
* 手工传代码
* 手工守着服务

Serverless 更像：

* 服务按功能拆分
* 请求到了再执行
* 平台帮你管理弹性和基础设施

## 这类架构的代价

老师在后面的 AWS 课里又补了几条特别重要的现实细节：

* 它通常是无状态的
* 不是你独占的持久化机器
* 一段时间没人访问，运行环境可能被回收
* 面试里很容易被追问 `冷启动`

## 它和 BFF 的关系

老师在技术演进的讲法里，倾向于把很多以前 `Node BFF` 扛的中间层工作，逐渐转成 serverless 形态去承接。

也就是说：

* 以前常见是 `前端 -> Node BFF -> 真后端`
* 现在越来越多场景会变成 `前端 -> serverless 编排层 -> 数据 / 服务`

## 老师对 K8s 的判断

老师不是说 K8s 不存在，而是在这门课的语境下更强调：

* 先接受弹性云
* 先接受 serverless
* 对普通开发者和当前就业目标来说，这条线更现实

## 这页最重要的一句话

Serverless 在这里不是潮流词，而是“把基础设施负担尽量往平台侧推”的工程策略。
